## 2. Как проверить здоровье

Менеджер пакетов — **pnpm**. ⚠️ На свежем клоне сначала `pnpm install` —
`node_modules` не хранятся в git (есть только `pnpm-lock.yaml`), без установки
ни одна из команд ниже не запустится.

```bash
pnpm install     # обязательно на свежем клоне (node_modules не в репозитории)
pnpm typecheck   # tsc --noEmit — типы
pnpm lint        # eslint .
pnpm test        # vitest run — юнит/интеграционные
pnpm build       # next build
pnpm test:e2e    # playwright test (только при наличии браузера)
```

**Что считать нормой:**

- `pnpm test` → **1026 passed, 80 skipped** (база Этапа 6 — 1013/76; пост-роадмап: +9 санитайзер, +4 подарок;
  +4 skip — интеграционные кейсы per_customer_limit и выдачи подарка, гоняются в CI с БД). Цифра skipped —
  это **ожидаемо и здорово**.
  Интеграционные и e2e тесты обёрнуты в `skipIf` и пропускаются, если в окружении нет
  `DATABASE_URL` / Redis / браузера. В dev-окружении PostgreSQL/Redis/Docker/браузера нет,
  поэтому они и должны быть skipped (гоняются в CI/проде). Падений (failed) быть не должно.
- `pnpm typecheck` → без ошибок.
- `pnpm lint` → без ошибок.
- `pnpm build` → успешно.

Если число passed упало или появились failed — это регрессия; разбираться до коммита.

## 5. Ключевые архитектурные принципы

- **1 магазин = 1 БД.** Мультитенантность реализуется конфигом (отдельный экземпляр на магазин),
  а не разделением данных внутри одной БД (ADR-003).
- **Флаги модулей.** Что включено для конкретного магазина — задаётся в `lib/config/modules`.
  Фичи проверяют флаг, а не хардкодят наличие.
- **`defineAction` (`lib/server/action.ts`).** Единый конвейер серверного действия:
  **guard (проверка прав) → Zod-валидация → работа с БД → revalidate → запись в audit**.
  Все мутации идут через него.
- **RBAC по правам, не по enum-ролям.** Доступ проверяется по гранулярным правам (ADR-005),
  роли — это наборы прав.
- **Headless backend + Storefront API.** Витрины (THE CASE, Gang Auto) — отдельные потребители
  публичного API `/api/storefront/v1` (ADR-008).
- **Anti-tamper: цены считает сервер.** Клиент не присылает итоговые суммы; сервер пересчитывает
  корзину и сохраняет снимки позиций (ADR-010).
- **Деньги в копейках.** Никаких float; целые числа (модуль `lib/orders/money`).
- **Mock-режим всех внешних интеграций.** СДЭК и платёжки умеют работать в mock-режиме
  (детект по отсутствию реальных ключей), чтобы dev/CI не зависели от внешних сервисов (ADR-011).
- **Идемпотентные миграции.** Каждая SQL-миграция безопасна к повторному запуску.
- **Идемпотентность СДЭК-webhook.** Повторная доставка одного события не ломает состояние (ADR-011).

  (ADR-015: self-hosted, «copy-paste без аккаунтов»).

Перед любой работой прогнать §2 (health-check), чтобы стартовать с зелёного состояния. Изменения —
коммитами на ветке `foundation/etap-0-1` + синхронизация документации (`docs/00`, `docs/03`, этот файл).
Пуш в `origin` запускает CI (job `docker-smoke` валидирует деплой; `gh` локально не залогинен — статус
прогона смотреть на GitHub).

## 8. Как ведётся работа

- **Оркестрация через агентов Claude со встроенной ролью.** Роли (Solution Architect,
  Documentation Manager, History Manager, инженеры) задаются в промпте/инструкциях агента.
  Кастомные типы агентов из `.claude/agents` в текущем харнессе **не зарегистрированы** —
  полагаться на них нельзя; роль передаётся явно при запуске.
- **Работа волнами.** Этап дробится на пакеты (волны); каждая волна — самодостаточный кусок
  с тестами.
- **Коммит после каждой волны** на ветке `foundation/etap-0-1`. Сообщения коммитов — на русском,
  в формате `feat(этапN): …` / `docs(этапN): …`, как в истории выше.
