# Журнал проекта (История кита Uni_BD)

> Ведётся агентом **History Manager**. Цель — в любой момент восстановить контекст кита
> без анализа скрытых сессий. Формат: новые записи сверху, даты в формате ГГГГ-ММ-ДД.
> Здесь фиксируется история **самого кита** (обезличенной платформы), а не история

---

## Что такое Uni_BD

**Uni_BD** — универсальный обезличенный кит платформы **Admik**: готовый headless-бэкенд +
**административная панель** + **база данных** (PostgreSQL) + публичный **Storefront API** +
интеграции (СДЭК, Т-Банк) + CMS + DevOps (Docker/Caddy) + мультиагентный инструментарий
Claude Code.

Назначение: из кита **другая сессия Claude разворачивает админку и БД для любого
интернет-магазина** — как правило, для владельца без программиста. Репозиторий копируется
в проект, где уже есть готовый макет витрины (Next.js, без БД и без админки); задача — 
прикрутить к макету эту админку и БД и ретематизировать данные под нишу магазина, **не
трогая бизнес-логику платформы**. Витрина становится чистым потребителем `/api/storefront/v1/*`.

Ключевой принцип кита — **обезличенность**: в нём не должно быть тематики какого-либо
конкретного магазина. Любой контент нейтрален и универсален («витрина магазина»,
«интернет-магазин», «магазин-потребитель Storefront API», «ваш магазин»). Единственные
намеренные исключения — `docs/06-референсные-магазины.md` и `docs/13-сращивание-the-case.md`,
где реальный магазин оставлен как разобранный worked-example по онбордингу.

### Модель развёртывания

Один магазин = одна БД + конфигурация на отдельном VPS. Кодовая база едина и не меняется
между магазинами. Отличаются только **конфигурация** (`.env` + таблица `shop_settings`:
реквизиты, ключи, модули, брендинг) и **содержимое БД** (каталог, контент). Подробно —
[`docs/02-модель-развёртывания.md`](02-модель-развёртывания.md); онбординг витрины —
[`docs/20-онбординг-витрины-playbook.md`](20-онбординг-витрины-playbook.md); контракт API —
[`docs/21-контракт-storefront-api.md`](21-контракт-storefront-api.md).

## История кита

### 2026-07-01 — Синхронизация функциональной дельты платформы + авто-подтягивание инструментария

В кит перенесена накопленная функциональная дельта эталонной платформы Admik — **без
какой-либо тематики конкретного магазина**. Что вошло:

- **Лиды и рассылка** — приём заявок, подписки email; миграции `0030_leads`,
  `0031_newsletter_subscribers`, `0032_leads_app_delete_grant`.
- **Каталог** — категории, атрибуты, варианты товаров (в т.ч. кэш атрибутов вариантов).
- **Заказы** — создание и ведение ручных заказов из админки.
- **Аудит** — UI просмотра `audit_log` в админке.
- **Платежи** — cron `reconcile-pending` (доведение «зависших» платежей до финального
  статуса).
- **СДЭК** — печать накладных отправлений.
- **Однопользовательский режим** — упрощённый онбординг магазина с единственным
  администратором.
- **CMS-блоки** — универсальные секции `valuesStrip` и `philosophy` для конструктора страниц;
  миграция `0033_cms_pages_og_text` (OG-текст страниц).
- **Тесты** — десятки новых юнит-тестов, покрывающих перечисленное.

Одновременно налажено **авто-подтягивание MCP-серверов и скиллов** при развёртывании кита:
`.mcp.json` (4 сервера), `.claude/settings.json`, установочный `.claude/setup.sh`. Онбординг
нового магазина переведён на **стадийную (staged) схему** — новая сессия проходит шаги
последовательно (`START-HERE.md` → `CLAUDE.md` → `docs/20` → `docs/21` → скилл
`/onboard-shop`). Политика инструментария — [`docs/19-инструментарий-mcp-скиллы.md`](19-инструментарий-mcp-скиллы.md).

### 2026-06-24 — Создание кита обезличиванием платформы

Кит **Uni_BD** создан обезличиванием эталонного репозитория платформы Admik: из кодовой базы,
БД, документации и инструментария удалены тематика, бренд и домены конкретного магазина;
контент заменён на нейтральные универсальные формулировки. Итог — переносимый шаблон,
пригодный для разворачивания под любую нишу без правки бизнес-логики.

- **База данных:** 33 миграции (`db/migrations/0001…0033`).
- **Код-гейт зелёный:** typecheck 0 ошибок / lint 0 ошибок / тесты ~1964 passed.
- **Инструментарий:** 4 MCP-сервера (`.mcp.json`), кастомные скиллы в `.claude/skills/`
  (в т.ч. `onboard-shop`, `gate`, `new-shop`, `deploy-stand`, `integration-check`,
  `module-toggle`, `audit-cycle`), 9 субагентов с узкими allowlist'ами. Политика — `docs/19`.
- **Готовность:** кит готов к онбордингу нового магазина по playbook `docs/20`.

## Принцип ведения журнала (для будущих сессий)

1. Фиксируй здесь только изменения **самого кита** (платформы Admik в обезличенном виде):
   новая функциональность, миграции, инструментарий, изменения процессов онбординга,
   принятые технические решения. Историю разворачивания конкретного магазина ведёт журнал
   того магазина, а не этот файл.
2. **Не вноси тематику конкретного магазина.** Любые примеры и формулировки — нейтральные и
   универсальные. Исключения-worked-example: `docs/06` и `docs/13`.
3. Новые записи — сверху, с датой и коротким заголовком «что и зачем». Детали выноси в
   профильные документы `docs/*` и ссылайся на них, а не дублируй.
4. Обновляй раздел «Текущее состояние» (число миграций, статус гейта) при каждом значимом
   изменении, чтобы новая сессия сразу видела актуальный снимок.
