# Проверка интеграций СДЭК + Т-Банк (mock)

У каждой интеграции есть mock-режим — магазин работает без боевых ключей.
Боевые ключи СДЭК/Т-Банк — по разрешению владельца; на стенде их по умолчанию НЕТ.

## Т-Банк (оплата) — mock-поток
- Витрина инициирует эквайринг через `initPayment` ТОЛЬКО для `card`/`sbp`;
  `cdek-pay` идёт в ЛК без онлайн-шлюза.
- `initPayment` ждёт `accessToken` | `email` (НЕ `token`).
- Demo без ключей: mock-страница оплаты (СТРОГО mock — `notFound` в бою) +
  `PaymentService.confirmMockPayment` (СТРОГО mock-gated).
- Доплата неоплаченного заказа — кнопка «Оплатить картой» в личном кабинете.
- Модуль `payments` на стенде включён через БД (`shop_settings.module_overrides`), не env.
  После прямой SQL-правки нужен `docker compose restart app`.

## СДЭК (доставка) — mock-режим
- Проверяй: расчёт стоимости, получение ПВЗ, создание отправления, трекинг/статусы, печать.
- Машина статусов: докрутка статуса + TOCTOU-гард отмены.
- Токен СДЭК кэшируется в `lib/cdek/token-cache.ts` (само-лечится по TTL).

## Прогон
1. Юнит: `pnpm test` по `tests/payments/*`, `tests/cdek/*` (часть mock-gated).
2. Живой e2e: `node scripts/verify-admin.mjs e2e` — каталог → карточка → корзина →
   checkout с СДЭК → оплата (mock card/sbp → demo-страница → confirmMockPayment).
3. Mock-гарды: убедись, что mock-страницы дают `notFound` при боевом флаге (нет утечки demo в прод).

## Итог
Сводка: расчёт/ПВЗ/отправление/трекинг (СДЭК) + init/confirm (Т-Банк) = OK/FAIL.
Подтверди, что боевые пути не задеты и mock-гарды на месте.
