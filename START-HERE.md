# START HERE — Uni_BD: как прикрутить админку и БД к готовому макету интернет-магазина

> **Этот файл новая сессия Claude Code читает ПЕРВЫМ.** Он объясняет, что это за
> репозиторий, что от тебя требуется и в каком порядке действовать. Порядок такой:
> прочитать этот файл → выполнить `bash .claude/setup.sh` → прочитать `CLAUDE.md`
> (правила платформы) и `docs/20-онбординг-витрины-playbook.md` (пошаговый ранбук)
> → запустить скилл `/onboard-shop`.

## 0. Предпосылки и первый запуск (Prerequisites)

> **Нетехническому владельцу — одна фраза для старта.** Выполни в терминале
> `bash .claude/setup.sh`, открой сессию Claude Code в этой папке и скажи:
> **«изучи проект и запусти онбординг магазина»** (или вызови скилл
> `/onboard-shop`). Дальше Claude проведёт интервью и всё сделает сам —
> подробный порядок шагов в §3.

**Обязательные предпосылки:**
- **Node.js ≥ 20** — нужен и для платформы (Next.js / pnpm), и для npx-MCP-серверов
  (playwright, context7). Проверка: `node --version`.
- **Docker** (+ Docker Compose) — база данных (PostgreSQL), а также Redis, MinIO и
  Caddy поднимаются контейнерами из коробки (`make up`). БД работает в Docker.
- **Разовый trust-диалог Claude Code.** При первом открытии папки Claude Code один
  раз спросит доверие к каталогу — **подтверди**. Иначе закоммиченный
  `.claude/settings.json` не применится и MCP-серверы не включатся.

**Запусти ПЕРВЫМ — один раз, сразу после клонирования:**
```bash
bash .claude/setup.sh
```
Скрипт пишет `.claude/settings.local.json` (включает MCP-серверы локально — они
действуют сразу, даже до подтверждения доверия) и проверяет предпосылки.

**Опционально (не критично для работы):**
- `FIRECRAWL_API_KEY` — веб-ресёрч (актуальные доки СДЭК/Т-Банк, импорт каталога).
  Без ключа сервер firecrawl просто не поднимется; всё остальное работает.
- **glyph** — символьная навигация по большому TS-репо (экономия контекста).
  Необязателен; сессия работает и без него.

## 1. Что это за репозиторий

**Uni_BD** — это **готовая универсальная платформа Admik**: headless-бэкенд +
административная панель + база данных + DevOps + мультиагентный инструментарий.
Здесь уже написан и протестирован **весь код админки и БД** (33 миграции, ~1100
юнит-тестов, аудит сошёлся — 0 настоящих багов в последних циклах). Тематики
конкретного магазина здесь НЕТ — она нейтральная заглушка.

Назначение: репозиторий клонируется/копируется в проект, где **уже есть готовый
макет витрины интернет-магазина** (Next.js, без БД и без админки). Твоя задача —
**прикрутить к этому макету эту админку и БД**, ретематизировав данные под нишу
магазина, **не трогая код бизнес-логики платформы**.

```
   ┌─────────────────────────────┐         ┌──────────────────────────────┐
   │  Макет витрины магазина      │  HTTP   │  Admik (этот репозиторий)    │
   │  (Next.js, ./storefront)     │ ──────▶ │  админка + Storefront API    │
   │  рендерит каталог/корзину/   │ ◀────── │  + БД (PostgreSQL) + СДЭК +  │
   │  оформление заказа           │  JSON   │  Т-Банк + CMS + настройки    │
   └─────────────────────────────┘         └──────────────────────────────┘
        потребитель API                     /api/storefront/v1/*  (контракт)
```

Витрина становится **чистым потребителем** публичного Storefront API
`/api/storefront/v1/*`: своей БД, каталога, заказов, расчёта доставки и оплаты у
неё нет — всё это на стороне Admik. Витрина только рендерит и собирает выбор
покупателя (корзина/wishlist могут оставаться клиентскими).

## 2. Твоя цель (Definition of Done)

К концу работы должно быть:

1. ✅ Поднята БД PostgreSQL, накатаны **все** миграции, засеяны права/роли/владелец.
2. ✅ Каталог, атрибуты и брендинг **ретематизированы** под нишу магазина
   (через сид-данные + `shop_settings` + `.env`), без правок бизнес-логики и
   без правок существующих миграций.
3. ✅ Макет витрины **подключён** к Storefront API: статические данные заменены
   на запросы к `/api/storefront/v1/*` (каталог, карточка, категории, корзина-quote,
   создание/трекинг заказа, СДЭК, оплата, CMS-страницы, настройки/брендинг).
4. ✅ Работает **админка** (`/admin`) — владелец заводит товары/категории/контент,
   и заведённое корректно доезжает до витрины.
5. ✅ Пройдены **две обязательные проверки**: код-гейт (`/gate`) и живой прогон в
   браузере (`/deploy-stand` или `/verify` + Playwright), интеграции — в mock-режиме.
6. ✅ Всё задокументировано (журнал `docs/00`, состояние `docs/10`, README инстанса).

## 3. Порядок действий новой сессии

```
1.  bash .claude/setup.sh — преполётная настройка инструментов (см. §0).
2.  Прочитай START-HERE.md (этот файл), затем CLAUDE.md — роль оркестратора и инварианты.
3.  Прочитай docs/20-онбординг-витрины-playbook.md (ранбук, фазы 0–6) и
    docs/21-контракт-storefront-api.md (контракт API для сращивания).
4.  Изучи макет витрины в ./storefront — его страницы и какие данные ему нужны от API.
5.  Проведи ДЕТАЛЬНОЕ интервью с владельцем по docs/22 (ниша, домены, модули,
    реквизиты, брендинг, каталожная модель).
6.  Стадийно подними БД: Docker → все миграции → seed прав/ролей/владельца.
7.  Ретематизируй каталог/атрибуты/брендинг под нишу (сид + shop_settings + .env) —
    без правок бизнес-логики и без правок существующих миграций.
8.  Срасти макет витрины со Storefront API: замени статические данные запросами
    к /api/storefront/v1/*.
9.  Пройди две проверки: код-гейт /gate и живой прогон в браузере; интеграции — mock.
    Не останавливайся до Definition of Done (см. §2) и зелёных проверок.
```

Весь процесс оркеструет скилл `/onboard-shop` — он ведёт его субагентами по этим
фазам. Если макет витрины ещё не в проекте — попроси положить его в каталог
`./storefront` (см. `docs/20`, фаза 0).

## 5. Ключевые факты для быстрого старта

**Стек:** Next.js 16 (App Router, standalone) · React 19 · TypeScript strict ·
PostgreSQL 15 (postgres.js) · Redis (ioredis) · S3/MinIO (sharp) · Zod · Vitest ·
pnpm · Docker + Caddy (auto-TLS).

**Запуск платформы (локально, из коробки):**
```bash
pnpm install
cp .env.example .env          # заполнить под магазин (см. §6)
make up                       # docker compose up -d (app+postgres+redis+minio+caddy)
make init                     # init-shop.sh: миграции → seed прав/ролей/владельца
make smoke                    # health-проверки эндпоинтов
```
Или без Docker для разработки: поднять Postgres/Redis/MinIO, прогнать
`scripts/init-shop.sh`, затем `pnpm dev`.

**Модули** (включаются через `ADMIK_MODULES` или БД `shop_settings.module_overrides`):
`catalog`, `orders`, `cdek`, `cms`, `payments`. Ядро (`core`: auth/RBAC/настройки/
аудит/заявки/подписчики/настройки витрины) — всегда включено.

**Storefront API** — публичный, авторизация по API-ключу (`X-Storefront-Key`) ИЛИ
по `Origin` (CORS allowlist). При пустых ключах/origins — **mock-режим** (открыт
всем + warn), удобно для локального demo. Полный контракт — `docs/21`.

**Интеграции работают без боевых ключей (mock):** СДЭК и Т-Банк при пустых
`CDEK_*`/`TBANK_*` отдают фикстуры/demo-страницу оплаты — можно пройти весь путь
покупателя локально. Боевые ключи подставляются ТОЛЬКО в `.env` на сервере.

## 6. Минимальная конфигурация магазина (`.env`)

Самое важное при онбординге (полный список — в `.env.example`):

| Переменная | Назначение |
|---|---|
| `SHOP_NAME` | Название магазина |
| `SHOP_DOMAIN` / `ADMIN_DOMAIN` | Домен витрины и поддомен админки (`admin.<домен>`) |
| `SHOP_ORDER_PREFIX` | Префикс номера заказа (напр. `GA` → `GA-2026-000123`) |
| `SHOP_CURRENCY` | Валюта (по умолчанию `RUB`) |
| `ADMIK_MODULES` | Включённые модули (`catalog,orders,cdek,cms,payments`) |
| `POSTGRES_*`, `APP_PASSWORD`, `MIGRATOR_PASSWORD`, `DATABASE_URL` | Доступ к БД |
| `OWNER_EMAIL` / `OWNER_PASSWORD` | Владелец админки (пустой пароль → автогенерация) |
| `STOREFRONT_API_KEYS` / `STOREFRONT_ALLOWED_ORIGINS` | Доступ витрины к API |
| `NEXT_PUBLIC_ADMIK_API_URL` | Публичный адрес API для клиентского бандла витрины |
| `SEED_DEMO_CATALOG` | `true` — налить нейтральный демо-каталог (для smoke); на бою `false` |
| `CDEK_*` / `TBANK_*` | Ключи интеграций — ПУСТО = mock; боевые только на сервере |

**Секреты — только в `.env` (gitignored), никогда в репозитории.**

## 7. Незыблемые инварианты (нарушать нельзя)

1. **TDD:** сначала тест (падает) → потом код до зелёного. Каталог тестов — `tests/`.
2. **Две проверки** перед коммитом: код-гейт `/gate` **и** живой прогон в браузере.
3. **Мультитенантность:** ноль хардкода под один магазин. Название, логотип, цвета,
   контакты, реквизиты, ниша — только в `.env` и `shop_settings`. Бизнес-логику
   платформы **не меняем** — переносим конфигурацией и данными.
4. **Миграции не редактируем:** существующие `db/migrations/*.sql` иммутабельны;
   новые — только аддитивные (`check-migrations.sh` это проверяет).
5. **Секреты только из env.** Боевые ключи СДЭК/Т-Банк/БД — на сервере, не в git.
6. **Документируй каждую правку** (агенты `documentation-manager` / `history-manager`).

> Подробное обоснование и worked-example сращивания (на примере реальной витрины,
> на которой платформа обкатана) — в `docs/13`. В `docs/06` — анализ референсных
> витрин. Эти два документа разбирают конкретный магазин как worked-example —
> используй их как образец метода, а не как тематику твоего магазина.
