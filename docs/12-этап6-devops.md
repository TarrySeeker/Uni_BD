# Этап 6 — DevOps: copy-paste-деплой, бэкапы, мониторинг, обновления без простоев

> **Кем ведётся.** Solution Architect (проектное решение, инварианты, ADR) + DevOps Engineer
> (реализация скриптов/конфигов) + Documentation Manager (`docs/09`, синхронизация `docs/02`).
> Спецификация для реализаторов: каждый пакет — отдельный коммит, «сначала проверки/тесты,
> потом код» (ADR-004 в DevOps-прочтении: shellcheck/`bash -n`, smoke, restore-drill,
> idempotency-чек). Это **последний** этап роадмапа (`docs/03` §«Этап 6», 6.1–6.5).
>
> **Формат дат.** ISO 8601. Дата составления: **2026-06-16**.

## 0. Краткое следствие (TL;DR)

Этап 6 не вводит новой бизнес-логики — он **финализирует модель развёртывания** (`docs/02`) до
продакшн-готовности силами **стороннего человека** по принципу «копировать-вставить». Инфраструктура
Этапа 0 (Dockerfile, `docker-compose.yml` с 5+1 сервисами и healthcheck, `Caddyfile` с авто-SSL и
JSON-логами, `scripts/init-shop.sh`, CI) уже покрывает «поднять стек». Не хватает пяти эксплуатационных
кусков, которые и составляют Этап 6:

1. **6.1 Smoke + единый сценарий** — обёртка `scripts/smoke.sh` (curl-проверки `/api/health` и
   ключевых эндпоинтов после `up` + `init`), плюс `Makefile`/`scripts/deploy.sh` как один copy-paste.
2. **6.2 Бэкапы** — `scripts/backup.sh` (`pg_dump` + `mc mirror` MinIO), `scripts/restore.sh`,
   `scripts/restore-drill.sh` (проверяемое восстановление в одноразовую БД), отдельный
   compose-сервис-планировщик `backup` (как существующий `cron`), ретенция по N дней.
3. **6.3 Мониторинг/логи** — структурированные **JSON-логи приложения** (`lib/logger.ts`, stdout →
   `docker logs`/драйвер), мониторинг `/api/health` лёгким `scripts/healthcheck-monitor.sh` (cron),
   расширение `/api/health` до проверки зависимостей (БД/Redis/S3) под флагом. Внешний APM — out of
   scope (точка расширения).
4. **6.4 Обновления без простоев** — `scripts/update.sh` (pull → build нового образа → миграции
   только-аддитивные → health-gate → переключение → откат при провале); правило «миграции только
   аддитивные» + проверочный линтер `scripts/check-migrations.sh`.
5. **6.5 Инструкция для не-разработчика** — `docs/09-инструкция-разворачивания.md` (анонсирована в
   `docs/02` п.7).

**Все проверки запускаемы статически в dev** (нет Docker/Postgres/браузера): `shellcheck` + `bash -n`
для скриптов, Vitest для `lib/logger.ts`. Интеграционные (реальный `pg_dump`/restore/up) — помечены
**«гоняется на VPS/CI»**. Нужен **ADR-015** (стратегия бэкапов и обновлений single-VPS) — текст ниже,
§6, для переноса в `docs/01`.

## 1. Цель, охват, Definition of Done

### 1.1. Цель этапа

Довести `docs/02` до состояния, когда сторонний человек **без участия разработчика**:
поднимает магазин, получает подтверждение готовности (smoke), имеет **автоматические бэкапы с
проверенным восстановлением**, видит, что сервис **жив** (мониторинг/логи), и может **безопасно
обновить** код магазина без потери данных и с возможностью отката. Universal: всё через `.env`,
ничего магазино-специфичного в коде/скриптах (ADR-003).

### 1.2. In scope

- **6.1** Единый copy-paste-сценарий деплоя + smoke-проверка (`/api/health` и ключевые эндпоинты).
- **6.2** Бэкапы PostgreSQL (`pg_dump`) и медиа (MinIO/S3), расписание, ретенция, **restore-drill**.
- **6.3** JSON-логирование приложения; мониторинг доступности (`/api/health`) и ресурсов уровня
  compose; простые алерты (webhook/email) без внешнего SaaS.
- **6.4** Процедура обновления single-VPS с health-gate и откатом; правило аддитивных миграций + чек.
- **6.5** `docs/09-инструкция-разворачивания.md` — пошагово, языком не-разработчика.

### 1.3. Out of scope (явно отложено, точки расширения)

- Внешние SaaS-мониторинг/APM (Datadog/Sentry/Grafana Cloud), Prometheus+Grafana-стек,
  централизованный лог-агрегатор (Loki/ELK) — **точка расширения** (§3.6.4); на single-VPS избыточны.
- Kubernetes / несколько реплик / blue-green на нескольких нодах — противоречит ADR-003
  («1 магазин = 1 VPS»). Zero-downtime достигается **в рамках одного VPS** (§3.4).
- Offsite-репликация бэкапов в чужое облако «из коробки» — даём **точку подключения** (env
  `BACKUP_S3_*`/rclone-remote), но конкретного провайдера не хардкодим.
- Автоматический CD-деплой на прод из CI (push-to-deploy) — оставляем ручной `update.sh` (безопаснее
  для разнородных VPS); CI остаётся гейтом качества (typecheck/lint/test/build).

### 1.4. Definition of Done (этап)

- [ ] `shellcheck` и `bash -n` зелёные для всех скриптов в `scripts/` (новый job в CI).
- [ ] `scripts/smoke.sh` проверяет `/api/health` + ключевые публичные эндпоинты; код возврата ≠0 при
      любом провале; задокументирован в `docs/02`/`docs/09`.
- [ ] `scripts/backup.sh` создаёт согласованный дамп БД + зеркало медиа; `restore-drill.sh`
      **восстанавливает в одноразовую БД и проверяет** (гоняется на VPS/CI). Ретенция работает.
- [ ] compose-сервис `backup` поднимается вместе со стеком; расписание из `.env`.
- [ ] `lib/logger.ts` отдаёт JSON-логи; покрыт Vitest; ключевые точки (Server Actions/Storefront/cron)
      пишут через него; секреты не логируются.
- [ ] `scripts/update.sh` выполняет обновление с health-gate и откатом; миграции — только аддитивные;
      `check-migrations.sh` ловит запрещённые DDL.
- [ ] `docs/09-инструкция-разворачивания.md` доводит человека без опыта от `git clone` до зелёного
      smoke; каждый шаг проверяем.
- [ ] `docs/02` обновлён (ссылки на новые скрипты/инструкцию); чек-лист «copy-paste» подтверждён.
- [ ] ADR-015 внесён в `docs/01`; роадмап `docs/03` отмечает 6.1–6.5 done.

## 3. Подсистемы (модель · файлы · как проверяется)

### 6.1. Скрипт развёртывания «из коробки» + smoke

**Модель.** Единый copy-paste-сценарий = существующая последовательность `docs/02` (clone → `.env` →
`docker compose up -d` → `init-shop.sh`) **плюс** финальный smoke. Оборачиваем в `Makefile` (цели для
тех, кто привык к `make`) и в `scripts/deploy.sh` (для тех, кто копирует одну команду). Сами шаги не
меняются — добавляется **подтверждение готовности**.

**Файлы.**
- `scripts/smoke.sh` (новый) — набор curl-проверок против запущенного стека:
  - `GET /api/health` → ожидаем HTTP 200 и `"status":"ok"` (база);
  - `GET /api/storefront/v1/categories` и `/products` → 200 (read-каталог жив; модуль `catalog`);
  - `GET /admin/login` → 200 (админка отдаётся);
  - опц. по флагам модулей: `/api/storefront/v1/settings`, `/api/storefront/v1/pages` (если `cms`).
  - Базовый URL — из `.env` (`SMOKE_BASE_URL`, по умолчанию `http://localhost`); ретраи с таймаутом
    (стек поднимается не мгновенно), понятный вывод OK/FAIL по каждому эндпоинту, exit≠0 при провале.
  - **Не требует боевых ключей** (mock-режимы СДЭК/Storefront/S3 уже есть) — smoke зелёный на чистом
    магазине.
- `scripts/deploy.sh` (новый) — тонкая обёртка: проверка `.env` → `docker compose up -d --build` →
  `docker compose exec app /app/scripts/init-shop.sh` (или хостовый запуск) → `scripts/smoke.sh`.
  Идемпотентна (повторный запуск безопасен — `up -d` и `init-shop` идемпотентны).
- `Makefile` (новый) — цели `make up` / `make init` / `make smoke` / `make deploy` / `make logs` /
  `make backup` / `make update` / `make down`. Тонкие алиасы над compose/скриптами — для удобства, не
  единственный путь (всё работает и голыми командами из `docs/09`).
- Правка `scripts/init-shop.sh` — в конце опционально вызвать `smoke.sh` (флаг `RUN_SMOKE=true`), чтобы
  «init» сам подтверждал готовность; по умолчанию печатает подсказку (как сейчас) + рекомендует
  `make smoke`.

**Как проверяется.**
- Статически (dev): `shellcheck scripts/smoke.sh scripts/deploy.sh`, `bash -n`. Makefile — `make -n`.
- Интеграционно (VPS/CI, помечено): поднять стек, прогнать `smoke.sh`, ожидать exit 0; негативный
  кейс — остановить `app`, ожидать exit≠0.
- DoD-критерий: smoke ловит «поднялось, но не работает» (например, забыли `init-shop` → `/api/health`
  ok, но `/api/storefront/v1/products` 500 из-за отсутствия таблиц).

### 6.2. Бэкапы БД и медиа

**Модель.** Тома Docker — **не** бэкап. Делаем логические дампы и зеркало медиа в отдельный каталог
тома `backups`, с ретенцией и **проверяемым восстановлением**. Планировщик — отдельный compose-сервис
по образцу существующего `cron` (alpine + crond), расписание/ретенция — из `.env`. Опциональная
выгрузка offsite — через env-настраиваемый remote (rclone/`mc`), без хардкода провайдера.

**Файлы.**
- `scripts/backup.sh` (новый):
  - **БД:** `pg_dump` (формат `custom` `-Fc` → сжатие + гибкий restore) в
    `backups/db/<shop>-<UTC-timestamp>.dump`. Параметры подключения — из `.env` (PG\* как в
    `init-shop.sh`). Дамп под ролью владельца/`migrator` (полные права на чтение схемы).
  - **Медиа:** `mc mirror` (MinIO Client) бакета `S3_BUCKET` в `backups/media/` **или** `mc mirror`
    в offsite-remote. Для внешнего S3 (Timeweb/AWS) — тот же `mc`/`rclone` по `S3_*`.
  - **Атомарность/консистентность:** дамп БД и зеркало медиа — независимы; для согласованности
    «заказ ↔ его картинка» порядок: сперва БД, затем медиа (медиа аддитивно, удалённый позже файл
    хуже, чем отсутствующая в дампе картинка). Документируем как «eventually consistent backup»,
    достаточно для ИМ.
  - **Ретенция:** удалять дампы/зеркала старше `BACKUP_RETENTION_DAYS` (env, по умолчанию 14).
  - Логи в stdout (попадают в `docker logs backup`) в JSON-совместимом формате (§6.3).
- `scripts/restore.sh` (новый): `pg_restore --clean --if-exists` выбранного дампа в целевую БД +
  `mc mirror` медиа обратно. Требует явного подтверждения (перезапись данных) — переменная
  `CONFIRM=yes` или интерактивный вопрос; защита от случайного запуска на боевой БД.
- `scripts/restore-drill.sh` (новый, **ключевой проверочный артефакт**): берёт **последний** дамп,
  поднимает **одноразовую** временную БД (отдельный контейнер `postgres` или временная БД на том же
  сервере с суффиксом `_drill`), делает `pg_restore`, выполняет smoke-запросы к схеме
  (`SELECT count(*)` по ключевым таблицам: `users`, `products`, `orders`, `audit_log`), затем
  **уничтожает** временную БД. Возвращает exit 0 только если восстановление прошло и данные читаются.
  Это и есть «проверяемое восстановление» из критериев приёмки 6.2.
- Сервис `backup` в `docker-compose.yml` (новый, по образцу `cron`): alpine + `postgresql-client` +
  `minio/mc`, crond с расписанием `BACKUP_CRON` (env, по умолчанию `0 3 * * *` — ежесуточно ночью),
  том `backups:/backups`. Раз в неделю (`BACKUP_DRILL_CRON`) запускает `restore-drill.sh` и логирует
  результат (ранний сигнал «бэкапы нерабочие»).
- Новые env (в `.env.example` и при необходимости `lib/config/env.ts` — но backup-скрипты читают
  `.env` напрямую, валидация в env.ts не обязательна, т.к. это не рантайм приложения):
  `BACKUP_CRON`, `BACKUP_RETENTION_DAYS`, `BACKUP_DRILL_CRON`, опц. `BACKUP_OFFSITE_REMOTE`
  (rclone/`mc` alias) и его креды.
- Том `backups` в секции `volumes` compose.

**Как проверяется.**
- Статически (dev): `shellcheck`/`bash -n` всех трёх скриптов; проверка, что секреты берутся из env,
  а не хардкод.
- Интеграционно (VPS/CI, помечено): `backup.sh` → файл появился и непустой; `restore-drill.sh` →
  exit 0; ретенция → старые файлы удаляются (создать файл с прошлой датой, прогнать, проверить
  отсутствие). На CI можно гонять облегчённый drill на сервисном `postgres` GitHub Actions.
- DoD: «бэкап создаётся по расписанию; восстановление проверено» (роадмап 6.2).

### 6.3. Мониторинг и логирование

Реалистично для single-VPS **без внешних SaaS**: structured-логи в stdout (читаются `docker logs` и
любым драйвером), self-hosted health-монитор и алерты через webhook/email. Тяжёлый стек (Prometheus/
Grafana/Loki) — **точка расширения**, не входит.

#### 6.3.1. Структурированные JSON-логи приложения

**Модель.** Единый логгер `lib/logger.ts` — функции `info/warn/error` (+ `child` с контекстом),
выводящие **одну JSON-строку на событие** в `stdout`: `{ts, level, msg, ...context}`. Поля контекста:
`requestId`/`actor`/`module`/`action` где применимо. **Секреты не логируются** — переиспользуем уже
существующую логику санитизации (`lib/audit/log.ts` маскирует чувствительные поля; логгер применяет
тот же allow/deny-list). В dev можно включить pretty-вывод (`LOG_PRETTY=true`), в prod — чистый JSON.

**Куда.** stdout контейнера → собирается Docker logging-драйвером. На VPS по умолчанию `json-file` с
ротацией (добавляем в compose `logging: { driver: json-file, options: { max-size, max-file } }` —
защита от переполнения диска логами). Caddy уже пишет JSON access-логи. Кто хочет — направляет драйвер
в свой агрегатор (точка расширения, не хардкодим).

**Файлы.**
- `lib/logger.ts` (новый) — чистая функция форматирования + враппер над `console.log/error`.
- Интеграция в ключевые точки: `lib/server/action.ts` (`defineAction` — лог ошибок/значимых действий),
  Storefront-роуты (`lib/storefront/*` — 4xx/5xx, rate-limit), cron-роуты СДЭК, webhook СДЭК.
  Аддитивно — не меняет поведение, только добавляет наблюдаемость.
- `logging:`-блок в `docker-compose.yml` для `app` (и при желании прочих сервисов) с ротацией.
- env: `LOG_LEVEL` (debug|info|warn|error, дефолт info), `LOG_PRETTY` (дефолт false) — в `env.ts`.

#### 6.3.2. Мониторинг доступности `/api/health`

**Модель.** Расширяем `/api/health` до режима **deep-check** (`/api/health?deep=1` или отдельный
`/api/health/ready`): проверяет БД (`SELECT 1`), Redis (`PING`, если `REDIS_URL`), S3 (HEAD bucket,
если не mock). Базовый `/api/health` (liveness) остаётся лёгким и быстрым (для compose healthcheck —
не нагружает зависимости). Deep-check (readiness) — для внешнего монитора и при обновлении (health-gate
§6.4). Возвращает `{status, checks:{db,redis,s3}}` и HTTP 503, если критичная зависимость недоступна.

**Внешний монитор (self-hosted).** `scripts/healthcheck-monitor.sh` (новый) — периодический (через
существующий паттерн cron-сервиса или системный cron хоста) `curl` к публичному `https://<domain>/
api/health`; при N подряд неудачах шлёт алерт через `ALERT_WEBHOOK_URL` (Telegram/Slack/любой webhook)
и/или email (`mailx`/SMTP env). Это закрывает критерий «падение сервиса детектируется» без SaaS.
Опционально — внешний бесплатный пинг (UptimeRobot и т.п.) как независимая страховка (рекомендация в
`docs/09`, не зависимость).

**Мониторинг ресурсов.** Лёгкий уровень: `docker stats`-снимок + проверка свободного диска/RAM в том же
`healthcheck-monitor.sh` (алерт при `disk > 90%` — критично из-за бэкапов и логов). Полноценные метрики
(cAdvisor/node-exporter) — точка расширения §6.3.4.

#### 6.3.3. Алерты

Один канал — `ALERT_WEBHOOK_URL` (env). События: health down (N retries), restore-drill failed,
backup failed, disk > порога. Формат — JSON-payload, совместимый с Telegram bot API / Slack incoming
webhook (выбор канала — настройка, не код). При пустом `ALERT_WEBHOOK_URL` — алерты только в лог
(graceful degradation, как mock-режимы интеграций).

#### 6.3.4. Точка расширения (Out of scope, зафиксировано)

Prometheus + node-exporter + cAdvisor + Grafana (метрики/дашборды), Loki/Promtail (агрегация логов),
Sentry/Glitchtip (error-tracking приложения). Все они **подключаются поверх** без правки ядра: логи уже
JSON (Loki-friendly), `/api/health` — готовый probe, Docker-метрики стандартны. Документируем как
«как масштабировать мониторинг» в `docs/09`/комментарии compose, но **не включаем** в Этап 6.

**Как проверяется.**
- Статически (dev): Vitest на `lib/logger.ts` (формат JSON, маскирование секретов, уровни); unit на
  deep-check-функцию `/api/health` с моками БД/Redis/S3 (паттерн `skipIf` без реальных зависимостей —
  как существующие интеграционные тесты); `shellcheck` монитора.
- Интеграционно (VPS/CI): остановить БД → `/api/health?deep=1` отдаёт 503 + алерт ушёл в тестовый
  webhook; убедиться, что секреты не попадают в логи (grep по выводу).

### 6.4. Обновления без простоев

**Модель (single-VPS, реалистично).** За Caddy стоит один инстанс `app`. Полный k8s-rolling недоступен,
но **near-zero-downtime** достижим: собрать **новый образ заранее**, прогнать **только-аддитивные**
миграции (старый код продолжает работать на новой схеме), затем **быстро пересоздать** контейнер `app`
с health-gate. Caddy с `restart`-семантикой и коротким окном — простой измеряется секундами. Для
честного zero-downtime опционально описываем **two-slot** приём (поднять `app2` на новом образе, дождаться
healthy, переключить `reverse_proxy` Caddy на `app2`, погасить старый) — но это **опция**, не дефолт
(усложняет; для ИМ-админки секундный простой приемлем).

**Ключевой инвариант — миграции только аддитивные.** Идемпотентность (`IF NOT EXISTS`) ≠ обратная
совместимость. Во время выката **одновременно** живут старый и новый код относительно новой схемы.
Поэтому:
- **Разрешено:** `ADD COLUMN` (nullable/с дефолтом), `CREATE TABLE/INDEX IF NOT EXISTS`, новые seed
  `ON CONFLICT DO NOTHING`, расширяющие `CHECK` через `NOT VALID` + позднее `VALIDATE`.
- **Запрещено в одном релизе:** `DROP COLUMN`, `DROP TABLE`, `DROP CONSTRAINT`, `DROP DEFAULT`,
  `DROP NOT NULL`, `RENAME`, сужение/смена типа, `NOT NULL` без дефолта, удаление/переименование
  значения enum, **`DROP INDEX`**. Деструктивные изменения — только **многошаговым expand/contract**
  через несколько релизов (сначала добавить новое и писать в оба, затем переключить чтение, и лишь в
  будущем релизе удалить старое).
- **`DROP INDEX` (волна 5):** снятие индекса убирает инвариант, а в Admik уникальность и
  идемпотентность держатся именно на UNIQUE-индексах (`orders_idempotency_uniq`, `orders_number_uniq`,
  `uq_tbank_payment_log_idem`, `uq_cdek_status_idem`, `inventory_unit_uniq`,
  `promo_redemptions_order_uniq`, `customers_email_uniq`, `users_email_uniq`), часть — partial-индексы
  (только индекс, не constraint, поэтому правило `DROP CONSTRAINT` их не покрывало). Снятие такого
  индекса в релизе молча возвращает дубли/двойные списания. Правило ловит `DROP INDEX [IF EXISTS]` и
  `DROP INDEX CONCURRENTLY`; `CREATE [UNIQUE] INDEX` остаётся аддитивным и не блокируется.
- Все существующие миграции `0001–0029` уже аддитивны (`ADD COLUMN IF NOT EXISTS`, `CREATE ...`,
  `CREATE OR REPLACE FUNCTION` + `DROP TRIGGER IF EXISTS`/`CREATE TRIGGER`) — правило закрепляет
  статус-кво.

**Файлы.**
- `scripts/update.sh` (новый):
  1. `git pull` (или развёртывание нового тега) — фиксируем `git rev-parse HEAD` старой версии для
     отката.
  2. `docker compose build app` — собрать новый образ (старый контейнер ещё работает).
  3. `scripts/backup.sh` — **обязательный бэкап перед миграциями** (страховка отката данных).
  4. `check-migrations.sh` — статически убедиться, что новые миграции аддитивны (см. ниже); провал →
     стоп до выката.
  5. Накатить миграции (`init-shop.sh` — он идемпотентен и накатывает только новые файлы безопасно;
     схема аддитивна → старый контейнер продолжает работать).
  6. `docker compose up -d app` (пересоздать `app` новым образом) — Docker дождётся healthcheck.
  7. **Health-gate:** `curl /api/health?deep=1` с ретраями; если не зелёный за таймаут → **откат**:
     вернуть предыдущий образ/тег (`git checkout <old>` + `up -d` старого образа) и оповестить.
  8. Лог результата (версия до/после, длительность простоя) в JSON.
- `scripts/check-migrations.sh` (новый) — grep/линтер новых `*.sql` (по diff с предыдущим тегом или по
  списку нанакаченных) на запрещённые токены (`DROP COLUMN`, `DROP TABLE`, `ALTER ... RENAME`,
  `DROP NOT NULL`-наоборот, `SET NOT NULL` без default и т.п.). Эвристика, не доказательство, но ловит
  очевидные нарушения; exit≠0 → блокирует `update.sh`. Запускается также в CI (см. §5 W4).
- **Откат данных:** если миграция всё же что-то испортила — `restore.sh` из бэкапа шага 3
  (документируем как «крайний случай»; обычный откат — только код/образ, данные совместимы по правилу
  аддитивности).

**Как проверяется.**
- Статически (dev): `shellcheck`/`bash -n` `update.sh`/`check-migrations.sh`; **unit-проверка
  `check-migrations.sh`** на фикстурах (плохая миграция с `DROP COLUMN` → exit≠0; хорошая → 0) — это
  «тест до кода» для DevOps. Прогон `check-migrations.sh` по реальным `0001–0024` → должен быть зелёным
  (доказательство, что статус-кво аддитивен).
- Интеграционно (VPS/CI): прогнать `update.sh` со старой на новую версию на тестовом стенде, замерить
  простой, проверить отсутствие потери данных (счётчики строк до/после); сценарий отката (новая версия
  специально нездорова → health-gate откатывает).

## 4. Сквозные инварианты Этапа 6

1. **Секреты — только в `.env`/секретных хранилищах.** Ни один скрипт (backup/restore/update/monitor)
   не хардкодит паролей/ключей; всё через env (как `init-shop.sh`). В репозитории — только
   `.env.example`-плейсхолдеры. Бэкапы могут содержать данные → каталог `backups` не в git
   (`.gitignore`), offsite — в приватное хранилище.
2. **Идемпотентность.** `deploy.sh`/`init-shop.sh`/`update.sh`/`backup.sh` безопасны при повторном
   запуске; `up -d` не пересоздаёт здоровое; миграции идемпотентны и аддитивны.
3. **Ничего магазино-специфичного в коде/скриптах (ADR-003).** Домен, имена БД/бакета, расписания,
   ретенция, каналы алертов — из `.env`. Скрипты одинаковы для THE CASE и Gang Auto.
4. **Mock-режимы не мешают эксплуатации.** Smoke/деплой/бэкап зелены без боевых ключей СДЭК/Storefront/
   внешнего S3 (MinIO локально). Алерты без `ALERT_WEBHOOK_URL` деградируют в лог.
5. **Статическая проверяемость в dev.** Всё, что можно — `shellcheck`/`bash -n`/Vitest без Docker/PG/
   браузера; интеграционное явно помечено «гоняется на VPS/CI» и автоматизировано там.
6. **Обратная совместимость данных (docs/02 чек-лист).** Обновление не ломает данные: только аддитивные
   миграции + обязательный бэкап перед накатом + health-gate + откат.

### ADR-015. Стратегия бэкапов и обновлений для single-VPS-магазина

**Контекст.** ADR-003 («1 магазин = 1 БД = 1 VPS») и `docs/02` («копировать-вставить» сторонним
человеком) задают эксплуатационную модель: один процесс `app` за Caddy, локальные PostgreSQL/MinIO в
compose. Этап 6 должен дать продакшн-готовность (бэкапы, мониторинг, обновления без простоев) **без**
оркестратора (k8s) и **без** внешних SaaS, оставаясь универсальным (всё через `.env`). Нужно
зафиксировать, как делаются бэкапы и как обновляется магазин, чтобы обновление не теряло данные.

**Решение.**
1. **Бэкап = логический дамп + зеркало медиа, не Docker-тома.** PostgreSQL — `pg_dump -Fc`; медиа —
   `mc mirror`. Расписание/ретенция — отдельный compose-сервис-планировщик (паттерн существующего
   `cron`), параметры из `.env`. **Восстановимость обязательна к проверке:** периодический
   `restore-drill` восстанавливает последний дамп в одноразовую БД и читает ключевые таблицы; «бэкап
   без проверенного restore — не бэкап». Offsite — опциональная env-настраиваемая точка (rclone/`mc`),
   провайдер не хардкодится.
2. **Обновление — near-zero-downtime в рамках одного VPS:** собрать новый образ заранее → обязательный
   бэкап → накат **только аддитивных** миграций (старый код совместим с новой схемой) → пересоздать
   `app` с **health-gate** (`/api/health` deep-check) → при провале **автоматический откат** на
   предыдущий образ. Деструктивные изменения схемы — только многошаговым **expand/contract** через
   несколько релизов, никогда в одном выкате с кодом, который от них зависит. Идемпотентность миграций
   (`IF NOT EXISTS`) **не равна** обратной совместимости — последняя обеспечивается правилом
   аддитивности и статической проверкой (`check-migrations.sh`).
3. **Мониторинг/логи — self-hosted уровня compose:** JSON-логи приложения в stdout (`lib/logger.ts`),
   deep-health-probe, лёгкий внешний монитор + алерты через webhook/email. Тяжёлый стек наблюдаемости
   (Prometheus/Grafana/Loki/Sentry) — точка расширения поверх готовых JSON-логов и health-probe, в
   базовую поставку не входит.

**Рассмотренные альтернативы.** *(а)* Полагаться на снапшоты тома/диска VPS вместо логических дампов —
отвергнуто: непереносимо между провайдерами, не даёт гранулярного restore, не проверяет целостность
схемы. *(б)* Blue-green/rolling на нескольких репликах — отвергнуто: противоречит ADR-003 (одна нода),
избыточно для админки ИМ. *(в)* Внешний SaaS-мониторинг по умолчанию — отвергнуто: добавляет внешнюю
зависимость и стоимость, ломает «copy-paste без аккаунтов»; оставлено как опция.

**Последствия.** Новые скрипты `scripts/{smoke,deploy,backup,restore,restore-drill,update,
check-migrations,healthcheck-monitor}.sh`, `Makefile`, сервис `backup` и том `backups` в compose,
`lib/logger.ts`, deep-режим `/api/health`. Новые env: `BACKUP_*`, `LOG_*`, `ALERT_WEBHOOK_URL`,
`SMOKE_BASE_URL`. **Правило аддитивных миграций становится обязательным** для всех будущих этапов
(линтер в CI). Все скрипты универсальны (ADR-003) и проверяемы статически в dev (shellcheck) +
интеграционно на VPS/CI.
