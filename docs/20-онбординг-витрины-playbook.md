# 20 — Плейбук онбординга: прикрутить админку Admik и БД к готовому макету магазина

> **Это пошаговый ранбук для автономной сессии Claude Code.** Цель — взять
> репозиторий Uni_BD (готовую платформу Admik) и **прикрутить её к уже
> существующему макету витрины** конкретного интернет-магазина, ретематизировав
> данные под нишу магазина и **не меняя бизнес-логику платформы**.
>
> Прежде чем начать: прочитай `START-HERE.md`, `CLAUDE.md`, `docs/21` (контракт API).
> Этот плейбук исполняется скиллом `/onboard-shop` (он оркеструет фазы субагентами),
> но ты можешь идти и вручную по фазам ниже.
>
> **Инварианты (нарушать нельзя):** TDD (сначала тест) · две проверки (код+браузер) ·
> мультитенантность (ноль хардкода) · миграции иммутабельны/аддитивны · секреты
> только из env · документировать каждую правку.

## Фаза 0 — Пред-полёт

1. **Найди макет витрины.** Ожидается, что готовый макет магазина лежит рядом —
   рекомендуемый путь `./storefront` (он уже прописан в `docker-compose.yml`
   как build-context и в `.gitignore`). Если макета нет или он в другом месте:
   - попроси владельца положить его в `./storefront`, **или**
   - уточни путь и поправь `context:` сервиса `storefront` в `docker-compose.yml`.
2. **Поставь зависимости платформы:** `pnpm install`.
3. **Заведи `.env`:** `cp .env.example .env`. Пока можно с дефолтами (mock-режим
   интеграций, локальные пароли) — финально заполнишь в фазе 3.
4. **Прочитай ключевое:** `docs/02` (модель развёртывания), `docs/09` (разворачивание),
   `docs/21` (контракт API), `docs/16` (стек). Для конкретных API библиотек (Next 16,
   React 19, Zod 4, postgres.js) — сверяйся через MCP **context7**, не по памяти.
5. **Короткое интервью с владельцем** (если ещё не было): ниша/ассортимент, какие
   модули нужны (`catalog,orders,cdek,cms,payments`), домены, реквизиты/контакты,
   валюта, префикс заказа, нужны ли боевые интеграции сейчас или mock.

**Готово, когда:** макет на месте, `pnpm install` прошёл, `.env` создан, интервью собрано.

## Фаза 1 — Анализ макета витрины → «профиль магазина»

Цель — понять, какие данные нужны магазину, чтобы спланировать ретематизацию и
сращивание. Запусти параллельных read-only исследователей (агент `Explore`) по
макету `./storefront`. Собери **профиль магазина**:

1. **Каталог:** какие категории/разделы; какие у товара поля (название, цена,
   старая цена, описание, бренд); есть ли **варианты** (размер/цвет/объём) →
   ложатся в `product_variants`; какие **атрибуты/характеристики** нужны (EAV:
   `attributes` + `attribute_values`); единицы (вес/габариты — для расчёта СДЭК).
2. **Навигация и страницы:** меню шапки/футера; статические страницы (О нас,
   доставка, оплата, оферта, контакты) → это CMS-страницы (`cms_pages`).
3. **Брендинг:** название, логотип, favicon, фирменные цвета, контакты, соцсети,
   реквизиты юрлица → `shop_settings.branding/contacts/legalEntity`.
4. **Главная страница:** блоки hero/about/«преимущества»/доставка → `shop_settings.home`.
5. **Корзина/оформление/оплата/доставка:** как сейчас устроено в макете (статика?
   свой бэкенд? localStorage?) — это будет заменено вызовами Storefront API.
6. **Стек витрины:** Next.js App/Pages Router? где статические данные каталога;
   где формы; есть ли свой Dockerfile (`output: standalone`, порт 3000).

**Сохрани профиль** во временный файл/заметку и зафиксируй краткое резюме в
`docs/00` (журнал). Он — вход для фаз 3 и 4.

**Готово, когда:** есть структурированный профиль магазина (категории, атрибуты,
варианты, страницы, брендинг, способ оформления/оплаты, стек витрины).

## Фаза 2 — Поднять БД + миграции + seed

Вариант А (рекомендуется, как на бою) — через Docker:
```bash
make up            # docker compose up -d: app + postgres + redis + minio + caddy
make init          # scripts/init-shop.sh ВНУТРИ контейнера app:
                   #   extensions+роли БД → миграции 0001..NNNN → seed permissions/roles
                   #   → seed владельца (OWNER_EMAIL/PASSWORD) → опц. demo-каталог
make smoke         # health-проверки /api/health и ключевых эндпоинтов
```
Вариант Б (локальная разработка без полного Docker): поднять Postgres/Redis/MinIO
(или dev-БД), экспортировать `DATABASE_URL`/`PG*`, прогнать `scripts/init-shop.sh`,
затем `pnpm dev`.

Что важно знать про `init-shop.sh` (идемпотентен, см. `docs/09`/`docs/12`):
- роли БД least-privilege: `admik_migrator` (DDL/владелец таблиц), `admik_app`
  (рантайм DML; на `audit_log` только SELECT/INSERT — append-only);
- миграции накатываются по порядку имён, журнал в `schema_migrations`;
- сид владельца: при пустом `OWNER_PASSWORD` пароль генерируется и печатается ОДИН раз;
- демо-каталог — только при `SEED_DEMO_CATALOG=true` (для smoke; на бою `false`).

**Проверь:** `GET /api/health` → `{"status":"ok"}`; вход в `/admin/login` под
`OWNER_EMAIL`; в админке открываются разделы включённых модулей.

**Готово, когда:** БД поднята, все миграции накатаны, владелец заходит в админку,
health зелёный.

## Фаза 3 — Ретематизация под нишу магазина (БЕЗ правок бизнес-логики)

Перенос на новый магазин = **конфигурация + данные**, не код (ADR-003; скилл `new-shop`).

### 3.1 `.env` под магазин
Заполни (см. `.env.example` и `START-HERE.md §6`): `SHOP_NAME`, `SHOP_DOMAIN`,
`ADMIN_DOMAIN`, `SHOP_ORDER_PREFIX`, `SHOP_CURRENCY`, `ADMIK_MODULES` (только нужные
модули), `STOREFRONT_ALLOWED_ORIGINS`/`STOREFRONT_API_KEYS`,
`NEXT_PUBLIC_ADMIK_API_URL`, доступы БД/MinIO/Redis. `CDEK_*`/`TBANK_*` — оставь
пустыми для mock, боевые ключи подставишь только на сервере.

### 3.2 Каталог под нишу
Два пути (выбери по ситуации):
- **Через админку (предпочтительно для боевого наполнения):** владелец заводит
  категории/атрибуты/бренды/товары в `/admin/catalog/*`. Твоя задача — убедиться,
  что структура (категории, варианты, атрибуты) из «профиля магазина» поддержана.
- **Через сид (для демо/smoke/быстрого старта):** адаптируй `db/seed/demo-catalog.sql`
  под нишу — это нейтральный идемпотентный пример (категории, атрибуты Цвет/Размер,
  товары, варианты, остатки на естественных ключах `slug/sku/code`). Замени
  нейтральные значения на категории/атрибуты/товары магазина. **TDD:** обнови
  соответствующий seed-тест в `tests/seed/` под новые данные, прогони `pnpm test`.
  Накат — под флагом `SEED_DEMO_CATALOG` через `init-shop.sh`.

> Структура каталога: EAV-атрибуты (`attributes`/`attribute_values`/
> `product_attributes`, `is_variant` для вариантных), варианты `product_variants`
> (цена `price_override`/`price_delta`, остатки в `inventory`), медиа `product_media`.
> Вес/габариты товара/варианта — для расчёта СДЭК (миграции 0018/0026).

### 3.3 Брендинг и контент через `shop_settings`
Через админку `/admin/settings` (разделы: Брендинг, Главная, Навигация и футер,
Валюта и единицы, Реквизиты и контакты, Каталог/доставка/заказы, Модули, SEO) задай
данные магазина из «профиля». Они сразу доезжают до витрины через `GET /settings`.
Кэш эффективных настроек инвалидируется автоматически при правке через UI; при
прямой правке БД нужен `docker compose restart app`.

### 3.4 CMS-страницы
Статические страницы магазина (О нас, доставка, оферта) заведи в `/admin/cms` —
витрина читает их через `GET /pages` / `GET /pages/:slug`.

**Готово, когда:** `.env` заполнен под магазин; структура каталога/атрибутов
соответствует нише; брендинг/контакты/навигация/главная заданы; CMS-страницы есть;
демо-сид (если используешь) ретематизирован и seed-тесты зелёные.

## Фаза 4 — Сращивание: подключить макет витрины к Storefront API

Переводим макет из «статики/своего бэкенда» в **чистого потребителя** API.
Полный контракт — `docs/21`. Метод и worked-example — `docs/13`.

### 4.1 Конфигурация витрины (Next.js)
- **Два адреса API намеренно:**
  - SSR (server components / route handlers витрины) → `http://app:3000` (внутри
    docker-сети), переменная `ADMIK_API_URL`;
  - браузерный бандл (checkout, автокомплит города) → `https://${ADMIN_DOMAIN}`,
    переменная `NEXT_PUBLIC_ADMIK_API_URL` (build-arg, «впекается» в сборку).
- **Авторизация:** для SSR — заголовок `X-Storefront-Key: <ключ>` (значение из
  `STOREFRONT_API_KEY`, и оно же в `STOREFRONT_API_KEYS` на стороне Admik); для
  браузера достаточно совпадения `Origin` ∈ `STOREFRONT_ALLOWED_ORIGINS`. Локально
  можно оставить mock-режим (ключи/origins пусты) — открыто всем.
- **Медиа:** изображения приходят готовыми публичными URL (через Caddy `/media/*`
  → MinIO). Дополнительная проксировка не нужна.

### 4.2 Замена данных макета на вызовы API (по маппингу `docs/21 §7`)
Пройди по макету и замени источники данных:
- статический список товаров → `GET /products` (+фильтры);
- карточка → `GET /products/:slug`; навигация → `GET /categories`/`GET /brands`;
- корзина-предпросмотр → `POST /cart/quote` (источник истины по цене/итогу);
- оформление → `POST /orders` (сохрани `accessToken` для ЛК/оплаты);
- трекинг/ЛК → `GET /orders/:number?token=`;
- доставка СДЭК → `cities`/`pvz`/`calculate`; оплата → `POST /payments/tbank/init`
  и редирект на `paymentUrl`;
- статические страницы → `GET /pages*`; шапка/футер/логотип/цвета/контакты/главная
  → `GET /settings`; промо-баннер → `GET /promotions`; формы → `POST /leads`/`/newsletter`.

**Anti-tamper:** НЕ отправляй цены/итоги/вес с витрины — сервер их игнорирует и
считает сам. Витрина передаёт только выбор (id/qty/промокод/адрес/способ).

### 4.3 Если в макете чего-то не хватает в API
Сверься с `docs/21`. Если действительно нет нужного публичного эндпоинта — это
расширение платформы: проектируй через агента `solution-architect`, реализуй по
TDD в `app/api/storefront/v1/*` + `lib/storefront/*` (сначала тест), аддитивно.
Не добавляй в витрину обходные пути со своей бизнес-логикой/своей БД.

**Готово, когда:** витрина рендерит каталог/карточку/категории из API; корзина
считается через `quote`; оформляется заказ; работает доставка и оплата (mock);
брендинг/контент тянутся из `/settings` и `/pages`.

## Фаза 5 — Две обязательные проверки

1. **Код-гейт** — скилл `/gate`: `pnpm typecheck` (0) · `pnpm lint` (0) ·
   `pnpm test` (vitest, все зелёные) · `scripts/check-migrations.sh` (аддитивность) ·
   `security-review` при правках оплаты/RBAC/токенов/webhook/секретов. ВСЕГДА перед коммитом.
2. **Живой прогон в браузере** — скилл `/deploy-stand` (на стенде) или `/verify` +
   MCP **playwright** локально: пройди путь покупателя (каталог → карточка → корзина
   → оформление → доставка СДЭК → оплата mock) и админку (логин → завести товар →
   увидеть его на витрине). Только mock-режим интеграций.
3. **Интеграции** — скилл `/integration-check`: e2e СДЭК + Т-Банк в mock end-to-end.
4. **Smoke** — `make smoke` / `scripts/smoke.sh`: health + ключевые эндпоинты.

**Готово, когда:** обе проверки зелёные (код + браузер), integration-check и smoke прошли.

## Troubleshooting

| Симптом | Причина / решение |
|---|---|
| `401` от Storefront API | Нет ключа/Origin. Задай `STOREFRONT_API_KEYS` + `X-Storefront-Key` (SSR) или `STOREFRONT_ALLOWED_ORIGINS` (браузер); локально — оставь оба пустыми (mock). |
| Раздел админки/эндпоинт `404`/`503` | Модуль выключен. Включи в `ADMIK_MODULES` (рестарт) или `shop_settings.module_overrides` (скилл `/module-toggle`). |
| Правка настройки не видна на витрине | Прямая правка БД не инвалидирует кэш → `docker compose restart app`. Через UI — инвалидируется автоматически. |
| `check-migrations.sh` падает | Ты отредактировал/добавил деструктивную миграцию. Миграции иммутабельны и только аддитивны (без DROP/RENAME/TYPE/SET NOT NULL без DEFAULT). |
| Цена/итог на витрине не совпадает | Витрина шлёт свои суммы — сервер их игнорирует. Источник истины — `quote`/`orders`. Убери клиентский расчёт. |
| СДЭК/оплата «не настоящие» локально | Это mock (пустые `CDEK_*`/`TBANK_*`) — так и задумано. Demo-оплата — `app/mock/tbank/pay`. Боевые ключи только на сервере. |
| `next build` OOM на слабом VPS | Включи swap (`server-bootstrap.sh`), см. `docs/09`/`docs/12`. |

> Возобновление прерванной сессии: `docs/10` + `docs/00`. Карта инструментов
> (MCP/скиллы/агенты) — `docs/19` и низ `CLAUDE.md`.
