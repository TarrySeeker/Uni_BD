## Краткое резюме проекта

**Admik** — универсальная, переиспользуемая административная панель для интернет-магазинов (ИМ).
Проектируется как CMS/ERP-платформа для сети магазинов: единая кодовая база, отличия между магазинами — только в конфигурации.

**Ключевая модель развёртывания:** один ИМ = одна БД. Перенос на новый магазин — по принципу «копировать-вставить» (его будут выполнять другие люди, поэтому процесс обязан быть максимально простым). Подробно — `docs/02-модель-развёртывания.md`.

## Принятые решения

- **Стек:** Next.js 16 (App Router) + React 19 + TypeScript (strict) + PostgreSQL + Lucia Auth + Docker + Caddy + S3/MinIO. См. `docs/01-архитектурные-решения.md`.
- **Источник паттернов админки:** проект 2x2 (Server Actions, ролевая модель, CMS, аудит, безопасность).
- **Источник логики СДЭК:** проект carre (Manager→Client→Services, webhook-идемпотентность, cron-воркеры).
- **Мультитенантность:** 1 магазин = 1 БД; код single-tenant в рантайме; универсальность через конфиг. См. `docs/02-модель-развёртывания.md`.
- **Принцип:** «сначала тесты, потом код» (из CLAUDE.md).
- **Оркестрация:** ведётся через универсальный агент `claude` с встроенной ролью. Кастомные типы агентов из `.claude/agents/` не зарегистрированы в текущем харнессе — это ограничение среды, не проекта (состав агентов в `.claude/agents/` сохраняется как спецификация ролей).
- **RBAC (ADR-005):** гибкая модель «роли → права» (`users—user_roles—roles—role_permissions—permissions`); проверки в коде идут по правам, не по именам ролей.
- **Auth (ADR-006):** Lucia-подход — сессии в БД (без JWT), хеширование паролей argon2id, две роли БД (`app`/`migrator`), `audit_log` append-only на уровне прав БД.
- **Crash-recovery:** git-коммиты на вехах (на каждой волне работ) используются как механизм восстановления при падении сессии (по требованию пользователя о защите от падения). Документация (docs/00–04) — основной источник восстановления контекста.
- **e2e Playwright вынесен в отдельный скрипт `test:e2e`:** браузерные e2e-тесты (a11y/навигация каркаса админки) не входят в обязательный прогон `vitest` и гоняются отдельно в CI/проде. Это держит локальный `vitest` зелёным в окружении без браузера.
- **Криптопримитивы сессий — на `node:crypto`:** пакет `oslo` (изначально предложенный в ADR-001/спецификации Этапа 1) отклонён как deprecated; криптослучайные id сессий и константное сравнение реализованы стандартной библиотекой Node.
- **ADR-008 (Admik = headless commerce backend + публичный Storefront API):** платформа «приставляется» к внешним SPA-витринам (Netlify), а не несёт собственную витрину. Витрины ходят в платформу через публичный **Storefront API `/api/storefront/v1/*`** (read-каталог: products/categories/brands; в будущем — заказ и СДЭК-расчёт/ПВЗ). Аутентификация витрины (API-ключ/домен), CORS, версионирование, rate-limit, DTO без утечки внутренних данных (наружу отдаётся `inStock`, а не точный остаток). Подтверждено двумя разными тематиками (медформа vs автозапчасти) на одной кодовой базе — различия только в данных/настройках (ADR-003). См. `docs/06-референсные-магазины.md`.
- **ADR-009 (ценообразование/флаги/бренды каталога — минимально, универсально, опционально):**
  - **`compare_at_price`** (в `products` и `product_variants`) — каталожная цена «было»; **скидка % и предикат «со скидкой» вычисляются, не хранятся** (нет рассинхрона). Выбрано единое поле вместо пары `old_price`/`sale_price`; источник истины актуальной цены — `base_price`/вариантные поля.
  - **Флаги товара:** `is_featured` — ручной маркетинговый «Хит/Рекомендуемый» (вычислить нельзя); `is_new` — **троичная логика** (`NULL` → новизна вычисляется по `created_at` и настройке `SHOP_NEW_PRODUCT_DAYS`, дефолт 30; `true/false` → явный override редактора); «со скидкой» — производная цен, отдельным флагом не хранится.
  - **Бренды — отдельной опциональной таблицей `brands`** + `products.brand_id` (nullable, `ON DELETE SET NULL`), а не EAV: бренд — сущность с лого/описанием/страницей/фасетом; пуста для магазинов без брендов (THE CASE).
  - **VIN/совместимость (fitment)** — только точка расширения: будущий опциональный модуль `fitment` (обобщённый, не «авто»-хардкод), в `0011` не реализуется.
  - **Валюта** — настройка магазина `SHOP_CURRENCY` (дефолт `'RUB'`, ₽), форматирование — в слое представления/Storefront API; в данных каталога валюта не хардкодится; поля валюты в `0011` не добавляются (одна валюта на магазин — ADR-003).
- **Технический нюанс `'use server'`:** модуль с директивой `'use server'` может экспортировать **только async-функции**. Поэтому класс `CatalogError` вынесен из `actions.ts` в отдельный `lib/catalog/errors.ts` (а не-action-экспорты не размещаются в server-action-модулях). На Этапе 3 аналогично — `lib/orders/errors.ts`.
- **ADR-010 (anti-tamper + снимки позиций заказа):** **итог заказа считает только сервер** — витрине доверяем лишь `variantId`+`qty`+выборы доставки/промокода, все деньги пересчитываются заново из каталога (защита от подмены цены на клиенте). `POST /cart/quote` и `POST /orders` считают итог независимо; цена из тела запроса игнорируется. Позиции заказа (`order_items`) — **денормализованный снимок** (name/sku/price/attributes на момент покупки): изменение каталога не переписывает историю; ссылки `product_id`/`variant_id` — `ON DELETE SET NULL`. ADR-010 добавлен в конец `docs/01`.
- **Деньги в заказах — целые копейки (без float):** денежные расчёты домена `orders` ведутся в **целых копейках** (`lib/orders/money.ts`), что исключает ошибки округления float; в БД деньги — `numeric(14,2)` (как в каталоге).
- **Резерв остатков — атомарным `UPDATE inventory`:** резерв при создании заказа выполняется `UPDATE inventory SET reserved = reserved + qty WHERE … AND quantity - reserved >= qty` с проверкой числа затронутых строк (`= 1`), всё в одной транзакции. Конкурентные заказы на последний остаток: только один резервирует, второй получает `0` строк → откат транзакции (гонко-безопасно, без явных блокировок/дедлоков).
- **Доступ к трекингу заказа (`GET /orders/{number}`)** — по **HMAC-токену** (ключ `APP_PASSWORD`) либо по совпадению email; защита от перебора чужих номеров заказов.
- **Стоимость доставки на Этапе 3 — заглушка `0`:** реальный расчёт доставки (СДЭК) — **Этап 4**; на Этапе 3 в `quote`/создании заказа стоимость доставки фиксируется заглушкой (0 / порог бесплатной доставки), поля под доставку (тип/адрес/ПВЗ/стоимость/трек/`delivery_status`) уже заложены в схеме `0012`.
- **Способ оплаты на Этапе 3 — фиксация + mock:** на Этапе 3 фиксируются только способ и статус оплаты (`payment_method`/`payment_status`/`paid_at`/`payment_ref`); реальные платёжные провайдеры (карты/СБП/СДЭК PAY/эквайринг) — будущий под-этап, подключаются аддитивно без рефактора модели заказа.
- **ADR-011 (СДЭК: слоистый порт carre + mock-режим + идемпотентный webhook):** логика СДЭК портирована из carre (PHP/Yii2) на TS/Next.js со слоистой архитектурой **Manager → Client → Services**. `Client` инкапсулирует HTTP/OAuth и кеш токена; `Services` (calculator/pvz/order/tracking/webhook/print/status-map/delivery-status) — доменная логика; `Manager` — фасад для остального кода. **Mock-режим** определяется конфигом (`isCdekMock`): при отсутствии боевых ключей CDEK_* клиент работает на заглушках, реальная сеть не дёргается. **Webhook идемпотентен** (дедуп через таблицу `cdek_status_log`), всегда отвечает `200`, защищён IP-whitelist (+CIDR). Развязка `orders ↔ cdek` — ленивым импортом адаптера `lib/orders/delivery-cost.ts` (заменил заглушку доставки `0` из Этапа 3). ADR-011 добавлен в `docs/01`; спецификация — `docs/08-этап4-cdek.md`.
- **Никогда не копировать чужие секреты (СДЭК/платёжки):** ключи СДЭК и платёжных провайдеров **принадлежат третьим лицам** — запрещено копировать их или вносить в репозиторий. В коде только mock-режим и env-плейсхолдеры (`CDEK_*`); боевые ключи подставляют владельцы магазина при развёртывании. Реальных ключей СДЭК в репозитории нет.
- **ADR-014 (промо-механики «N по M») — реализован.** Акции N×M (BOGO/scope-скидка) реализованы аддитивно поверх задела заказов Этапа 3, без рефактора существующих `percent`/`fixed`/`free_delivery`. Схема: `ALTER promo_codes` (`apply_scope`/`priority`/`stackable`/`min_qty`/`gift_*` + CHECK согласованности BOGO-пары) + таблица `promo_targets` (FK CASCADE/CHECK/UNIQUE/INDEX), миграция `0024`. Расчёт — **чистые детерминированные функции в копейках** (`bogoDiscountMinor`/`scopeDiscountMinor`/`combineDiscountsMinor`, `lib/orders/pricing.ts`), интегрированные в `quoteCart`/`createOrder` (резолв целей; итог считает только сервер — ADR-010). Admin UI формы промокода (apply_scope/targets/gift_*/min_qty) + публичный `GET /api/storefront/v1/promotions`. Подробно — `docs/01` и `docs/11`. **MVP-ограничение:** применяется один промокод на заказ (множественные/`stackable` отложены — см. «Отложенные задачи»).
- **ADR-015 (стратегия бэкапов/обновлений/мониторинга для single-VPS) — реализован (Этап 6).** Для модели «один магазин = один VPS» приняты три решения:
  - **Бэкап = логический дамп + restore-drill.** Резервное копирование PostgreSQL — логическим дампом (`scripts/backup.sh`), восстановление проверяется регулярным «учебным» прогоном (`scripts/restore-drill.sh`), а не только наличием файла бэкапа; сервис `backup` в `docker-compose` + том `backups` (вне git). Медиа S3/MinIO — по тому же регламенту.
  - **Обновление = аддитивные миграции + health-gate + откат.** Обновление кода существующего магазина не ломает данные: миграции **только аддитивны** (это проверяет линтер `scripts/check-migrations.sh` — запрещает деструктивные `DROP`/несовместимые `ALTER`, statement-level матчинг после fix `484e41e`); `scripts/update.sh` применяет обновление за **health-gate** (`/api/health?deep=1`) и при провале **автоматически откатывает**.
  - **Мониторинг/логи — self-hosted JSON.** Структурированные JSON-логи (`lib/logger.ts`) с ротацией `json-file` в compose; health с deep-режимом (`lib/health.ts`, `/api/health?deep=1` — проверка БД/Redis/S3); алерты на падение — `scripts/healthcheck-monitor.sh` + `ALERT_WEBHOOK_URL` (без внешнего APM на старте; offsite/Prometheus — точки расширения из бэклога). Санитайзер чувствительных полей вынесен в `lib/audit/sanitize.ts`. Подробно — `docs/01` (ADR-015) и `docs/09`.
- **Процессный кэш с инвалидацией — только на `globalThis` под registry-Symbol (2026-06-24).** Любое in-memory-состояние, которое инвалидируется из одного места кода, а читается из другого (особенно через границу Server Actions ↔ Route Handlers/RSC), **нельзя** хранить в модульных `let`: Next.js может инстанцировать один и тот же модуль в **разных серверных бандлах**, и инвалидация в одном бандле не видна другому. Эталон — кэш `getEffectiveSettings` в `lib/config/settings.ts`: состояние (`cached`/`inflight`/`cacheEpoch`) вынесено на `globalThis` под `Symbol.for('admik.config.settings.effectiveCache')` → один кэш и одна инвалидация на весь процесс. **Оговорка:** прямая правка `shop_settings` через SQL не вызывает `invalidateSettingsCache()` и по-прежнему требует `docker compose restart app`; фикс покрывает только правки через UI/экшены приложения.

  **Корень.** Memo-кэш `getEffectiveSettings` (`cached`/`inflight`/`cacheEpoch`) хранился в **модульных** `let` в

  Route Handlers/RSC) → `invalidateSettingsCache()` из экшена сбрасывал memo своего экземпляра, а публичный
  route-handler `/settings` читал memo **другого** экземпляра и бесконечно отдавал устаревший снимок (до рестарта app).

  **Фикс.** Состояние кэша вынесено на `globalThis` под registry-Symbol `Symbol.for('admik.config.settings.effectiveCache')`
  → один кэш и одна инвалидация на весь процесс, общие для всех бандлов. Добавлен guard «снимаем только свой in-flight».
  Файлы: `lib/config/settings.ts` (правка), `tests/config/settings.test.ts` (новый регресс-тест «кэш живёт на globalThis»).

  **Оговорка.** Прямая правка `shop_settings` через SQL по-прежнему требует `docker compose restart app` (она не вызывает
  invalidate). Фикс покрывает только правки через UI/экшены приложения.

  **Две проверки (принцип проекта).** (1) **Код-гейт:** `tsc --noEmit` чисто, полный `vitest run` = **1660 passed / 105
  skipped**. (2) **Живой прогон на стенде** (Playwright): логин → правка «О бренде» в админке → публичный API и витрина
  отражают правку **без рестарта** → исходное значение восстановлено.

  **Деплой.** rsync фикса на стенд `/opt/admik` → пересборка образа `admik-app:current` → `docker compose up -d app` →
  health-gate `?deep=1` = ok (db/redis/s3). Точка отката: образ помечен `:rollback`, предеплойный дамп БД в
  `/opt/admik/backups/`.
