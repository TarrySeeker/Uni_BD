# Инструментарий: MCP-серверы, скиллы, агенты (кит Uni_BD / платформа Admik)

> Единая политика авто-применения MCP-серверов, скиллов и привязки инструментов к субагентам.
> Краткая выжимка — в `CLAUDE.md` (блок «Инструментарий»). Здесь — подробно.
> Принцип: MCP и скиллы — часть обязательного флоу (две проверки, TDD, сверка документации), а не «по желанию».

---

## 0. Механика Claude Code (по официальной документации)

Источники: `code.claude.com/docs/en/subagents`, `.../mcp-servers`, `.../skills`, `.../hooks`.

- Поле `tools` во frontmatter субагента — **comma-separated allowlist**. Если поле **опущено** — агент наследует ВСЕ инструменты (built-in + все MCP). Если **указано** — это исключающий список: при добавлении MCP нужно перечислить заново и старые tools, и новые MCP-записи.
- Синтаксис MCP в `tools`:
  - **Весь сервер:** `mcp__<server>` либо `mcp__<server>__*` (эквивалентны).
  - **Поштучно:** `mcp__<server>__<tool>` (точные имена доступны только при подключённом сервере).
  - Wildcard `mcp__*` работает в `disallowedTools` (запрет всех MCP), в `tools` — нет.
- В allowlist агентов используется форма «весь сервер» (`mcp__<server>`): не зависит от точных имён тулзов. Прописать `mcp__<server>` в allowlist ДО фактического подключения сервера безопасно — тулзы просто неактивны и включатся автоматически после старта сервера (именно так заранее заложен опциональный `postgres` — см. §1.5).
- **Авто-применение** обеспечивается ИНСТРУКЦИЯМИ (этот документ + блок в `CLAUDE.md` + промпты агентов), которые загружаются в контекст; hooks для тяжёлых проверок НЕ используются (см. §5).

## 1. MCP-серверы

Кит поставляет **4 MCP-сервера**, объявленных в `.mcp.json` (project scope, checked in): **playwright**, **context7**, **firecrawl**, **glyph**. Тот же список продублирован в `.claude/settings.json` (`enabledMcpjsonServers`). Секретов в `.mcp.json` нет — ключ firecrawl подставляется из окружения (`${FIRECRAWL_API_KEY:-}`).

### 1.0 Как включаются MCP — честные нюансы для владельца (нетех)

Код трогать не нужно, но есть 5 практических нюансов; без них серверы могут «молча не подняться»:

1. **Trust-диалог.** Закоммиченный `.claude/settings.json` перечисляет 4 сервера, но Claude Code применяет их **только после того, как ты один раз подтвердишь доверие к папке** (диалог доверия при первом открытии каталога). До подтверждения серверы неактивны.
2. **Быстрый путь без ожидания — `.claude/setup.sh`.** Скрипт пишет `.claude/settings.local.json` (личный файл, не коммитится) с тем же списком серверов; он действует **сразу**, даже до trust-диалога. Запусти один раз после клонирования: `bash .claude/setup.sh`. Он же проверит предпосылки (Node, ключ firecrawl, бинарь glyph).
3. **Node.js обязателен для трёх серверов.** playwright, context7, firecrawl запускаются через `npx` — им нужен установленный **Node.js ≥ 20**. Нет Node → эти три сервера не поднимутся.
4. **glyph — опциональный бинарь.** Запускается не через npx, а как локальная команда `glyph`. Нет бинаря в PATH → сервер просто не активируется, сессия полноценно работает без него.
5. **firecrawl — опциональный ключ.** Серверу нужен `FIRECRAWL_API_KEY` в переменных окружения. Без ключа сервер не поднимется — это не критично: он нужен только для веб-ресёрча (доки СДЭК/Т-Банк, импорт каталога).

> Итог для владельца: `bash .claude/setup.sh` → при первом открытии подтвердить доверие к папке → (по желанию) задать `FIRECRAWL_API_KEY` и поставить `glyph`. playwright и context7 работают сразу при наличии Node.js.

### 1.1 playwright — браузерная автоматизация
**Транспорт:** `npx @playwright/mcp@latest --browser chromium` (см. `.mcp.json`). Нужен Node.js; браузер Chromium докачивается при первом запуске.

**Авто-триггеры (ОБЯЗАН):**
- После ЛЮБОЙ правки UI витрины/админки — «ручная проверка» из правила двух проверок.
- При закрытии задачи с визуальным/интерактивным результатом: форма, кнопка, статус заказа, корзина, оплата, фильтр каталога.
- После деплоя на стенд — smoke ключевых путей на реальном URL.

**Примеры задач:** оформление заказа каталог→корзина→подтверждение; подписка на рассылку; форма обратной связи; demo-страница оплаты Т-Банка в mock и возврат на success/fail; фильтры каталога и фото из объектного хранилища (S3/MinIO); смена статуса заказа отражается на витрине; один и тот же сценарий на двух магазинах-инстансах (нет хардкода под конкретный магазин).

**Не применять / предостережения:** не замена Vitest для логики; не гонять по проду с реальной оплатой/СДЭК — только mock; не вводить реальные секреты; флаку изолировать, не «чинить» отключением проверки.

### 1.2 context7 — актуальная документация библиотек
**Транспорт:** `npx -y @upstash/context7-mcp` (см. `.mcp.json`). Нужен Node.js; обязательного ключа нет.

**Авто-триггеры (ДОЛЖЕН):**
- Перед использованием API свежих зависимостей: Next.js (App Router, Server Actions), React, Zod, postgres.js, ioredis, TanStack Table, Tiptap, React Hook Form, @dnd-kit.
- При написании теста ДО кода (TDD), если контракт библиотеки неочевиден — сверить до фиксации ожидания.
- При ошибке типов/рантайма, похожей на смену API между версиями.

**Не применять / предостережения:** не для бизнес-логики платформы (домен — в `docs/` и промптах агентов); не подменять чтение нашего кода; примеры с хардкодом/секретами адаптировать под мультитенантность и env.

### 1.3 firecrawl — web-скрейпинг/краулинг
**Транспорт:** `npx -y firecrawl-mcp` (см. `.mcp.json`). Нужен Node.js **и** ключ `FIRECRAWL_API_KEY` в окружении — без ключа сервер не поднимается (см. §1.0.5). Ключ — только из env, не в репозитории; при случае ротировать.

**Авто-триггеры (ДОЛЖЕН):**
- Чтение онлайн-документации интеграций: API СДЭК (расчёт, ПВЗ, создание/трекинг отправлений, статусы, печать) и Т-Банк (оплата card/SBP, формат уведомлений).
- Импорт данных каталога: товары, характеристики, фото-источники.

**Не применять / предостережения:** не краулить большие сайты широким захватом — точечные страницы; ключ только из env, не в логи; скрейп товаров — мультитенантно (в конфиг/данные конкретного магазина); уважать robots/ToS.

### 1.4 glyph — символьные outline кода (tree-sitter)
**Транспорт:** локальная команда `glyph mcp` (см. `.mcp.json`). **Опциональный бинарь:** есть `glyph` в PATH → сервер поднимается; нет — сессия работает без него. Установка (по желанию, нужен Go-тулчейн): `go install github.com/benmyles/glyph@latest`, либо иным способом из проекта glyph.

**Авто-триггеры (ДОЛЖЕН, когда установлен):** навигация по большому TS-репо перед правкой (outline вместо чтения целиком); поиск точек расширения при добавлении модуля; трассировка паттерна мутаций по символам.
**Предостережения:** outline не заменяет чтение критичного участка; источник истины по API — context7, по поведению — playwright/тесты. В allowlist субагентов glyph не выдаётся — это инструмент навигации оркестратора (главной сессии).

### 1.5 postgres — read-only MCP к БД (опциональный, НЕ в поставке)
**Статус:** в `.mcp.json` кита его нет. Подключается **per-shop**: read-only ролью к БД конкретного магазина, в local scope, с кредами из env. Пакет — `@modelcontextprotocol/server-postgres` (read-only by design). Заранее прописан в allowlist субагентов backend/qa/security (§3) — активируется автоматически, если магазин добавит сервер.

**Как подключить (для магазина):** завести read-only роль в БД магазина → `claude mcp add --scope local postgres …` со строкой подключения из окружения (не в репозиторий).

**Авто-триггеры (после активации):** отладка реальных данных при баге заказа/каталога; сверка схемы перед миграцией; подтверждение, что мутация записалась и `audit_log` заполнен.
**Предостережения:** ТОЛЬКО read-only — запись лишь через Server Actions; креды из env; не тащить персональные данные клиентов в логи/контекст.

## 2. Скиллы (slash-команды)

### 2.1 Встроенные — когда применять автоматически
| Скилл | Когда применять автоматически | Правило |
|---|---|---|
| **verify** / **run** | После правки бизнес-логики/UI — вторая из двух проверок (реальный прогон). | две проверки |
| **code-review** | Перед коммитом на рабочий diff; effort `high`+ для денег/RBAC/оплаты. | качество diff |
| **security-review** | Перед коммитом в путях оплаты, RBAC, токенов заказа, webhook СДЭК/Т-Банк, `.env`/секретов. | секреты из env |
| **deep-research** | Сверка актуального API интеграций, когда context7 недоступен (СДЭК/Т-Банк, миграции библиотек). | интеграции |
| **init** | Заведение нового магазина-инстанса — генерация/обновление CLAUDE.md инстанса. | мультитенантность |

Ситуативные (`simplify`, `fewer-permission-prompts`, `loop`, `schedule`, `update-config`, `review`, `claude-api`) в авто-флоу не вшиты.

Кит распределяет по субагентам 3 из 4 поставляемых MCP (playwright, context7, firecrawl); **glyph** остаётся инструментом оркестратора и субагентам не выдаётся. Значения отражают фактические allowlist `tools` в `.claude/agents/*.md`.

| project-manager | — | — | — | — | координация; MCP не нужен |

## 4. Порядок одобрения / активации

**MCP scope:** `--scope local` (приватно этому пользователю в проекте) · `--scope project` (`.mcp.json`, checked in) · `--scope user` (`~/.claude.json`, глобально).

- **4 поставляемых сервера** (project scope, `.mcp.json`): включаются после разового trust-диалога при первом открытии папки. Быстрый путь без ожидания — `bash .claude/setup.sh` (пишет `.claude/settings.local.json`, действует сразу). `claude mcp list` покажет статус.
- **firecrawl** дополнительно требует `FIRECRAWL_API_KEY` в окружении (§1.3); **glyph** — установленного бинаря `glyph` (§1.4). Оба опциональны: без них остальные серверы работают.
- **postgres** (опциональный, не в поставке): подключается per-shop read-only ролью к БД магазина в local scope с кредами из env (§1.5).

`.mcp.json` безопасен для коммита — секретов нет (ключ firecrawl берётся из env). Любые креды (firecrawl, postgres) держим вне репозитория, только в окружении.

## 5. Hooks — почему НЕ внедряем
Hooks — детерминированные действия харнесса вне модели. Авто-`PostToolUse: Edit|Write → pnpm test/lint` для платформы вреден: длинный vitest-сьют блокирует каждый Edit, flaky-e2e сорвут поток. Правило платформы — гейт перед **коммитом** (скилл `gate`) + CI, а не после каждого Edit. Деплой/прямой SQL на стенд owner-gated — авто-hook здесь опасен (риск `rsync --delete` без `.env`). Поэтому авто-применение держим на инструкциях (CLAUDE.md + промпты агентов + этот документ), а не на hooks.

## 6. Инвариант
MCP и скиллы дополняют, но не отменяют правила платформы: TDD (сначала тест), две проверки (код + браузер), мультитенантность (ноль хардкода под один магазин), секреты только из env, документирование каждой правки.
