# Модель развёртывания и переноса на новый магазин

> **Ключевое требование пользователя:** в будущем БД под новый магазин будут разворачивать **другие люди**.
> Поэтому процесс обязан быть **«копировать-вставить»** — простым, повторяемым, без ручной правки кода.

---

## Принцип

**Один интернет-магазин = одна БД = одна конфигурация.**
Кодовая база универсальна и **не меняется** при подключении нового магазина. Меняются только:

1. **Конфигурация** (`.env` + таблица настроек в БД) — реквизиты, ключи интеграций, включённые модули, брендинг.
2. **Содержимое БД** — каталог, контент, настройки конкретного магазина.

Запуск нового магазина = развёртывание того же кода + создание чистой БД + заполнение конфига. **Переписывания бизнес-логики нет.**

---

## Требования к «copy-paste» развёртыванию

Чтобы сторонний человек смог поднять новый магазин без участия разработчика:

1. **Один скрипт инициализации.** `./scripts/init-shop.sh` (или `make init`) создаёт БД, накатывает миграции, заполняет seed, создаёт владельца.
2. **Идемпотентные миграции.** Все `CREATE ... IF NOT EXISTS`; повторный запуск безопасен. Накат на любую пустую БД — одной командой.
3. **Шаблон конфига.** `.env.example` со всеми переменными и понятными комментариями; человек копирует в `.env` и заполняет.
4. **Никаких хардкодов магазина в коде.** Название, логотип, контакты, юрлицо, цвета, ключи СДЭК/платежей/S3 — только в `.env`/настройках.
5. **Docker «из коробки».** `docker compose up -d` поднимает весь стек (app, postgres, redis, minio, caddy). Минимум предварительных требований на сервере.
6. **Проверка готовности.** `/api/health` + скрипт smoke-теста подтверждают корректный запуск.
7. **Подробная инструкция.** `docs/09-инструкция-разворачивания.md` — пошагово, языком не-разработчика (создана на Этапе 6, пакет 6.5).
8. **Бэкап из коробки.** Compose-сервис `backup` делает дамп БД (`scripts/backup.sh`) и зеркало медиа по расписанию (`BACKUP_CRON`) с ретенцией; восстановление проверяется учебным прогоном (`scripts/restore-drill.sh`). Ручной бэкап — `scripts/backup.sh`, восстановление — `scripts/restore.sh`. См. ADR-015.
9. **Обновление одной командой с откатом.** `scripts/update.sh` обновляет магазин near-zero-downtime: новый образ → обязательный бэкап → только-аддитивные миграции → health-gate (`/api/health?deep=1`) → автоматический откат при провале. См. ADR-015, §6.4 в `docs/12`.
10. **Подтверждение готовности (smoke).** `scripts/smoke.sh` (`make smoke`) даёт объективное «готово/не готово»: проверяет `/api/health` и ключевые публичные эндпоинты после `up`+`init`; ловит «поднялось, но не инициализировано».
11. **Логи и алерты.** Структурные JSON-логи приложения в stdout (`docker compose logs` / `make logs`); опциональный внешний монитор (`scripts/healthcheck-monitor.sh`) и алерты через `ALERT_WEBHOOK_URL` (Telegram/Slack/email), без внешнего SaaS.

## Процесс переноса на новый магазин (целевой)

```text
1. Скопировать репозиторий (git clone / архив).
2. cp .env.example .env  и заполнить значения нового магазина.
3. docker compose up -d            # поднять инфраструктуру (или: make up)
4. ./scripts/init-shop.sh          # миграции + seed + создание владельца (или: make init)
5. make smoke                      # подтверждение готовности (или: curl /api/health)
6. Зайти в /admin, сменить пароль, загрузить логотип, включить нужные модули.
7. (опционально) Заполнить каталог импортом или вручную.
8. (опционально) ./scripts/backup.sh  # убедиться, что бэкап делается (далее — автоматически по расписанию).
```

Цель: шаги 1–5 — буквально copy-paste команд (есть `make`-алиасы: `up`/`init`/`smoke`/`deploy`/`logs`/`backup`/`update`); шаги 6–7 — через UI админки. Пошаговое руководство языком не-разработчика — `docs/09-инструкция-разворачивания.md`.

## Что это накладывает на разработку (чек-лист для всех агентов)

- [ ] Любая новая настройка магазина выносится в `.env` и/или таблицу настроек, а не хардкодится.
- [ ] Любая новая таблица создаётся идемпотентной миграцией.
- [ ] Любой новый модуль имеет флаг включения/выключения в конфиге.
- [ ] Любой секрет читается из переменной окружения; в репозитории — только `.env.example`.
- [ ] Любая зависимость от внешнего сервиса имеет mock-режим (для магазинов без боевых ключей).
- [ ] Обновление кода существующего магазина не ломает данные: миграции **только аддитивные** (`ADD COLUMN`, `CREATE ... IF NOT EXISTS`); деструктив (`DROP COLUMN/TABLE`, `RENAME`, сужение типа, `SET NOT NULL` без дефолта) — только многошаговым **expand/contract** через несколько релизов, никогда в одном выкате с зависящим от него кодом. Идемпотентность (`IF NOT EXISTS`) ≠ обратная совместимость. Проверяется CI-гейтом `scripts/check-migrations.sh` (см. ADR-015, `docs/12` §6.4).
