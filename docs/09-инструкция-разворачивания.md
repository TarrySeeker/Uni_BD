# Инструкция по разворачиванию магазина (для не-разработчика)

> **Для кого эта инструкция.** Для человека **без опыта разработки**, которому нужно поднять
> новый интернет-магазин на отдельном сервере (VPS). Каждый шаг — это **одна команда**, описание
> **что должно произойти** и **что делать, если пошло не так**. Жаргон расшифрован по ходу.
>
> **Принцип проекта (ADR-003, `docs/02`).** Один магазин = один сервер = одна база данных. Код
> **не меняется** под магазин — всё настраивается через файл `.env` и через админку. Поэтому
> разворачивание сводится к копированию команд.
>
> **Топология (один VPS, два домена).** На одном сервере поднимаются **две** части магазина под
> **одним вашим доменом** по схеме поддоменов (`docs/13`):
> - **витрина THE CASE** — публичный сайт магазина на `${SHOP_DOMAIN}` (и `www.${SHOP_DOMAIN}`);
> - **Admik** — админка, публичный API и медиа (картинки товаров) на поддомене
>   `admin.${SHOP_DOMAIN}` (переменная `ADMIN_DOMAIN`).
>
> Обе части — отдельные контейнеры в одном стеке `docker compose`; БД (PostgreSQL) и хранилище
> файлов (MinIO) живут на этом же сервере. Внешние хостинги (Netlify и т. п.) **не используются**.
> Веб-сервер Caddy сам выпускает бесплатные TLS-сертификаты для обоих имён (HTTPS из коробки).
>
> **Если совсем застряли** — см. последний раздел «Когда звать разработчика».

## 1. Что вам понадобится

Перед началом подготовьте четыре вещи. Без них дальше не пройти.

| Что | Зачем | Как получить |
|---|---|---|
| **VPS с Linux** | Сервер, на котором будут жить и витрина, и Admik. Минимум 2 ГБ RAM и 2 ядра (рекомендуется больше — на VPS со скромной памятью сборка образов спасается swap-файлом, его включает скрипт подготовки). | Купить у любого хостинг-провайдера (Timeweb, Selectel, и т.п.). При заказе выбрать **Ubuntu 24.04**. |
| **Домен** | Адрес магазина в интернете (например `myshop.ru`). Нужны **три** A-записи на IP вашего VPS (см. ниже): `myshop.ru`, `www.myshop.ru` и `admin.myshop.ru`. | Купить у регистратора доменов. A-записи создаются в настройках домена (DNS). |
| **Доступ по SSH** | Способ войти в командную строку сервера. | Провайдер пришлёт IP-адрес, логин (обычно `root`) и пароль/ключ. Войти: `ssh root@IP-адрес-сервера`. |
| **Docker и Docker Compose** | Программа, которая запускает магазин «из коробки» одним стеком контейнеров. | Устанавливается скриптом подготовки сервера (см. ниже). |

### DNS: три A-записи на IP вашего VPS

В панели управления доменом создайте **три** A-записи, все указывающие на **один и тот же** IP
вашего VPS (замените `myshop.ru` на ваш домен, `203.0.113.10` — на IP сервера):

| Имя (host) | Тип | Значение | Что открывается |
|---|---|---|---|
| `myshop.ru` (или `@`) | A | `203.0.113.10` | Витрина THE CASE (публичный сайт) |
| `www` | A | `203.0.113.10` | То же (редирект на основной адрес) |
| `admin` | A | `203.0.113.10` | Admik: админка, API, картинки товаров |

> **Проверка.** После создания записей `dig +short myshop.ru`, `dig +short www.myshop.ru` и
> `dig +short admin.myshop.ru` должны вернуть IP вашего VPS. DNS может обновляться до нескольких
> минут/часов.

> **VPS** (Virtual Private Server) — это «компьютер в аренду» в дата-центре, к которому вы
> подключаетесь по сети.
> **SSH** — защищённый способ управлять этим компьютером из командной строки.
> **Docker** — система, которая упаковывает приложение со всеми зависимостями в «контейнеры»,
> чтобы оно одинаково запускалось на любом сервере.

### Подготовка сервера (одна команда)

Сначала скачайте код магазина (он содержит скрипт подготовки), затем запустите этот скрипт.

```bash
git clone URL-репозитория admik && cd admik
./scripts/server-bootstrap.sh
```

> Вместо `URL-репозитория` подставьте адрес, который вам дали. Витрина THE CASE должна лежать
> **рядом**, в подкаталоге `./THE CASE` (её собирает тот же стек). Если репозиторий уже включает
> её — отлично; если витрину присылают отдельно, поместите её в `admik/THE CASE`.

**Что должно произойти.** Скрипт `server-bootstrap.sh` идемпотентно подготовит чистый сервер:
- установит **Docker Engine + плагин Docker Compose** (если ещё не стоят);
- включит **swap-файл** (по умолчанию 4 ГБ) — страховка от нехватки памяти при сборке образов
  на VPS со скромным RAM (создаётся только если swap'а ещё нет);
- настроит базовый **firewall** (`ufw`): откроет порты `22` (SSH), `80` и `443` (веб);
- напечатает следующие шаги.

> Скрипт **не хранит секретов** и **не запускает** магазин — он только готовит окружение.
> Размер swap можно изменить: `SWAP_SIZE=2G ./scripts/server-bootstrap.sh`. Если firewall'ом
> управляет облачная панель провайдера — пропустите его: `SKIP_UFW=1 ./scripts/server-bootstrap.sh`.

**Проверьте, что Docker работает:**

```bash
docker --version && docker compose version
```

Вы должны увидеть две строки с номерами версий (например `Docker version 27.x` и
`Docker Compose version v2.x`).

**Если пошло не так:**
- `command not found: docker` после скрипта — установка не прошла. Повторите запуск скрипта,
  проверьте доступ в интернет (`ping -c1 get.docker.com`).
- `permission denied` при запуске `docker` — вы вошли не под `root`. Либо войдите под `root`,
  либо добавьте себя в группу docker: `sudo usermod -aG docker $USER`, затем выйдите и зайдите
  по SSH заново.

**Все последующие команды выполняются из папки `admik`** (где вы сейчас находитесь).

> **git** — программа для скачивания и версионирования кода. **репозиторий** — хранилище с кодом
> проекта. Если `git` не установлен: `apt-get update && apt-get install -y git`. Если репозиторий
> приватный — используйте выданный токен/ссылку с ключом; либо распакуйте присланный архив:
> `tar -xzf admik.tar.gz && cd admik`.

## Шаг 1. Настроить файл `.env`

`.env` — это текстовый файл с настройками магазина (домен, пароли, реквизиты). Скопируйте шаблон:

```bash
cp .env.example .env
```

**Что должно произойти.** Появится файл `.env` — копия шаблона со всеми настройками и
комментариями. Откройте его для редактирования:

```bash
nano .env
```

> **nano** — простой текстовый редактор. Сохранить: `Ctrl+O`, затем `Enter`. Выйти: `Ctrl+X`.

### Минимум обязательных полей (ОБЯЗАТЕЛЬНО ЗАМЕНИТЕ)

Найдите в файле эти строки и впишите свои значения. Остальные поля можно оставить как есть —
у них рабочие значения по умолчанию.

| Поле | Что вписать | Пример | Почему важно |
|---|---|---|---|
| `SHOP_DOMAIN` | Ваш домен витрины (без `https://`) | `myshop.ru` | По нему открывается **витрина**; выпускается SSL для apex и `www`. |
| `ADMIN_DOMAIN` | Поддомен админки/API/медиа | `admin.myshop.ru` | По нему открывается **Admik** (админка, API, картинки товаров); свой SSL. |
| `ACME_EMAIL` | Ваш email | `you@myshop.ru` | На него приходят уведомления об SSL-сертификатах. |
| `NEXT_PUBLIC_ADMIK_API_URL` | Публичный адрес API (= поддомен Admik) | `https://admin.myshop.ru` | Витрина «впекает» его в браузерный код (checkout, города СДЭК). |
| `S3_PUBLIC_URL` | Адрес отдачи картинок через Admik | `https://admin.myshop.ru/media/admik-media` | По нему браузер грузит фото товаров (через Caddy → MinIO). Первый сегмент после `/media` = имя бакета (`admik-media`). |
| `STOREFRONT_ALLOWED_ORIGINS` | apex и `www` витрины | `https://myshop.ru,https://www.myshop.ru` | Admik пускает витрину к API по её Origin (без ключа). |
| `STOREFRONT_API_KEY` **+** `STOREFRONT_API_KEYS` | **Один общий ключ в двух местах** (одинаковое значение): `STOREFRONT_API_KEY=<key>` и `STOREFRONT_API_KEYS=storefront:<key>` | `<key>` = `openssl rand -hex 32` | Серверные (SSR) запросы витрины идут **без Origin** → по нему авторизоваться не могут и получают 401. Общий ключ даёт SSR-каталогу витрины данные. `STOREFRONT_API_KEY` — **не** `NEXT_PUBLIC`, в браузер не попадает. |
| `POSTGRES_PASSWORD` | Надёжный пароль базы данных | `LkQ7-длинный-случайный` | Защита базы. **ЗАМЕНИТЕ** `change-me-strong-password`. |
| `DATABASE_URL` | Та же строка, но с вашим паролем БД внутри | `postgres://admik:ВАШ_ПАРОЛЬ@postgres:5432/admik` | Приложение подключается к базе по этой строке — **пароль должен совпадать** с `POSTGRES_PASSWORD`. |
| `APP_PASSWORD` | Пароль служебной роли приложения | случайный | **ЗАМЕНИТЕ** `change-me-app-password`. |
| `MIGRATOR_PASSWORD` | Пароль служебной роли миграций | случайный | **ЗАМЕНИТЕ** `change-me-migrator-password`. |
| `MINIO_ROOT_PASSWORD` | Пароль хранилища файлов (MinIO) | случайный, мин. 8 символов | **ЗАМЕНИТЕ** `change-me-minio-password`. |
| `S3_SECRET_KEY` | То же значение, что `MINIO_ROOT_PASSWORD` | то же | Должно **совпадать** с паролем MinIO. |
| `OWNER_EMAIL` | Email владельца магазина | `owner@myshop.ru` | Под ним вы войдёте в админку. |
| `OWNER_PASSWORD` | **Оставьте пустым** для автогенерации | (пусто) | Если пусто — при первом деплое скрипт сам сгенерирует надёжный пароль и **покажет его один раз**. |

> **Подстановка домена.** В `.env.example` поля `NEXT_PUBLIC_ADMIK_API_URL`,
> `STOREFRONT_ALLOWED_ORIGINS` и `S3_PUBLIC_URL` написаны через `${SHOP_DOMAIN}`/`${ADMIN_DOMAIN}`.
> docker compose подставляет переменные **в том же файле** только при ссылке вида `${VAR}` — это
> работает, но проще и надёжнее **вписать домен явно** (как в примерах выше), чтобы не путаться.

> **Совет по паролям.** Сгенерировать случайный пароль на сервере: `openssl rand -base64 24`.
> Запускайте по разу для каждого пароля и вставляйте результат.

> **Важно про совпадение паролей.** Пароль базы упоминается в **двух** местах
> (`POSTGRES_PASSWORD` и внутри `DATABASE_URL`) — впишите одинаковое значение. Пароль MinIO —
> тоже в двух (`MINIO_ROOT_PASSWORD` и `S3_SECRET_KEY`). Если не совпадут, приложение не
> подключится.

Сохраните файл (`Ctrl+O`, `Enter`, `Ctrl+X`).

**Если пошло не так:**
- Не уверены, что заполнили всё — пробегитесь по файлу и найдите слова `change-me` и
  `ЗАМЕНИТЕ`/`ОБЯЗАТЕЛЬНО`: эти места требуют вашего внимания.
- Секция `CDEK_*` (доставка СДЭК) **не обязательна** на старте — у интеграции есть mock-режим,
  магазин поднимется и без боевых ключей. Боевые/тестовые ключи СДЭК вписываются только в `.env`
  на сервере (в репозиторий не коммитятся), см. комментарии в `.env.example`.

## Шаг 2. Поднять магазин (`make deploy`)

Один VPS, два домена — но команда деплоя по-прежнему **одна**. Она соберёт оба образа
(Admik из `Dockerfile` и витрину из `./THE CASE/Dockerfile`), поднимет весь стек,
инициализирует магазин и прогонит smoke-проверку:

```bash
make deploy
```

(то же самое: `./scripts/deploy.sh`)

**Что должно произойти.** Docker соберёт и запустит сервисы:
- `app` — Admik (админка/API), `storefront` — витрина THE CASE;
- `postgres` (база), `redis` (кеш), `minio` (файлы);
- `caddy` (веб-сервер с авто-SSL для **обоих** доменов);
- `cron` (фоновые задачи СДЭК) и `backup` (бэкапы по расписанию).

После подъёма стека `deploy.sh` **автоматически создаёт бакет медиа в MinIO** (`S3_BUCKET`, по
умолчанию `admik-media`) и включает на нём публичное чтение — отдельных ручных действий не нужно.
Шаг **идемпотентен**: при повторном `make deploy` бакет не пересоздаётся, деплой не падает.
Затем выполнится инициализация магазина внутри контейнера `app` (таблицы, роли, владелец) и
smoke-проверка. Первый запуск собирает оба приложения и может занять несколько минут.

> **Где взять пароль владельца.** Если `OWNER_PASSWORD` в `.env` пуст, при инициализации скрипт
> сгенерирует надёжный пароль и **напечатает его ОДИН РАЗ**. Скопируйте и сохраните его — повторно
> он не показывается; смените при первом входе.

**Проверьте состояние сервисов:**

```bash
docker compose ps
```

**Как понять, что всё хорошо.** В колонке `STATUS` у сервисов — `Up`, а где есть проверка
здоровья — `(healthy)`. Особенно важны `app` и `storefront`: дождитесь у обоих `healthy`.

> Можно поднять стек и вручную: `docker compose up -d --build` (флаг `-d` — «в фоне»), затем
> `make init` (инициализация) и `make smoke` (проверка). `make deploy` делает это за один шаг.

**Если пошло не так:**
- Сервис в статусе `Restarting` или `Exited` — посмотрите его логи:
  `docker compose logs app` или `docker compose logs storefront` (или имя проблемного сервиса).
  Чаще всего причина — опечатка в `.env` (несовпадение паролей БД). Исправьте `.env` и повторите
  `make deploy`.
- **Порт занят** (`address already in use`, `bind: ... :80`) — на сервере уже что-то слушает
  порт 80 или 443 (другой веб-сервер, nginx/apache). Остановите его
  (`systemctl stop nginx`) или освободите порты, затем повторите `make deploy`.
- `storefront` не собирается / падает — витрина собирается из `./THE CASE`. Убедитесь, что этот
  каталог есть рядом и содержит свой `Dockerfile` (см. вступление к инструкции). Сборке нужен
  публичный адрес API: проверьте `NEXT_PUBLIC_ADMIK_API_URL` в `.env`.
- `app` или `storefront` долго `starting` — подождите ещё; если висит больше 2–3 минут, смотрите
  `docker compose logs <сервис>`.

## Шаг 3. Проверить оба домена

`make deploy` в конце сам прогоняет **smoke-проверку** ключевых адресов Admik (`/api/health`,
`/api/storefront/v1/categories`, `/products`, `/admin/login`). Если она зелёная — бэкенд готов.
Осталось убедиться, что **оба домена** открываются по HTTPS снаружи.

> Повторно запустить smoke вручную: `make smoke`. По умолчанию он стучится на `http://localhost`
> (внутрь сервера). Чтобы проверять по реальному поддомену Admik, задайте в `.env`
> `SMOKE_BASE_URL=https://admin.ВАШ_ДОМЕН`.

**Откройте в браузере (или проверьте `curl`):**

| Адрес | Что должно открыться |
|---|---|
| `https://ВАШ_ДОМЕН/` | Витрина THE CASE (главная) |
| `https://www.ВАШ_ДОМЕН/` | Та же витрина (перенаправляет на адрес без `www`) |
| `https://admin.ВАШ_ДОМЕН/admin/login` | Форма входа в админку Admik |
| `https://admin.ВАШ_ДОМЕН/api/health` | JSON `{"status":"ok"}` (API живо) |

После того как в админке появятся товары с фото — картинки на витрине грузятся с
`https://admin.ВАШ_ДОМЕН/media/admik-media/...` (медиа MinIO, отдаётся через Caddy по HTTPS).

**Если что-то не открывается:**
- Сайт не открывается по домену, а по IP/`localhost` — да: проблема с DNS/SSL. Проверьте, что
  все три A-записи (`@`, `www`, `admin`) указывают на IP сервера (`dig +short admin.ВАШ_ДОМЕН`),
  и подождите несколько минут — Caddy выпускает сертификаты автоматически при первом обращении.
- Витрина открывается, а админка (`admin.`) — нет: проверьте именно A-запись `admin` и
  переменную `ADMIN_DOMAIN` в `.env`.
- Картинки товаров не грузятся (битые превью): сверьте `S3_PUBLIC_URL` — он должен быть
  `https://admin.ВАШ_ДОМЕН/media/admik-media` (первый сегмент после `/media` = имя бакета
  `S3_BUCKET`, по умолчанию `admik-media`). Сам бакет создаётся автоматически шагом `make deploy`
  (с публичным чтением); если в его логах было предупреждение о MinIO — повторите `make deploy`
  после исправления `MINIO_ROOT_PASSWORD`/`S3_SECRET_KEY`.
- В админке/`/api/health?deep=1` ошибка S3, медиа не загружается: бакет не создан. Повторите
  `make deploy` — шаг создания бакета идемпотентен; смотрите `docker compose logs minio`.
- Smoke `FAIL` на `/api/storefront/...`, но `/api/health` `OK` — инициализация не прошла;
  повторите `make deploy` (или `make init`).

## Шаг 4. Первый вход в админку

Откройте в браузере поддомен Admik:

```text
https://admin.ВАШ_ДОМЕН/admin/login
```

**Что должно произойти.** Появится форма входа. Введите:
- **Email** — значение `OWNER_EMAIL` из `.env`;
- **Пароль** — пароль владельца (заданный в `.env` или сгенерированный при деплое, см. Шаг 2).

После входа сразу выполните базовую настройку:

1. **Смените пароль владельца** на свой (раздел профиля/безопасности). Особенно если пароль
   был сгенерирован автоматически.
2. **Брендинг и контакты.** Загрузите логотип, укажите название, контакты, реквизиты. Эти
   настройки хранятся в базе (таблица настроек, см. `docs/02` и Этап 5) и **переопределяют**
   значения по умолчанию из `.env` — менять код не нужно.
3. **Включите нужные модули.** В настройках включите/выключите функциональные блоки (каталог,
   заказы, СДЭК-доставка, CMS-страницы). Перечень активных модулей задаётся переменной
   `ADMIK_MODULES` в `.env` (по умолчанию `catalog,orders,cdek,cms`) и/или настройками в админке.
4. **Заведите пару товаров с фото** — они появятся на витрине (`https://ВАШ_ДОМЕН/`) через
   Storefront API (наполнение каталога подробно — `docs/14`).

**Если пошло не так:**
- Страница не открывается — проверьте A-запись `admin` и что smoke (Шаг 3) зелёный.
- Неверный логин/пароль — сверьте email с `OWNER_EMAIL`; по паролю см. FAQ ниже.

## Эксплуатация: логи, бэкапы, обновления

После запуска магазин обслуживает себя сам (бэкапы по расписанию, авто-перезапуск упавших
сервисов). Ниже — команды, которые могут понадобиться вручную.

### Посмотреть логи

```bash
make logs
```

(то же: `docker compose logs -f`) — показывает логи всего стека в реальном времени.
Выход — `Ctrl+C` (сам стек при этом **не** останавливается). Логи отдельного сервиса:
`docker compose logs -f app`.

### Сделать бэкап вручную

Бэкапы и так делаются автоматически по расписанию (сервис `backup`, расписание `BACKUP_CRON` в
`.env`, по умолчанию каждую ночь). Но можно сделать копию прямо сейчас:

```bash
./scripts/backup.sh
```

**Что произойдёт.** Скрипт создаст логический дамп базы (`pg_dump`) и зеркало медиа-файлов в
томе `backups`, удалит копии старше `BACKUP_RETENTION_DAYS` (по умолчанию 14 дней). Восстановление
из бэкапа — `./scripts/restore.sh` (требует подтверждения `CONFIRM=yes`, так как **перезаписывает**
данные).

> Раз в неделю автоматически прогоняется «учебное восстановление» (`restore-drill.sh`) — оно
> восстанавливает последний бэкап во временную базу и проверяет данные. Это гарантия, что бэкапы
> рабочие. «Бэкап без проверенного восстановления — не бэкап» (ADR-015).

### Обновить магазин (новая версия кода)

```bash
./scripts/update.sh
```

**Что произойдёт.** Скрипт безопасно обновит магазин (ADR-015, обновление «без простоев»):
скачает новый код → соберёт новый образ → сделает **обязательный бэкап** → проверит, что
миграции безопасны → накатит их → пересоздаст приложение и дождётся, что оно здорово
(**health-gate**). Если новая версия окажется нездоровой — произойдёт **автоматический откат** на
предыдущую. Данные не теряются.

> **Примечание.** Те же операции доступны короткими целями `Makefile`: `make backup` и
> `make update` (вызывают `./scripts/backup.sh` и `./scripts/update.sh`).

### Алерты (необязательно)

Если задать `ALERT_WEBHOOK_URL` в `.env` (Telegram/Slack-вебхук), магазин будет присылать
оповещения при проблемах (сервис недоступен, бэкап/восстановление не удались, мало места на
диске). Если поле пустое — оповещения просто пишутся в лог.

### FAQ — типичные ошибки

**Порт занят** (`address already in use` на :80/:443). На сервере уже работает другой
веб-сервер. Остановите его (`systemctl stop nginx` / `systemctl stop apache2`) или освободите
порты, затем `docker compose up -d`.

**Домен не резолвится** (магазин не открывается по адресу, а по IP/localhost — да). A-запись
домена не указывает на IP сервера или ещё не обновилась. Проверьте: `dig +short ВАШ_ДОМЕН` должен
вернуть IP вашего VPS. После исправления подождите несколько минут (DNS + выпуск SSL Caddy).

**Забыл сменить / потерял пароль владельца.** Войдите на сервер и проверьте логи инициализации:
`docker compose logs app | grep -i пароль` (сгенерированный пароль печатался один раз). Если не
нашли — задайте `OWNER_PASSWORD` в `.env` вручную и **обратитесь к разработчику** для безопасного
сброса (учётка владельца идемпотентна, но смена пароля существующего владельца требует аккуратной
процедуры — см. следующий раздел).

**Магазин «упал» / сервис перезапускается.** `docker compose ps` покажет проблемный сервис,
`docker compose logs ИМЯ_СЕРВИСА` — причину. Большинство проблем — опечатки в `.env`.

## Когда звать разработчика

Эта инструкция покрывает разворачивание и базовую эксплуатацию. **Зовите разработчика**, если:

- `make smoke` стабильно `FAIL` после правильного выполнения шагов 1–4, и логи непонятны;
- нужно **изменить поведение/функциональность** магазина (не просто настройку через админку
  или `.env`);
- нужно **восстановиться из бэкапа** на боевой базе (`restore.sh` перезаписывает данные —
  ошибка здесь стоит дорого);
- **обновление откатилось** (`update.sh` сообщил об откате) — значит новая версия нездорова,
  нужна диагностика;
- требуется **сбросить пароль** существующего владельца или добавить новых сотрудников с
  правами на уровне ниже владельца;
- нужно подключить **внешний мониторинг/метрики** (Prometheus/Grafana/Sentry) — это точка
  расширения, в базовую поставку не входит (ADR-015);
- нужно настроить **offsite-бэкап** в облако (`BACKUP_OFFSITE_REMOTE`) или боевые ключи
  интеграций (СДЭК, платёжная система).

> **Граница ответственности.** Не-разработчик отвечает за: разворачивание по этой инструкции,
> заполнение `.env`, настройку магазина через админку, наблюдение за логами и за тем, что бэкапы
> идут. Всё, что касается **изменения кода, восстановления данных и диагностики нетривиальных
> сбоев** — зона разработчика.
