# Аудит: что НЕЛЬЗЯ отредактировать из админки

**Дата:** 2026-06-23 · **Заказчик:** Саша · **Объём:** платформа Admik (backend + админка) + витрина THE CASE (потребитель Storefront API)

> Источник: read-only аудит (24 агента, адверсариальная проверка каждого пробела по коду). Запрос владельца: «выяснить, что именно нельзя отредактировать из админки» — блок «О бренде» был лишь одним примером из многих. Контекст прерванной сессии — память `crashed-session-2026-06-23-admin-gaps`.

---

## 1. TL;DR

**Подтверждено 18 пробелов** (G-01…G-16, G-18, G-19). Один заявленный пробел **опровергнут** (G-17 — на самом деле МОЖНО).

**Корневая причина №1 (G-01):** витрина THE CASE **вообще не запрашивает** `GET /api/storefront/v1/settings`. В её API-клиенте (`THE CASE/src/lib/admik/client.ts`) есть `listProducts/getProduct/getCategories/quoteCart/createOrder/initPayment/getOrder/cdek*` — но **нет `getSettings`** (`grep -i settings` по `THE CASE/src` = 0). Поэтому даже то, что админка УЖЕ умеет редактировать и API УЖЕ отдаёт (брендинг, контакты, реквизиты, SEO), физически не доходит до сайта.

| Область | Пробелов | Тип проблемы |
|---|---|---|
| Архитектура (корень) | 1 (G-01) | end-to-end разрыв: витрина не читает /settings |
| Главная (home-контент, hero, philosophy, изображения, вкладки) | 6 (G-02…G-05, G-18, G-19) | хардкод в коде витрины |

| Личный кабинет (статусы) | 1 (G-15) | хардкод словаря |
| SEO (метаданные) | 1 (G-16) | админка готова, витрина игнорирует |

**WIP ADR-018 (незакоммичено):** полностью не закрывает ни одного; **5 частично** (G-01, G-02, G-03, G-05, G-06). Добавил схему `home`, `updateHomeAction`, DTO-поле `home`, дефолты `lib/config/home-defaults.ts`, server-side загрузку изображений — но **без UI-формы** и **без чтения на витрине**. Для владельца пробелы актуальны на 100%.

**Нюанс приоритезации:** ~5 пробелов (G-01, G-06, G-07, G-08, G-16) — это НЕ дыры в админке, а разрыв интеграции на стороне витрины. Бэкенд и формы готовы — нужно научить витрину читать API. Это дешевле, чем строить новые схемы/формы.

## 7. Runtime-находка R-01 (2026-06-24) — НЕ из списка G-xx

> Отдельный **класс** дефекта: не «контент нельзя отредактировать», а «отредактированное не доходит до витрины из-за
> рантайм-кэша». Поэтому оформлено как **R-01**, вне нумерации G-01…G-19.

**Симптом (сообщил Саша).** Правка контента в админке не отражалась на витрине. Пример: блок «О бренде» на главной
(`settings.home`) — админка сохранила в БД `about.title = "О брендеqqq"` (+ правленый параграф), но витрина
https://erfgq.website и публичный API `GET /api/storefront/v1/settings` отдавали старое «О бренде».

**Диагностика.** Трёхслойное сравнение на стенде: БД `shop_settings.home` = новое ✅; публичный API админки = старое ❌;
витрина (читает API) = старое. app — один процесс `node server.js` (uptime 12ч, без реплик); settings-action исправно
вызывает `invalidateSettingsCache()`.

**Корень.** Memo-кэш `getEffectiveSettings` (`cached`/`inflight`/`cacheEpoch`) хранился в **модульных** `let` в
`lib/config/settings.ts`. Next.js инстанцирует один и тот же модуль в **разных** серверных бандлах (Server Actions vs
Route Handlers/RSC) → `invalidateSettingsCache()` из экшена сбрасывал memo своего экземпляра, а публичный route-handler
`/settings` читал memo **другого** экземпляра и бесконечно отдавал устаревший снимок (до рестарта app).

**Фикс.** Состояние кэша вынесено на `globalThis` под registry-Symbol `Symbol.for('admik.config.settings.effectiveCache')`
→ один кэш и одна инвалидация на весь процесс, общие для всех бандлов. Добавлен guard «снимаем только свой in-flight».
Файлы: `lib/config/settings.ts` (правка), `tests/config/settings.test.ts` (новый регресс-тест «кэш живёт на globalThis»).

**Оговорка.** Прямая правка `shop_settings` через SQL по-прежнему требует `docker compose restart app` (она не вызывает
invalidate). Фикс покрывает только правки через UI/экшены приложения.

**Проверки (две).** (1) код-гейт — `tsc --noEmit` чисто, полный `vitest run` = 1660 passed / 105 skipped; (2) живой
