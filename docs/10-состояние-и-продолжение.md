# 10 — Состояние кита и гайд продолжения

> **Назначение файла.** Это компактный «гайд продолжения» для новой сессии Claude Code
> по киту **Uni_BD**. Прочитав только его, можно понять: что это за кит и где он, как
> проверить его здоровье, как устроен репозиторий и что делать дальше — без анализа
> скрытых сессий. Если репозиторий расходится с этим файлом — источником правды считать
> `git`, `START-HERE.md`, `docs/00-журнал-проекта.md` и `docs/03-роадмап.md`, а этот файл обновить.

---

## 1. Что это за кит и где он

**Uni_BD** — готовая универсальная платформа **Admik**: headless-бэкенд +
административная панель + база данных + Storefront API + интеграции (СДЭК, Т-Банк) +
CMS + DevOps + мультиагентный инструментарий Claude Code. Весь код админки и БД уже
написан и покрыт тестами; **тематики конкретного магазина здесь НЕТ** — данные
нейтральные (заглушки/демо).

- **Расположение:** `/home/claudeuser/Uni_BD` (git-репозиторий, origin `TarrySeeker/Uni_BD`).
- **Назначение:** кит копируется в проект, где **уже есть готовый макет витрины**
  (Next.js, без БД и без админки). Задача новой сессии — **прикрутить к макету эту
  админку и БД** и ретематизировать данные под нишу магазина, **не трогая бизнес-логику
  платформы**. Витрина становится чистым потребителем `/api/storefront/v1/*`.
- **Модель:** «1 магазин = 1 БД + конфигурация на отдельном VPS». Кодовая база едина;
  между магазинами меняются только `.env` + таблица `shop_settings` и содержимое БД.

Стек: Next.js 16 (App Router, `standalone`) · React 19 · TypeScript strict ·
PostgreSQL 15 (postgres.js) · Redis (ioredis) · S3/MinIO (sharp) · Zod · Vitest +
Playwright · pnpm · Docker + Caddy (auto-TLS).

## 2. Как проверить здоровье кита

**Код-проверки (первая из двух обязательных).** На свежем клоне сначала `pnpm install`
(`node_modules` не в git):
```bash
pnpm install
pnpm typecheck      # tsc --noEmit → 0 ошибок
pnpm lint           # eslint → 0
pnpm test           # vitest run (~1100 юнит-тестов; часть с БД — на стенде/CI, помечены skip)
```
Скилл `/gate` — полный код-гейт (typecheck + lint + vitest + `check-migrations.sh` +
security). Принцип: **сначала тест (падает) → потом код до зелёного**. Часть тестов
всегда `skipped` без `DATABASE_URL`/Redis/браузера — это норма; `failed` быть не должно.

**БД и стек через Docker (если доступен):**
```bash
cp .env.example .env   # заполнить под магазин (см. START-HERE.md §6)
make up                # docker compose up -d (app + postgres + redis + minio + caddy)
make init              # init-shop.sh: расширения+роли → 33 миграции → seed прав/ролей/владельца
make smoke             # HTTP-проверки эндпоинтов
```
`make deploy` = `up → init → smoke` одним сценарием. Health-эндпоинт: `GET /api/health`
(и `?deep=1` — с проверкой БД/Redis/S3, ожидается `{"status":"ok"}`). Админка — `/admin`.

**Миграции:** `db/migrations/0001…0033` (33 идемпотентные SQL-миграции). Существующие
**иммутабельны**, новые — только аддитивные (это проверяет `check-migrations.sh`).

## 3. Как устроен репозиторий

```
START-HERE.md          ← новая сессия читает ПЕРВЫМ
CLAUDE.md              ← правила платформы, роль оркестратора, инварианты
README.md              ← краткий обзор и quickstart
.env.example           ← ВСЕ переменные окружения (шаблон .env; секреты — только в .env)
.mcp.json / .claude/setup.sh  ← MCP-серверы (playwright, context7, firecrawl, glyph) + их включение

app/
  admin/               ← административная панель (каталог/заказы/промо/СДЭК/роли/аудит/настройки/CMS)
  api/storefront/v1/   ← ПУБЛИЧНЫЙ Storefront API (контракт — docs/21)
  api/{health,cdek,payments,cron}/  ← health, webhook'и, cron
  mock/                ← demo-страница mock-оплаты Т-Банк
lib/                   ← бизнес-логика по доменам (catalog/orders/cdek/cms/auth/audit/analytics/…)
  config/              ← env (Zod), флаги модулей, эффективные настройки (env ⊕ БД)
  storefront/          ← конвейер Storefront API: auth, CORS, DTO, rate-limit
  server/action.ts     ← defineAction: guard(права) → Zod → БД → revalidate → audit (все мутации)
db/
  migrations/*.sql     ← 33 миграции (НЕ редактировать существующие; только аддитивные)
  seed/                ← roles.sql, permissions.sql, нейтральный demo-catalog.sql
scripts/               ← init-shop, deploy, backup/restore/restore-drill, update, smoke, check-migrations
docs/                  ← документация (00–21)
.claude/
  agents/*.md          ← 9 субагентов (роли + allowlist инструментов)
  skills/*/SKILL.md    ← onboard-shop, gate, deploy-stand, new-shop, integration-check,
                          module-toggle, audit-cycle
tests/                 ← Vitest-тесты по доменам (TDD)
Dockerfile · docker-compose.yml · Caddyfile · Makefile  ← DevOps из коробки
```

**Модули** (вкл/выкл через `ADMIK_MODULES` или `shop_settings.module_overrides`):
`catalog`, `orders`, `cdek`, `cms`, `payments`. Ядро (`core`: auth/RBAC/настройки/аудит/
заявки/подписчики) — всегда включено. **Интеграции** без боевых ключей работают в
**mock-режиме** (СДЭК — фикстуры, Т-Банк — demo-страница оплаты) → весь путь покупателя
проходится локально. Боевые ключи — только в `.env` на сервере, не в git.

## 4. Что делать новой сессии (онбординг магазина)

1. **Настрой инструменты:** `bash .claude/setup.sh` (один раз после клонирования —
   включает MCP-серверы, проверяет Node ≥ 20 и ключи).
2. **Прочитай по порядку:** `START-HERE.md` → `CLAUDE.md` →
   `docs/20-онбординг-витрины-playbook.md` (ранбук, фазы 0–6) →
   `docs/21-контракт-storefront-api.md` (контракт для сращивания).
3. **Изучи фронтенд:** положи макет витрины в каталог `./storefront` и проанализируй его
   (каталог/категории/варианты/атрибуты, навигация→CMS, брендинг/контакты, способ корзины
   и оформления/оплаты) — это Фаза 1 плейбука.
4. **Детальное интервью с владельцем** (Фаза 0 плейбука `docs/20`): ниша, домены, нужные
   модули, валюта, реквизиты, mock/боевые интеграции. Зафиксируй профиль магазина в `docs/00`.
5. **Запусти скилл `/onboard-shop`** — он оркеструет остальное субагентами:
   **стадийно поднять БД** (`make up` → `make init` → `make smoke`, вход в `/admin`),
   ретематизировать каталог/атрибуты/брендинг **конфигурацией и данными** (без правок
   бизнес-логики), срастить макет со Storefront API (маппинг `docs/21 §7`), пройти
   **две проверки** и всё задокументировать.

**Definition of Done:** БД+миграции+seed готовы; каталог/брендинг ретематизированы через
`.env`+`shop_settings`+seed (ноль хардкода ниши в коде); витрина подключена к
`/api/storefront/v1/*`; админка работает и заведённое доезжает до витрины; `/gate` зелёный
+ живой прогон в браузере (mock-интеграции) пройден; журнал `docs/00` и этот файл обновлены.

## 5. Незыблемые инварианты

1. **TDD** — сначала тест (падает) → код до зелёного.
2. **Две проверки** перед коммитом: код-гейт `/gate` **и** живой прогон в браузере.
3. **Мультитенантность** — ноль хардкода ниши: название, логотип, цвета, контакты,
   реквизиты — только в `.env` и `shop_settings`; бизнес-логику платформы не меняем.
4. **Миграции** существующие иммутабельны, новые — аддитивны (`check-migrations.sh`).
5. **Секреты только из env** — боевые ключи СДЭК/Т-Банк/БД на сервере, не в git.
   Реальных ключей в репозитории нет и быть не должно (только плейсхолдеры в `.env.example`).
6. **Anti-tamper:** цены/итоги считает сервер, деньги — в копейках (не float).
7. **Документируй каждую правку** (`documentation-manager` / `history-manager`).
