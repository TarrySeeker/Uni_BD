## Принципы, обязательные для каждой задачи

Эти правила вытекают из `CLAUDE.md`, ADR-004 и `docs/02-модель-развёртывания.md` и
**применяются ко всем задачам-разработки во всех этапах**. В декомпозиции ниже они не
повторяются дословно у каждой задачи — но обязательны:

1. **Сначала тесты, потом код (ADR-004).** Для каждой задачи-разработки сперва пишутся
   тесты (Vitest — логика/валидация, Playwright — e2e/a11y админки), которые **падают**
   на отсутствующей реализации, затем пишется код до зелёных тестов. QA проверяет
   покрытие до приёмки. В декомпозиции это отражено отдельным первым пунктом «Тесты».
2. **Идемпотентные миграции.** Любая новая таблица/изменение схемы — миграцией с
   `CREATE ... IF NOT EXISTS` (и безопасными `ALTER`), повторный накат не ломает данные.
   Накат на пустую БД — одной командой.
3. **Флаги модулей.** Любой функционал, относящийся к модулю, уважает флаг включения из
   `lib/config/modules.ts` (переменная окружения `ADMIK_MODULES`, значения:
   `catalog`, `orders`, `cdek`, `cms`). Выключенный модуль не отдаёт UI/API/cron.
4. **Секреты — только в env.** Любой ключ/пароль/токен читается из переменной окружения
   через `lib/config/env.ts` (Zod-валидация). В репозитории — только `.env.example`.
5. **Mock-режим интеграций.** Любая зависимость от внешнего сервиса (СДЭК, платежи, S3,
   email) имеет mock/demo-режим, чтобы магазин без боевых ключей поднимался и проходил
   smoke-тест.
6. **Никаких хардкодов магазина.** Название, логотип, контакты, юрлицо, цвета, реквизиты,
   ключи — только в `.env`/таблице настроек, не в коде. Код единый и неизменный при
   подключении нового магазина (ADR-003, `docs/02`).
7. **Обратная совместимость.** Обновление кода существующего магазина не ломает его данные.
