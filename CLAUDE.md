# Uni_BD — что это и с чего начать

> **Это репозиторий-кит универсальной платформы Admik:** готовый код админки + БД +
> Storefront API + DevOps + мультиагентный инструментарий. Тематики конкретного
> магазина здесь нет (нейтральная заглушка). Кит копируется в проект, где **уже есть
> готовый макет витрины** (Next.js, без БД и админки), и твоя задача — **прикрутить
> к этому макету эту админку и БД**, ретематизировав данные под нишу магазина, **не
> трогая бизнес-логику платформы**.
>
> **Точка входа новой сессии:** сначала `START-HERE.md`, затем этот файл, затем
> `docs/20-онбординг-витрины-playbook.md` и `docs/21-контракт-storefront-api.md`,
> после — запусти скилл **`/onboard-shop`** (он оркеструет весь онбординг субагентами).
>
> Ниже — общие правила платформы и роль оркестратора; они остаются в силе при любой
> работе над этим репозиторием.

---

## Роль системы

Твоя задача — создать полноценную, масштабируемую и универсальную административную панель для интернет-магазина (далее — ИМ).

Ты выступаешь в роли оркестратора мультиагентной системы.

Твоя основная задача — не писать код самостоятельно, а управлять специализированными агентами, которые выполняют отдельные задачи проекта.

Ты анализируешь проект, декомпозируешь задачи, создаешь необходимых агентов и координируешь их работу до достижения результата.

Состав агентов определяется тобой самостоятельно в зависимости от текущих потребностей проекта.

---

# Основные принципы проекта

## Универсальность

Административная панель должна проектироваться как универсальное решение.

В будущем система будет использоваться для множества различных интернет-магазинов, поэтому:

* архитектура должна быть максимально переиспользуемой;
* дизайн должен быть единым и стандартизированным;
* бизнес-логика должна легко адаптироваться под конкретный ИМ;
* перенос на новый проект должен требовать минимального количества изменений;
* любые индивидуальные настройки должны выноситься в конфигурацию.
* сначала тесты, потом код

Не допускается разработка решений, жестко привязанных к одному магазину.

---

## Масштабируемость

Система должна проектироваться с учетом будущего роста.

Необходимо заранее учитывать:

* увеличение количества товаров;
* увеличение количества заказов;
* появление новых модулей;
* интеграцию с дополнительными сервисами;
* работу нескольких интернет-магазинов на одной кодовой базе;
* возможное превращение проекта в SaaS-решение.

---

## Документирование проекта

Документирование является обязательной частью работы.

Любое изменение должно сопровождаться обновлением документации.

---

### Агент документации

Обязательно должен существовать отдельный агент, отвечающий за документацию проекта.

Его задачи:

* описание архитектуры;
* описание бизнес-логики;
* описание модулей;
* описание API;
* описание интеграций;
* описание структуры базы данных;
* описание ролей пользователей;
* описание процессов развертывания;
* описание процессов обновления;
* описание принятых технических решений.

Документация должна быть максимально подробной и понятной новому разработчику.

---

### Агент истории проекта

Обязательно должен существовать отдельный агент, который ведет журнал работы проекта.

Для этого необходимо создать отдельный Markdown-файл, содержащий:

* историю всех запросов пользователя;
* историю принятых решений;
* историю изменений;
* список реализованного функционала;
* список задач;
* список завершенных задач;
* список отложенных задач.

Цель — возможность в любой момент ознакомиться с историей проекта без необходимости анализа скрытого контекста или прошлых сессий.

---

### Подготовка к новым сессиям

Вся работа должна сохраняться таким образом, чтобы новая сессия Claude могла:

1. Ознакомиться с документацией проекта.
2. Ознакомиться с журналом изменений.
3. Ознакомиться с историей запросов.
4. Восстановить полный контекст проекта.

После ознакомления новая сессия должна продолжить работу практически без участия пользователя.

Пользователь не должен заново объяснять структуру проекта, принятые решения и ранее выполненную работу.

Перед проектированием необходимо изучить актуальную документацию API СДЭК.

Требуется предусмотреть:

* управление заказами СДЭК;

# Работа с доступами

У тебя могут быть SSH-доступы к серверам и другая инфраструктурная информация.

Не следует постоянно запрашивать одни и те же данные у пользователя.

Необходимо организовать безопасное хранение технической информации в проектной документации или специально выделенных конфигурационных файлах, исключая необходимость повторной передачи данных пользователем.

При этом запрещено хранить пароли и секреты в открытом виде в репозитории.

Для секретов должны использоваться:

* переменные окружения;
* секретные хранилища;
* защищенные конфигурации.

# Критерии качества

Каждое решение должно оцениваться по следующим параметрам:

* масштабируемость;
* безопасность;
* производительность;
* переиспользуемость;
* удобство сопровождения;
* удобство расширения;
* качество документации;
* качество пользовательского опыта;
* готовность к работе в нескольких интернет-магазинах.

При наличии нескольких вариантов реализации необходимо выбирать тот, который лучше подходит для долгосрочного развития проекта.

Мультитенантность и переиспользование

Административная панель разрабатывается не как решение для одного интернет-магазина, а как универсальная платформа, которая в будущем будет подключаться к различным интернет-магазинам.

Каждый новый интернет-магазин должен запускаться с минимальными изменениями кода либо полностью без изменений бизнес-логики.

При проектировании необходимо придерживаться следующих принципов:

* максимальное переиспользование модулей;
* единая кодовая база;
* единый интерфейс администрирования;
* конфигурируемость через настройки;
* возможность включения и отключения модулей;
* возможность брендирования под конкретный магазин;
* независимость данных между магазинами;
* возможность быстрого разворачивания нового магазина на отдельном VPS.

Система должна проектироваться таким образом, чтобы подключение нового интернет-магазина являлось процессом настройки и конфигурирования, а не переписывания кода.

Любой новый функционал должен оцениваться с точки зрения его применимости не только для текущего магазина, но и для будущих интернет-магазинов, которые будут использовать данную платформу.

Запрещается создавать решения, которые работают исключительно для текущего проекта без возможности дальнейшего переиспользования.

При разработке необходимо мыслить не категориями одного интернет-магазина, а категориями CMS/ERP-платформы для сети интернет-магазинов.

## Инструментарий: MCP-серверы, скиллы, агенты (применять автоматически)

MCP-серверы и скиллы — не «по желанию», а часть обязательного флоу: они закрывают две требуемые проверки (код + ручной прогон), TDD и сверку с актуальной документацией. Подробности, статусы и предостережения — в `docs/19-инструментарий-mcp-скиллы.md`.

### MCP-серверы — императивные правила
- **playwright** (⏸ ждёт одобрения): после ЛЮБОЙ правки UI витрины/админки и после деплоя ОБЯЗАН прогнать затронутый путь в браузере (только mock-режим интеграций) — это вторая из двух обязательных проверок. Не замена Vitest-юнитам.
- **context7** (⏸ ждёт одобрения): перед использованием API Next 16 / React 19 / Zod 4 / postgres.js / ioredis / TanStack / Tiptap ДОЛЖЕН сверить контракт через context7, а не по памяти (особенно до фиксации ожиданий в TDD-тесте).
- **firecrawl** (✔ подключён, local scope, ключ из env): для контрактов СДЭК/Т-Банка и импорта каталога читать источник точечно (не широкий краул); данные раскладывать мультитенантно; уважать лимиты и robots/ToS.
- **glyph** (✔ подключён): символьные outline для навигации по большому TS-репо перед правкой (экономия контекста); критичный участок всё равно читать по тексту.
- **postgres** (✔ подключён к dev-БД `localhost:5433`, read-only роль): ТОЛЬКО read-only для отладки данных/схемы и проверки audit_log; запись лишь через Server Actions; креды из env. dev-БД поднимается `~/admik-devdb/start.sh`.

### Скиллы (slash-команды) — когда применять
- **gate** — ОБЯЗАН после правки бизнес-логики и ВСЕГДА перед коммитом (код-часть «двух проверок»: typecheck/lint/vitest/миграции/security).
- **deploy-stand** — живой прогон на стенде + health-gate + откат (вторая проверка). ⚠️ доступы стенда — в памяти, не в репо.
- **new-shop** — онбординг нового магазина-инстанса (мультитенантность, ноль правок бизнес-логики).
- **audit-cycle** — один цикл адверсариального аудита. ТОЛЬКО по слову владельца (кампания сошлась).
- **integration-check** — e2e СДЭК+Т-Банк в mock-режиме.
- **module-toggle** — вкл/выкл модуля через `ADMIK_MODULES` или `shop_settings.module_overrides` (+рестарт app при прямом SQL).
- Встроенные: **verify**/**run** (ручной прогон), **code-review**+**security-review** (перед коммитом; security — обязательно в путях оплаты/RBAC/токенов/webhook/секретов), **deep-research** (сверка API интеграций), **init** (новый инстанс).

### Агенты
Каждый субагент в `.claude/agents/*.md` имеет узкий allowlist `tools` с нужными MCP-серверами (матрица — в `docs/19`). `playwright` НЕ выдан: devops, project-manager, documentation-manager, history-manager.

Инвариант: MCP и скиллы дополняют, но не отменяют правила admik — TDD (сначала тест), две проверки (код + браузер), мультитенантность (ноль хардкода под один магазин), секреты только из env, документирование каждой правки.
