## ADR-001. Технологический стек

**Решение:** строим на стеке проекта 2x2.

| Слой | Технология |
|------|-----------|
| Рантайм | Node 20+ |
| Фреймворк | Next.js 16 (App Router, standalone) |
| UI | React 19 + Tailwind CSS 4 |
| Язык | TypeScript (strict) |
| БД | PostgreSQL 15+ |
| Драйвер БД | postgres.js (tagged templates → защита от SQL-инъекций) |
| Авторизация | Lucia-подход (сессии в БД) + RBAC |
| Валидация | Zod |
| Таблицы админки | TanStack Table |
| Rich-text | Tiptap |
| Хранилище медиа | S3-совместимое (MinIO локально, Timeweb/AWS в проде) |
| Кеш/rate-limit | Redis (для горизонтального масштабирования) |
| Контейнеризация | Docker + Docker Compose |
| Reverse-proxy | Caddy (авто-SSL) |
| Тесты | Vitest (unit) + Playwright (e2e/a11y) |
| Пакетный менеджер | pnpm |

**Обоснование:** единый язык (TS) на фронте и бэке, проверенные на 2x2 паттерны, быстрый старт, современный и поддерживаемый стек.

**Последствия:** переиспользуем код и паттерны 2x2; логику СДЭК из carre (PHP/Yii2) портируем в TS.

## ADR-002. Источники паттернов

- **Админка:** проект `2x2` (`/root/claude-project/2x2`). Берём: унифицированный паттерн Server Actions (гвард → Zod → БД → инвалидация кеша → аудит), ролевую модель, CMS на `page_sections` (JSONB), `audit_log`, меры безопасности (anti-SQLi, magic-bytes для загрузок, rate-limit, timing-attack mitigation, идемпотентность форм), идемпотентные миграции, две роли БД (app/migrator).
- **СДЭК:** проект `carre` (`/root/claude-project/carre`). Берём слоистую архитектуру `Manager → Client → Services` (Calculator, OrderService, PvzService, WebhookService, PrintService, StatusMap), кеш OAuth-токена, webhook с идемпотентностью (UNIQUE-ключ в `cdek_status_log` + IP-whitelist), cron-воркеры (create-pending, notify-stuck), email-уведомления по статусам.

**Чего избегаем (уроки референсов):**
- Не оставлять deprecated-таблицы в схеме (в 2x2 товары/заказы помечены deprecated).
- Не держать три разные таблицы заявок — единая модель.
- Не хранить секреты в репозитории (в carre ключи PayKeeper/YooKassa лежали открыто).
- Не использовать MyISAM / VARCHAR для дат и денег — InnoDB-аналог не наш случай (у нас Postgres), но принцип: правильные типы (TIMESTAMPTZ, NUMERIC) с самого начала.
- In-process кеш/rate-limit не масштабируется — используем Redis.

## ADR-003. Модель мультитенантности

**Решение:** один интернет-магазин = одна БД. Код single-tenant в рантайме. Никаких `tenant_id`/RLS.

Универсальность достигается **конфигурацией**, а не общими данными. Подробности и процесс переноса — `docs/02-модель-развёртывания.md`.

**Обоснование:** соответствует ТЗ («каждый ИМ на отдельном VPS») и явному требованию пользователя; физическая изоляция данных бесплатна и надёжна; перенос новым магазином силами сторонних людей проще, когда нет общей БД.

**Последствия:**
- Всё, что специфично для магазина, — в конфиге и таблице настроек, не в коде.
- Модули включаются/выключаются конфигом.
- Брендинг (логотип, тема, тексты) — через конфиг/БД.
- Миграции идемпотентны и накатываются на любую пустую БД одной командой.

## ADR-004. «Сначала тесты, потом код»

**Решение:** для каждого модуля сперва пишутся тесты (Vitest для логики/валидации, Playwright для e2e админки), затем реализация.

**Последствия:** каждая задача агента-разработчика включает тесты; QA-агент проверяет покрытие до приёмки.

## ADR-005. Модель прав доступа (RBAC): гибкие права, а не enum-роли

**Контекст:** Admik — универсальная платформа для сети ИМ (CLAUDE.md, ADR-003). Разные магазины
требуют разных наборов ролей и гранулярности доступа, при этом код должен оставаться неизменным при
подключении нового магазина («копи-paste», `docs/02`).

**Решение:** использовать **гибкую модель «роли → права»**:
`users —< user_roles >— roles —< role_permissions >— permissions`.
Проверки доступа в коде идут **по правам** (`requirePermission('orders.write')`), а не по именам
ролей. Права именуются `<домен>.<действие>` (`catalog.read`, `orders.write`, `users.manage`).
Роли — настраиваемые **данными** (seed) наборы прав. Особый пользователь-владелец помечается флагом
`users.is_owner = true` и проходит все проверки.

**Рассмотренная альтернатива — enum-роли** (роль зашита в коде): отвергнута. Каждый новый магазин/модуль
потребовал бы правки enum и кода проверок, что нарушает универсальность (ADR-003) и принцип
копи-paste-развёртывания.

**Последствия:**
- Новый модуль добавляет права строками в `permissions` (seed), без правки логики проверок.
- Роли настраиваются под магазин данными; отличия магазинов — в данных, не в коде.
- Чуть выше сложность схемы (4 таблицы + seed) — приемлемая цена за универсальность.
- Детали и DDL — `docs/04-этап1-ядро-auth-rbac.md`, §2.3 и §5.

## ADR-006. Подход к аутентификации, хешированию паролей и ролям БД

**Решение:**
- **Сессии в БД (Lucia-подход), без JWT.** Сессия — строка в таблице `sessions` (id, user_id,
  expires_at); id хранится в `httpOnly`/`Secure`/`SameSite=Lax` cookie. Отзыв/инвалидация сессий
  тривиальны (важно для админки: блокировка пользователя, ротация при смене пароля).
- **Хеширование паролей — argon2id** (`@node-rs/argon2`), PHC-строки. Рекомендация OWASP, память-hard,
  устойчивее к GPU-перебору, чем bcrypt. bcrypt — допустимый fallback, но по умолчанию argon2id.
- **Защита от timing-атак:** при неизвестном email выполняется фиктивная проверка argon2; единое
  сообщение об ошибке логина; константное сравнение токенов.
- **Rate-limit логина — в Redis** (`ioredis`), т.к. in-process не масштабируется (ADR-002);
  при отсутствии `REDIS_URL` — безопасный mock-fallback (для demo-магазинов без Redis).
- **Две роли БД** `admik_app` / `admik_migrator`: миграции — под `migrator` (или владельцем БД),
  рантайм приложения — **только под `app`** с минимальными правами. `app` лишён `UPDATE/DELETE` на
  `audit_log` → журнал append-only на уровне прав БД. Пароли ролей — из `.env`.
- **Lucia как примитивы, а не пакет:** используем низкоуровневый `oslo` (session id, константное
  сравнение) и реализуем тонкий слой сессий сами, не завязываясь на мейнтенанс-режим пакета `lucia`.

**Последствия:**
- Отзыв доступа мгновенный (удаление сессии), без ожидания истечения JWT.
- Аудит защищён от подделки/затирания на уровне СУБД.
- Новые env-переменные: `MIGRATOR_PASSWORD`, `APP_PASSWORD`, `OWNER_EMAIL`, `OWNER_PASSWORD`,
  `DATABASE_URL` (под ролью `app`, становится обязательной на Этапе 1).
- Детали, DDL и сигнатуры — `docs/04-этап1-ядро-auth-rbac.md`, §3–§4 и §9.

## ADR-007. Модель каталога: характеристики (гибридная EAV) и связь товар↔категория (many-to-many)

**Контекст:** Этап 2 (`catalog`) вводит товары, категории, варианты, характеристики
(цвет/размер/материал и т.п.) и медиа. Admik — универсальная платформа для сети ИМ
(CLAUDE.md, ADR-003): набор характеристик и структура категорий у каждого магазина **свои**,
а код обязан оставаться неизменным при подключении нового магазина («копи-paste», `docs/02`).
Нужно зафиксировать два архитектурно-определяющих выбора: (1) как хранить гибкие
характеристики товара; (2) как связывать товар с категориями. Эти решения влияют на схему БД,
фильтрацию/поиск, импорт и производительность, поэтому выносятся в отдельный ADR.

### Решение 1 — характеристики: справочная EAV-модель (атрибуты как данные), не «голый» JSONB

Используем **нормализованную справочную модель «атрибут → значение → привязка»**:
`attributes` (определение характеристики: код, тип, единица) — `attribute_values`
(словарь допустимых значений для select-атрибутов) — `product_attributes`
(связь товар↔атрибут↔значение). Это EAV-подобная модель, в которой **сами атрибуты —
данные (seed/UI), а не код**: новый магазин добавляет свои характеристики строками в
таблицы, без правки логики и схемы.

| Критерий | EAV-справочник (выбрано) | «голый» JSONB-столбец в `products` |
|---|---|---|
| Универсальность (разные наборы у разных магазинов) | высокая: атрибуты — данные, заводятся через UI/seed | высокая, но без контроля словаря значений |
| Целостность значений (общий словарь «Красный/Синий») | есть (FK на `attribute_values`) — нет разнобоя | нет: «красный»/«Красный»/«red» расходятся |
| Фильтрация в каталоге («все красные размера M») | индексируемые B-tree JOIN'ы по id значений | требует GIN по JSONB + аккуратных запросов; хуже для точных фасетов |
| Управляемость в админке (типы, единицы, обязательность) | метаданные в `attributes` → форма строится сама | метаданные негде хранить структурно |
| Производительность чтения карточки товара | JOIN, но небольшой объём на товар | быстрее (один row) |
| Риск «зоопарка» ключей/типов | низкий (контролируется справочником) | высокий |

**Выбор — справочная EAV.** Главные причины: контроль словаря значений (фасетная фильтрация
и единые значения между товарами — ключ к нормальному каталогу ИМ) и хранение **метаданных
характеристики** (тип, единица измерения, обязательность, порядок), из которых админка
**автоматически строит форму** — это и есть универсальность без хардкода. «Голый» JSONB
проще на запись, но теряет словарь и метаданные, что для каталога с фильтрами критично.

**Компромисс по производительности:** для редко-фильтруемых, чисто презентационных
характеристик допускается денормализованная JSONB-проекция (`products.attributes_cache`),
обновляемая при записи, — как ускорение чтения карточки, **не заменяя** нормализованный
источник истины. Это «гибрид»: нормализация — источник правды и фильтров, JSONB — кеш чтения.

### Решение 2 — товар↔категория: many-to-many (`product_categories`), не одна категория

Товар может принадлежать **нескольким категориям** (например «Платья» и «Распродажа», или
«Новинки»). Поэтому связь — **многие-ко-многим** через `product_categories`, с признаком
основной категории (`is_primary`) для построения хлебных крошек/канонического URL.

| Критерий | M2M `product_categories` (выбрано) | одна `products.category_id` |
|---|---|---|
| Реальные сценарии ИМ (товар в нескольких разделах, акции) | поддержаны | не поддержаны без костылей |
| Канонический путь / хлебные крошки | через `is_primary` | тривиально, но негибко |
| Сложность | +1 связующая таблица | проще |
| Расширяемость (промо-категории, подборки — Этап 5) | бесплатно | потребует рефактор схемы с данными |

**Выбор — M2M.** Одна категория проще, но почти любой реальный ИМ рано или поздно требует
товар в нескольких разделах; менять это позже — миграция с данными (нарушает «обновление не
ломает данные», `docs/02`). Закладываем M2M сразу, удобство «одной категории» сохраняем флагом
`is_primary`.

**Последствия:**
- Новые таблицы Этапа 2: `attributes`, `attribute_values`, `product_attributes`,
  `product_categories` (+ `categories`, `products`, `product_variants`, `product_media`,
  `inventory`). DDL и индексы — `docs/05-этап2-каталог.md` §2.
- Характеристики и категории конкретного магазина — **данные** (seed/UI/импорт), код единый
  (ADR-003). Импорт-заготовка (CSV) опирается на коды атрибутов как стабильные ключи.
- Чуть выше сложность схемы и запросов (JOIN'ы) — приемлемая цена за фильтрацию, целостность
  значений и универсальность. Презентационное чтение ускоряется JSONB-кешем без потери истины.
- Детали (типы атрибутов, варианты, медиа, остатки, индексы, FTS) — `docs/05-этап2-каталог.md`.

## ADR-008. Admik как headless commerce backend + публичный Storefront API

**Контекст (новый факт, 2026-06-15; уточнено 2026-06-22).** Платформа Admik «приставляется» к
внешним интернет-магазинам с разной спецификой каталога. Это **отдельные витрины** со своим
фронтендом, а не страницы, рендеримые Admik. Следовательно, для них Admik — **headless commerce
backend**: источник каталога, приёмник заказов и шлюз СДЭК, плюс админ-UI для управления;
**собственной публичной витрины Admik не отдаёт**.
> **Эволюция размещения витрин.** Изначально витрины планировались как независимые SPA. По факту
> реализации возможна иная топология: витрина (чистый потребитель Storefront API) может подниматься
> **отдельным контейнером `storefront` в общем `docker compose` на том же VPS, что и Admik**, а её
> исходный код — быть **поглощён в монорепо Admik** отдельным подкаталогом (с сохранением истории).
> Витрина также может оставаться полностью отдельным проектом. Для архитектурного решения это
> несущественно: контур Storefront API один и тот же независимо от способа хостинга и хранения
> витрины.
Витринам нужен **публичный программный интерфейс** для чтения каталога и оформления заказа. Это
определяющее архитектурное решение (новый класс потребителей API, безопасность, версионирование),
поэтому выносится в отдельный ADR. Подробный разбор витрин и следствий — `docs/06-референсные-магазины.md`.

**Решение.** Ввести **публичный Storefront API** под префиксом **`/api/storefront/v1/*`** —
отдельный от внутреннего админ-API контур, спроектированный для внешних витрин:

- **Состав (верхнеуровнево):**
  - **Каталог (read-only):** категории (дерево), товары (список с фасетной фильтрацией/пагинацией/
    поиском, карточка по slug), варианты, характеристики, **бренды** (фасет/страница), цены и
    **скидка** (`compare_at_price` → вычисляемый %), флаги (`is_featured`, вычисляемая «новизна»),
    остатки/доступность. Отдаёт **только опубликованные** (`status='active'`) сущности.
  - **Заказы:** создание заказа из корзины витрины, чтение статуса/трекинга заказа. Семантика и
    модель — **Этап 3** (`orders`).
  - **СДЭК:** расчёт стоимости/срока доставки, список ПВЗ. Реализация — **Этап 4** (`cdek`).
- **Аутентификация витрины:** по **API-ключу** магазина (заголовок) и/или **allowlist домена**
  (Origin/Referer). Ключи — в env/настройках, не в репозитории (ADR-006, `docs/02`). Публичные
  read-эндпоинты каталога могут быть доступны по домену; мутации (создание заказа) — строже.
- **CORS:** разрешённые Origin — из настроек магазина (домены витрин), не «*».
- **Версионирование:** в пути (`/v1`) — обратная совместимость витрин при эволюции API.
- **Rate-limit:** через Redis (ADR-002/006), отдельные лимиты на публичный контур; mock-fallback
  без Redis (demo).
- **Изоляция от админки:** Storefront API не использует админ-сессии/RBAC-права персонала; это
  отдельный публичный контур с собственной моделью доступа (ключ/домен). Внутренние Server Actions
  (`defineAction`, `catalog.*`/`orders.*`) остаются для админ-UI.

**Этапность.** **Каталожные read-эндпоинты** можно отдать уже в рамках **Этапа 2.x** (каталог
готов, Этап 2). **Заказы** — Этап 3, **СДЭК-расчёт/ПВЗ** — Этап 4. Детальная реализация контрактов —
в соответствующих этапах; здесь фиксируется направление и состав.

**Эскиз эндпоинтов (контракты уточняются поэтапно):**
```
GET  /api/storefront/v1/categories                 # дерево/список категорий (active)
GET  /api/storefront/v1/brands                      # бренды (active) — фасет/страницы
GET  /api/storefront/v1/products?category=&brand=&attr:color=&sort=&page=&q=&onSale=&isNew=
                                                    # список с фасетами/пагинацией/поиском (active)
GET  /api/storefront/v1/products/{slug}             # карточка: варианты, атрибуты, медиа, цена/скидка, бренд, остаток
POST /api/storefront/v1/orders                      # создать заказ из корзины          (Этап 3)
GET  /api/storefront/v1/orders/{id}                 # статус/трекинг заказа             (Этап 3/4)
POST /api/storefront/v1/delivery/cdek/calculate     # расчёт доставки СДЭК              (Этап 4)
GET  /api/storefront/v1/delivery/cdek/pvz?city=     # пункты выдачи СДЭК                (Этап 4)
```

**Рассмотренная альтернатива — отдавать витрину из Admik (не headless):** отвергнута. Витрины
существуют как самостоятельные фронтенды со своим UX/деплоем; навязывать им рендеринг из Admik
противоречит факту и снижает универсальность. Headless-контур переиспользуется любой будущей
витриной без правок ядра (ADR-003).

**Последствия:**
- Новый публичный контур `/api/storefront/v1/*` (роуты Next.js route handlers), не пересекающийся
  с админ-сессиями; читает через те же репозитории каталога (`lib/catalog/repository.ts`).
- Новые настройки магазина: API-ключ(и) витрин, allowlist доменов/Origin, лимиты — env/таблица
  настроек (Этап 5), не хардкод.
- Каталог должен отдавать поля, которые витрины показывают (цена «было»/скидка, флаги, бренд,
  доступность) — обеспечивается миграцией `0011` и ADR-009.
- Безопасность: публичный контур не раскрывает черновики/архив, не даёт доступа к админ-данным;
  rate-limit и CORS обязательны. Детали и разбор витрин — `docs/06-референсные-магазины.md`.

## ADR-009. Модель ценообразования, флагов товара и брендов каталога

**Контекст.** Референсные витрины (`docs/06`) выявили, что текущая модель каталога (`docs/05`,
миграции `0005–0010`) не покрывает: (1) **акционную цену** «было/стало» со скидкой % (например,
старая цена + новая + «–21%»); (2) **флаги/подборки** товара («New»; «Хиты/Новинки/Со скидкой»);
(3) **бренд** как фильтруемую сущность с лого/страницей (у одних магазинов бренды есть, у других —
их нет); (4) **подбор по совместимости** (VIN/fitment — актуально для части магазинов). Нужно
зафиксировать модель так, чтобы она была **минимальной, универсальной и опциональной** (ADR-003):
работала и для магазина с брендами/скидками, и для магазина без брендов/скидок, без хардкода тематики.
Эти решения определяют схему (`0011`), вычисления цен и точки расширения — поэтому отдельный ADR.

**Решение.**

1. **Акционная цена — единое поле `compare_at_price`** (в `products` и `product_variants`),
   **не** пара `old_price`/`sale_price`. Источник истины платимой цены остаётся `base_price` /
   `price_override`+`price_delta` (потребляется заказом, Этап 3). `compare_at_price` — «цена для
   сравнения» (зачёркнутая «было»). **Скидка % и признак «со скидкой» вычисляются**
   (`(compare_at − effective)/compare_at`; `compare_at > effective`), **не хранятся** —
   исключаем рассинхрон производных. Это **каталожное** поле (видно на витрине всегда), отдельное
   от динамических промо-кампаний Этапа 5.2 (которые накладываются поверх).
   *Альтернатива `old_price`+`sale_price`* отвергнута: две «настоящие цены» порождают вопрос «какая
   в корзине» и рассинхрон с `base_price`; одно опциональное поле проще и достаточно.

2. **Флаги товара — гибрид «ручной флаг + вычисление»:**
   - **`is_featured`** (boolean, default false) — **ручной** маркетинговый флаг «Хит/Рекомендуемый»
     (вычислить без аналитики продаж нельзя → хранимый).
   - **`is_new`** (boolean **nullable**, троичная логика) — **NULL** ⇒ «новизна» **вычисляется**
     по `created_at >= now() − SHOP_NEW_PRODUCT_DAYS`; **true/false** ⇒ явный override редактором.
     Покрывает «New» без устаревания и без ручного труда на 2000+ SKU.
   - **«Со скидкой»** — **не флаг**, а производная из `compare_at_price` (п.1).
   *Чисто-вычисляемый* подход не покрывает «Хит», *чисто-ручной* не масштабируется на «New» —
   отсюда гибрид. Произвольные доп. бейджи магазина — через EAV-атрибут boolean (ADR-007), без
   новой миграции.

3. **Бренд — лёгкая опциональная таблица `brands`** (id, slug, name, logo_key, description, seo*) +
   `products.brand_id` (**nullable**, `ON DELETE SET NULL`), **не** EAV-атрибут. Бренд — сущность с
   собственными атрибутами (лого/описание/страница) и первоклассный фасет/фильтр; таблица
   **существует всегда** (одна схема, ADR-003), но **наполняется данными** только где есть бренды
   (у магазина без брендов — пуста, `brand_id IS NULL`). Прочие технические характеристики товара
   остаются EAV (ADR-007); бренд выделен намеренно из-за лого/страницы/фасета.
   *Альтернатива EAV-бренд* отвергнута: негде хранить лого/страницу, фасет — через лишний JOIN без
   метаданных.

4. **Совместимость (VIN/fitment) — только точка расширения, не реализуется сейчас.** Будущий
   **опциональный модуль `fitment`** (флаг в `ADMIK_MODULES`, по умолчанию выключен) с **обобщённой**
   (без хардкода конкретной ниши) моделью осей совместимости (`fitment_dimensions`/`fitment_values`/
   `product_fitments`) и VIN-декодером как интеграцией (mock-режим). В миграции `0011` **отсутствует**.
   Это держит нишевую специфику вне общего каталога и сохраняет «обновление не ломает данные» (`docs/02`).

5. **Валюта/единицы — из настроек магазина** (`SHOP_CURRENCY='RUB'` по умолчанию; единицы —
   `attributes.unit`). В каталог поля валюты не добавляются (один магазин = одна валюта, ADR-003);
   мультивалютность при необходимости — отдельный модуль.

**Последствия:**
- Миграция **`0011_catalog_pricing_flags_brands.sql`**: `compare_at_price` (products+variants),
  `is_featured`/`is_new` (products), таблица `brands` + `products.brand_id`, индексы под фасеты,
  `app`-гранты. Все поля **nullable/с дефолтом** → не ломают существующие данные и магазины без
  фич. DDL, индексы, тесты и правки `lib/catalog/*` — `docs/06-референсные-магазины.md` §3–§4.
- Новые настройки: `SHOP_CURRENCY` (дефолт `'RUB'`), `SHOP_NEW_PRODUCT_DAYS` (дефолт `30`) в
  `lib/config/env.ts`/`.env.example`.
- Storefront API (ADR-008) отдаёт вычисляемые скидку/новизну, флаги и бренд витринам.
- Модуль `fitment` — будущий опциональный этап; на текущем этапе только спроектирован.

## ADR-010. Заказы: серверный расчёт корзины (anti-tamper) и снимки позиций

**Контекст (Этап 3, `docs/07-этап3-заказы.md`).** Платформа Admik — headless commerce backend для
внешних витрин (ADR-008). Витрина держит корзину у себя и
оформляет заказ через публичный Storefront API. Возникают два архитектурных вопроса, определяющих
безопасность и целостность данных модуля `orders`: **(1)** где живёт корзина и кто считает деньги;
**(2)** как хранить позиции заказа, чтобы изменение каталога не переписывало историю покупок. Эти
решения пронизывают схему БД (`orders`/`order_items`), эндпоинты (`/cart/quote`, `/orders`) и логику
резерва остатков, поэтому выносятся в отдельный ADR.

**Решение.**

1. **Корзина — клиентская (на витрине); расчёт итога и создание заказа — на сервере.** Платформа
   **не хранит** серверную корзину гостя. Витрина передаёт в API только **намерение** —
   `variantId`/`productId` + `qty` + выбор доставки/промокода. Сервер даёт два эндпоинта:
   `POST /cart/quote` (рассчитать итог без создания) и `POST /orders` (создать заказ).
   *Альтернатива — серверная корзина в БД (таблица `carts`/`cart_items`)* отвергнута для headless:
   витрина (SPA) и так управляет состоянием корзины в браузере; дублировать его на сервере — лишняя
   запись/синхронизация/мусорные строки от брошенных корзин без выгоды. Серверу нужен расчёт и
   создание заказа, а не хранение промежуточной корзины.

2. **Сервер — единственный источник истины по ценам и итогу (anti-tamper).** Цены, скидки,
   стоимость доставки и `grand_total` **пересчитываются на сервере из каталога** (`base_price`/
   `price_override`+`price_delta`, `compare_at_price` ADR-009, остатки `inventory`) и **в `quote`, и
   повторно внутри транзакции `POST /orders`**. Любая цена/сумма, присланная витриной, **игнорируется**.
   Это защищает от подмены цены/итога на клиенте (классический риск e-commerce: «купить за 1 ₽»).
   Промокод также ре-валидируется на сервере (лимиты/срок/мин.сумма) при создании, а не доверяется
   предрасчёту из `quote`.

3. **Позиции заказа — денормализованный снимок каталога на момент покупки.** `order_items` копирует
   `name_snapshot`/`sku_snapshot`/`attributes_snapshot`/`unit_price`/`compare_at_snapshot` из
   каталога при создании заказа. Ссылки `product_id`/`variant_id` сохраняются для навигации/аналитики/
   повторного заказа, но при удалении товара → **`ON DELETE SET NULL`** (а не CASCADE) — позиция
   остаётся читаемой по снимку. Аналогично в `orders` денормализованы `promo_code` и `customer_*`
   рядом со ссылками.
   *Альтернатива — читать имя/цену позиции из каталога по FK на момент просмотра* отвергнута:
   изменение каталога (цена выросла, товар переименован/удалён) переписало бы историю прошлых
   заказов, ломая чеки, возвраты и бухгалтерию. Снимок — стандарт e-commerce (заказ фиксирует факт
   сделки на свой момент).

4. **Идемпотентность создания и атомарность.** Создание заказа — одна транзакция: ре-валидация →
   резерв остатка (`inventory.reserved += qty` атомарным `UPDATE ... WHERE quantity - reserved >= qty`)
   → выдача номера (атомарный счётчик) → вставка `orders`/`order_items`/история → учёт промокода.
   Повторная отправка с тем же `Idempotency-Key` (UNIQUE partial-индекс на `orders.idempotency_key`)
   возвращает уже созданный заказ, **не дублируя** его. Любой сбой → `ROLLBACK`, заказ не создаётся.

**Последствия:**
- Схема `0012`: `orders` (суммы считает сервер; `idempotency_key` UNIQUE partial; денормализованные
  `promo_code`/`customer_*`), `order_items` (снимок; `product_id`/`variant_id` `ON DELETE SET NULL`).
- Эндпоинты Storefront `/cart/quote` и `/orders` считают независимо; цена из тела запроса не
  используется. `GET /orders/{number}` защищён токеном заказа/email (анти-перебор номеров).
- Резерв остатков — атомарным `UPDATE ... WHERE` без блокировок строк (гонко-безопасно, без
  дедлоков); конкуренция за последний остаток разрешается на уровне условия (`docs/07` §6).
- Решение переиспользуемо любой будущей витриной без правок ядра (ADR-003): контракт «намерение →
  серверный расчёт → заказ» одинаков для всех магазинов. Полная спецификация — `docs/07-этап3-заказы.md`.

## ADR-011. СДЭК: слоистый порт carre + mock-режим + идемпотентный webhook

**Контекст.** Этап 4 интегрирует СДЭК (компания — партнёр СДЭК): расчёт стоимости, ПВЗ, создание
отправлений, отслеживание, статусы, печать документов, управление и автоматизация (ТЗ, `CLAUDE.md`).
Логика берётся из референса `carre` (PHP/Yii2, `common/components/Cdek/*`). ADR-002 уже зафиксировал,
*что* портируем (слоистость `Manager→Client→Services`, кеш OAuth-токена, webhook-идемпотентность
`UNIQUE`+IP-whitelist, cron-воркеры) и чего избегаем (deprecated-таблицы, 3-табличная модель запроса,
секреты в репозитории, in-process кеш на нескольких инстансах). Этот ADR фиксирует *как* это ложится
на стек Admik (TypeScript/Next.js/Postgres/Redis) и условия mock-режима. Поля доставки на `orders`
(`0012`), статус-машина delivery (`lib/orders/status.ts`) и заглушка стоимости
(`stubDeliveryCost → '0.00'`) уже заложены — Этап 4 должен быть аддитивным, без рефакторинга ядра.

**Решение.**
1. **Слоистый порт** в `lib/cdek/*`: `CdekManager` (фасад, ленивые синглтоны, выбор real/mock по
   конфигу) → `CdekClient` (HTTP+OAuth2 `client_credentials`, retry на 401 и сетевые ошибки) →
   `Services` (`Calculator`, `PvzService`, `OrderService`, `TrackingService`, `WebhookService`,
   `PrintService`, `StatusMap`, `DeliveryQuoteGuard`). Сервисы 1:1 соответствуют классам carre.
2. **Кеш OAuth-токена** — отдельный `TokenStore` по паттерну `lib/auth/rate-limit.ts`: Redis при
   `REDIS_URL`, иначе memory-mock с разовым `warn`. TTL = `expires_in − 60`, single-flight при добыче.
3. **Идемпотентность webhook** — `UNIQUE (cdek_uuid, status_code, status_date_time)` в `cdek_status_log`
   + `INSERT ... ON CONFLICT DO NOTHING`. Защита роута `/api/cdek/webhook`: IP-whitelist + `?key=`
   секрет; ответ всегда `200` (СДЭК не должен бесконечно ретраить). Маппинг статуса в `delivery_status`
   идёт через статус-машину `lib/orders/status.ts` (переход применяется лишь если допустим).
4. **MOCK-режим** обязателен (ADR-002, `docs/02`): при пустых `CDEK_ACCOUNT`/`CDEK_SECRET`
   `CdekManager.isMock = true` — расчёт по формуле, ПВЗ из фикстур, фейковые `cdek_uuid`/трек,
   `is_mock`-флаги в БД, разовый `console.warn`. Demo-магазин и CI работают без боевых ключей.
5. **Аддитивность:** заглушка `stubDeliveryCost` заменяется `computeDeliveryCost` через тонкий
   адаптер `lib/orders/delivery-cost.ts` (без жёсткой связки `lib/orders → lib/cdek`; при выключенном
   модуле `cdek` поведение Этапа 3 сохраняется). Storefront-эндпоинты — под `/api/storefront/v1/
   delivery/cdek/*` (ADR-008), `from_location` всегда серверный (анти-tamper, ADR-010).
6. **БД с `0017`** (идемпотентные миграции, стиль `0012`): `cdek_shipments` (1:1 к `orders`,
   денормализованные `cdek_uuid`/`cdek_track` остаются на `orders` для списков/витрины) +
   `cdek_status_log`; `0018` добавляет `weight_g`/габариты товару и варианту (в каталоге их не было),
   дефолты — из env (`CDEK_DEFAULT_*`, аналог `cdek-dimensions.php`).

**Рассмотренная альтернатива — прямое использование `lib/cdek` внутри `lib/orders`** — отвергнута:
ядро заказов не должно зависеть от модуля, который может быть выключен; адаптер
`computeDeliveryCost` сохраняет работоспособность заказов без СДЭК.

**Последствия:**
- Новый модуль `lib/cdek/*` (флаг `cdek` уже в `lib/config/modules.ts`); при выключении —
  storefront-роуты `404`, admin-блок скрыт, cron — no-op, стоимость остаётся `'0.00'`.
- Миграции `0017_cdek_shipments`, `0018_product_weight_dims` (+ опц. `0019` кеш ПВЗ, если нет Redis).
- `env.ts` дополняется `CDEK_*` (`.optional()` → отсутствие = mock). Новых runtime-зависимостей нет
  (`fetch`/`undici`, `ioredis` и `zod` уже есть).
- Admin через `defineAction(permission:'cdek.manage')` + audit `cdek.*`; cron как защищённые роуты/
  скрипты, расписание — задача DevOps (Docker).
- Полная спецификация и декомпозиция на пакеты A–G — `docs/08-этап4-cdek.md`.

## ADR-012. CMS-страницы: модель `cms_pages`/`cms_page_sections` (JSONB-секции) и санитизация rich-text

**Контекст (Этап 5.1, `docs/11-этап5-cms-промо-seo-настройки.md`).** Платформе нужен управляемый
контент (лендинги, «о компании», промо-страницы) для витрин. Референс `2x2` решает это таблицей
`page_sections` (`page_path`, `section_key`, `content jsonb`) — плоский composite-key на все блоки всех
страниц (`010_page_sections.sql`). Разведка 2x2 рекомендовала добавить `website_id` для многосайтовости
и пометила два gap'а: отсутствие черновиков (только boolean `enabled`, контент-менеджер пишет сразу в
prod), отсутствие ограничения размера JSONB и версионирования. Модуль `cms`, права `cms.read`/`cms.write`
и пункт меню `/admin/cms` уже заведены (Этап 1). Нужно зафиксировать модель хранения и инвариант
безопасности rich-text.

**Решение.**
1. **Нормализованная модель вместо плоской.** Страница — `cms_pages` (id/slug citext UNIQUE/title/SEO),
   её блоки — дочерняя `cms_page_sections` (`page_id` FK `ON DELETE CASCADE`, `section_key`, `type`,
   `content jsonb`, `display_order`, `enabled`), а не одна плоская `(page_path, section_key)`. Даёт
   ссылочную целостность через CASCADE и атомарный reorder в транзакции.
2. **Дискриминированная Zod-валидация `content` по полю `type`** (`hero/text/banner/products_grid/faq/
   cta/gallery`) ПЕРЕД записью + CHECK `pg_column_size(content) < 65536` как страховка размера блока
   (закрывает gap «нет maxsize»). Каждый тип секции — свой контракт.
3. **Серверная санитизация rich-text — обязательный анти-XSS инвариант.** Tiptap отдаёт HTML/JSON с
   клиента; при `upsertCmsSection` сервер прогоняет `sanitizeHtml` (whitelist тегов/атрибутов, удаление
   `script`/`on*`/`javascript:`). Доверие клиенту запрещено — по аналогии с серверным расчётом цен
   (ADR-010). Без санитизации — хранимый XSS на витрине.
4. **Триада статусов `draft/published/archived`** вместо boolean `enabled` из 2x2 (закрывает gap «нет
   черновиков»); `published_at` ставится при первой публикации. Storefront отдаёт ТОЛЬКО `published`.
5. **Опциональные ревизии** `cms_page_revisions` (snapshot JSON при публикации) — закрывает gap
   «версионирование»; на MVP можно отложить (аудит правок и так идёт через `audit_log`; ревизии нужны
   для отката контента, а не для аудита).

**Рассмотренные альтернативы.**
- *Плоская `(page_path, section_key)` с `website_id` (рекомендация 2x2)* — отвергнута: `website_id`
  противоречит ADR-003 (1 магазин = 1 БД, мультитенантность через изоляцию БД), а плоская таблица без
  FK создаёт риск рассинхрона (в 2x2 был баг `homepage_sections` vs `page_sections`).
- *Хранить только Tiptap-JSON и рендерить на витрине* — частично принято как defense-in-depth, но
  серверная санитизация всё равно обязательна.

**Последствия.**
- Миграции `0022_cms_pages` (SEO/sitemap-поля включены сразу), `0023_cms_page_sections` (+опц.
  `cms_page_revisions`). Без `tenant_id`.
- Все мутации — через `defineAction(permission:'cms.write')` + `assertCmsEnabled()` + audit `cms.*`;
  репозиторий — только SELECT. Slug переиспользуется из `lib/catalog/slug` (не дублируется).
- Storefront `/api/storefront/v1/pages/[slug]` под `runStorefront(..., {module:'cms'})` (не дефолтный
  `catalog`); DTO `toPublicPageDto` скрывает id/status/timestamps/revisions/draft, отдаёт `enabled`-секции
  по `display_order`. `products_grid` хранит slug-фильтр, не FK на товары (модули `cms`/`catalog`
  независимы).
- Новые зависимости: `@tiptap/react` (admin) + серверный санитайзер (`sanitize-html`/`isomorphic-dompurify`).
- ADR-005 (RBAC) и ADR-008 (Storefront) не меняются — права `cms.*` и пайплайн `runStorefront`
  переиспользуются. Полная спецификация — `docs/11`, подсистема 5.1.

## ADR-013. Конфигурация магазина: env-дефолт ⊕ БД-оверрайд (`shop_settings`)

**Контекст (Этап 5.4, `docs/11`).** Сегодня магазино-специфика зашита в env (`lib/config/env.ts`:
`SHOP_NAME`/`SHOP_LOGO_URL`/`SHOP_CURRENCY`/`SHOP_NEW_PRODUCT_DAYS`/`SHOP_FREE_DELIVERY_THRESHOLD`/
`SHOP_ORDER_PREFIX`), а состав модулей — в `ADMIK_MODULES`. Любой новый магазин требует правки
окружения/кода, что противоречит требованию «копировать-вставить новый магазин без переписывания кода»
(`CLAUDE.md`). Референс 2x2 хранит глобальные настройки в `site_settings(key, value jsonb)`. Нужно
сделать конфигурацию управляемой через UI без правки кода, не ломая существующие стенды.

**Решение.**
1. **`shop_settings(setting_key citext PK, value jsonb, updated_at, updated_by)`** — гибрид
   key-value+JSONB (паттерн 2x2 `site_settings`, но **без `website_id`** — ADR-003). Одна строка = одна
   логическая группа (`branding/currency/units/contacts/legal_entity/catalog/delivery/orders/
   module_overrides/seo`), типизированная Zod-схемой ключа; CHECK `jsonb_typeof(value)='object'`.
2. **Приоритет слияния: `env-дефолт ⊕ строка БД`**, частичный merge на уровне полей внутри `value`.
   Отсутствие строки/поля → значение из env. Реализуется чистой функцией `mergeSettings(env, dbRows)`
   (тестируется без БД) + мемоизирующим `getEffectiveSettings()`/`getEffectiveModules()`. env остаётся
   безопасным дефолтом для bootstrap.
3. **`module_overrides` поверх `ADMIK_MODULES`** — частичный `Record<ModuleName, boolean>`; UI
   `/admin/settings/modules` включает/выключает модули без правки env. Отсутствие поля → значение env.
4. **`shop_settings` — core-always-on** (без флага `module` у пункта меню и эндпоинта): её читают все
   модули + каркас + витрина, и она же управляет оверрайдом модулей — поэтому не может прятаться за
   флагом, который сама переключает (иначе «курица-яйцо»: выключив модуль, потеряли бы доступ к
   настройкам). Защита от self-lock: `settings` неотключаема; UI-предупреждение; reset = `DELETE` →
   возврат к env.

**Рассмотренные альтернативы.**
- *Оставить всё в env* — отвергнута: новый магазин требует доступа к серверу/деплою, а не настройки
  через UI (нарушает «копировать-вставить»).
- *Таблица-на-настройку* (как старая «store» в 2x2) — отвергнута: размножение схемы при каждой новой
  настройке; JSONB-значение с Zod-валидацией гибче.
- *`website_id` в `shop_settings`* — отвергнута (ADR-003).

**Последствия.**
- Миграции `0019_shop_settings` (DDL), `0020_shop_settings_seed` (идемпотентный seed `ON CONFLICT DO
  NOTHING` — не затирает правки администратора).
- Меняется **источник значения** в `catalog`/`orders` (`newProductDays`/`freeDeliveryThreshold` теперь
  из `getEffectiveSettings()`), при сохранении контрактов DTO — при пустой БД поведение идентично
  текущим стендам (обратная совместимость).
- Эффективный состав модулей вычисляется `getEffectiveModules(env, dbOverrides)` поверх
  `getEnabledModules(env)`. Кеш — module-level memo с явной инвалидацией в каждом settings-action
  (Redis — задел на будущее, не сейчас: 1 магазин = 1 процесс).
- Мутации через `defineAction(permission:'settings.manage')` (право уже core); публичный `GET
  /settings` через `runStorefront` без `options.module`; `toPublicSettingsDto` скрывает приватные поля
  (`bankDetails`, `updated_by`, internal-ключи). Деньги в копейках. Полная спецификация — `docs/11`,
  подсистема 5.4.

## ADR-014. Движок акций «N по M» и комбинируемость промо-механик

**Контекст (Этап 5.2, `docs/11`).** Этап 3 заложил `promo_codes` (`kind` percent/fixed/free_delivery/
**bogo**, поля `bogo_buy_qty`/`bogo_pay_qty`), `promo_redemptions`, серверный расчёт корзины
(`lib/orders/pricing.ts::calculateQuote`, `quoteCart`/`createOrder`), но ветка `kind='bogo'` —
заглушка (`discount=0`, TODO). Референс `carre` поддерживает только % и фиксированную сумму на всю
корзину; «N по M», скидки на категорию/бренд/набор и комбинируемость — gap. Нужно реализовать
исполнение «купи N плати M» и scope-таргетинг, не трогая рабочие percent/fixed/free_delivery и сохранив
anti-tamper (ADR-010).

**Решение.**
1. **Расширение `promo_codes` одним `kind`-полем + новая `promo_targets`**, а не отдельная таблица на
   каждый тип акции. Добавляются `apply_scope` (`cart/category/brand/set`), `priority`, `stackable`,
   `min_qty`, задел `gift_*`. `promo_targets` (FK на `categories/brands/products/product_variants`)
   хранит, к каким товарам применяется scope/N×M-группировка — одна акция может бить по нескольким
   категориям/брендам.
2. **Детерминированный алгоритм N×M:** `floor(qty / bogo_buy_qty)` полных групп, в каждой бесплатны
   `bogo_buy_qty − bogo_pay_qty` **самых дешёвых** единиц; группировка — в пределах scope (а не всей
   корзины при scope≠cart). Скидка = `freeUnits × unitPrice` в целых копейках (без процентов → без
   ошибок округления).
3. **Правило комбинируемости (MVP):** применяется не более **одной** не-stackable акции (выбор
   детерминирован: по `priority asc`, tie-break по `code`) + любое число `stackable`, каждая считается
   от ОСТАТКА после предыдущих; суммарный discount жёстко `clamp ≤ itemsTotal` и `≥ 0`. **Один
   `promoCode` на заказ** в Этапе 5.2 (multi-promo — расширение текущей модели `orders.promo_code_id`,
   отложено).
4. **Вся скидка — на сервере (расширяет ADR-010):** принадлежность линии scope определяет сервер из
   каталога (`product_categories`/`products.brand_id`) через `getPromoWithTargets`; `pricing` остаётся
   чистым (принимает уже размеченные линии). Новые чистые функции `bogoDiscountMinor`/`scopeDiscountMinor`
   добавляются **рядом** со старым `promoDiscountMinor` (сигнатуры percent/fixed не меняются →
   существующие тесты зелёные). Скидка идёт агрегатом в `orders.discount_total` +
   `promo_redemptions.discount_applied` (без помарочной разбивки `order_items`). **Инвариант денег:**
   `bogoDiscountMinor`/`scopeDiscountMinor` считают в **копейках** (`int`), а `discount_total`/
   `discount_applied` — `numeric(14,2)` в **рублях** (схема Этапа 3); при записи копейки конвертируются
   `fromMinor` → рубли, ровно как существующая ветка percent/fixed (запрещено писать копейки в numeric-поля).
5. **gift-позиция (товар-подарок) — задел, исполнение отложено.** Колонки `gift_*` + UI за фичефлагом;
   вставка подарка в `order_items` (резерв остатка) вынесена в отдельный будущий пакет/ADR, чтобы не
   дестабилизировать транзакцию `createOrder`. N×M/scope-скидку (без добавления товаров) реализуем полностью.

**Рассмотренные альтернативы.**
- *Отдельная подсистема/таблица акций* — отвергнута: дублирует `defineAction`/`quoteCart`/`createOrder`/
  `promo_redemptions`; задел bogo уже в `0014`/типах/валидаторе.
- *Помарочная разбивка скидки по `order_items`* — отложена: требует ALTER `order_items`; агрегат в
  `discount_total` сохраняет аддитивность.

**Последствия.**
- Миграция `0024_promo_mechanics_nxm` (ALTER `promo_codes` `ADD COLUMN IF NOT EXISTS` + CHECK
  `promo_bogo_pair_chk` через DO-блок; `CREATE TABLE promo_targets`). Без backfill.
- Модуль `orders` (денежный движок, не контент); мутации через `defineAction(permission:'orders.write')`
  + `assertOrdersEnabled()`; новых прав/nav-пунктов нет. Форма `/admin/promo` расширяется
  (scope/targets/bogo/priority/stackable).
- В ответе `/cart/quote` `quote.promo.discount` для bogo теперь корректен (раньше 0); `createOrder`
  применяет ту же логику в транзакции (ре-валидация). Опц. `GET /api/storefront/v1/promotions`
  (read-only бейджи «3 по 2»). Деньги в копейках, без `tenant_id`.
- ADR-010 (anti-tamper) расширен, не изменён. Полная спецификация — `docs/11`, подсистема 5.2.
  Примечание о нумерации: ранее в ADR-011 упоминалась *опциональная* миграция `0019` (кеш ПВЗ СДЭК) —
  она не создавалась; Этап 5 занимает диапазон `0019`–`0024` (см. `docs/11` §2), кеш ПВЗ при
  необходимости получит свободный номер после `0024`.

## ADR-015. Стратегия бэкапов и обновлений для single-VPS-магазина

**Контекст.** ADR-003 («1 магазин = 1 БД = 1 VPS») и `docs/02` («копировать-вставить» сторонним
человеком) задают эксплуатационную модель: один процесс `app` за Caddy, локальные PostgreSQL/MinIO в
compose. Этап 6 должен дать продакшн-готовность (бэкапы, мониторинг, обновления без простоев) **без**
оркестратора (k8s) и **без** внешних SaaS, оставаясь универсальным (всё через `.env`). Нужно
зафиксировать, как делаются бэкапы и как обновляется магазин, чтобы обновление не теряло данные.

**Решение.**
1. **Бэкап = логический дамп + зеркало медиа, не Docker-тома.** PostgreSQL — `pg_dump -Fc`; медиа —
   `mc mirror`. Расписание/ретенция — отдельный compose-сервис-планировщик (паттерн существующего
   `cron`), параметры из `.env`. **Восстановимость обязательна к проверке:** периодический
   `restore-drill` восстанавливает последний дамп в одноразовую БД и читает ключевые таблицы; «бэкап
   без проверенного restore — не бэкап». Offsite — опциональная env-настраиваемая точка (rclone/`mc`),
   провайдер не хардкодится.
2. **Обновление — near-zero-downtime в рамках одного VPS:** собрать новый образ заранее → обязательный
   бэкап → накат **только аддитивных** миграций (старый код совместим с новой схемой) → пересоздать
   `app` с **health-gate** (`/api/health` deep-check) → при провале **автоматический откат** на
   предыдущий образ. Деструктивные изменения схемы — только многошаговым **expand/contract** через
   несколько релизов, никогда в одном выкате с кодом, который от них зависит. Идемпотентность миграций
   (`IF NOT EXISTS`) **не равна** обратной совместимости — последняя обеспечивается правилом
   аддитивности и статической проверкой (`check-migrations.sh`).
3. **Мониторинг/логи — self-hosted уровня compose:** JSON-логи приложения в stdout (`lib/logger.ts`),
   deep-health-probe, лёгкий внешний монитор + алерты через webhook/email. Тяжёлый стек наблюдаемости
   (Prometheus/Grafana/Loki/Sentry) — точка расширения поверх готовых JSON-логов и health-probe, в
   базовую поставку не входит.

**Рассмотренные альтернативы.** *(а)* Полагаться на снапшоты тома/диска VPS вместо логических дампов —
отвергнуто: непереносимо между провайдерами, не даёт гранулярного restore, не проверяет целостность
схемы. *(б)* Blue-green/rolling на нескольких репликах — отвергнуто: противоречит ADR-003 (одна нода),
избыточно для админки ИМ. *(в)* Внешний SaaS-мониторинг по умолчанию — отвергнуто: добавляет внешнюю
зависимость и стоимость, ломает «copy-paste без аккаунтов»; оставлено как опция.

**Последствия.** Новые скрипты `scripts/{smoke,deploy,backup,restore,restore-drill,update,
check-migrations,healthcheck-monitor}.sh`, `Makefile`, сервис `backup` и том `backups` в compose,
`lib/logger.ts`, deep-режим `/api/health`. Новые env: `BACKUP_*`, `LOG_*`, `ALERT_WEBHOOK_URL`,
`SMOKE_BASE_URL`. **Правило аддитивных миграций становится обязательным** для всех будущих этапов
(линтер в CI). Все скрипты универсальны (ADR-003) и проверяемы статически в dev (shellcheck) +
интеграционно на VPS/CI.

## ADR-016 — Выдача товара-подарка (gift_*) отдельной позицией заказа (пост-роадмап)

**Статус:** принято (2026-06-16). Реализует отложенный «задел» ADR-014 (поля `promo_codes.gift_*`).

**Контекст.** ADR-014 заложил в схему поля `gift_product_id`/`gift_variant_id`/`gift_qty`
(товар-подарок к промокоду), но фактическая ВЫДАЧА подарка отдельной строкой заказа была отложена.
Нужно выдавать подарок, не нарушая anti-tamper (ADR-010) и инвариант «деньги считает сервер».

**Решение (MVP).**
1. **Подарок — отдельная позиция заказа** (`order_items`) с `unit_price = 0`, `line_total = 0`,
   `compare_at = каталожная цена` (показывает «ценность»), новый флаг `is_gift = true` (миграция
   `0025`, аддитивно). Флаг отличает подарок от обычной позиции, не полагаясь на цену 0.
2. **Подарок вне итога.** Не входит в `itemsTotal`/скидку/порог бесплатной доставки — считается
   ОТДЕЛЬНО и добавляется ПОСЛЕ расчёта (чистый билдер `giftQuoteLine`, `lib/orders/pricing.ts`).
3. **Триггер.** Подарок выдаётся, когда применённый промокод ВАЛИДЕН и у него `gift_qty ≥ 1` и задан
   `gift_variant_id` ИЛИ `gift_product_id` (вариант приоритетен). Любой `kind` промокода может нести
   подарок (аддитивно к percent/fixed/free_delivery/bogo). Товар/цена подарка берутся из каталога
   (anti-tamper), не из запроса.
4. **Инвентарь — best-effort.** Подарок резервируется как обычная позиция (анти-оверселл,
   атомарный `reserveUnit`). Если остатка подарка нет — подарок просто НЕ добавляется, заказ НЕ
   падает (бесплатный подарок не должен блокировать оплаченный заказ). `quoteCart` показывает подарок
   в превью только при наличии остатка; `createOrder` повторно проверяет резерв на момент оформления.
5. **Один подарок на заказ** (следствие MVP «один промокод на заказ», см. ADR-014, бэклог M1).

**Рассмотренные альтернативы.** *(а)* Отдельная таблица `order_gifts` — отвергнуто: подарок — та же
доменная сущность «позиция заказа» (снимок ADR-010), отдельная таблица дублировала бы снимок и логику
отгрузки/возврата. *(б)* Маркировать подарок только ценой 0 без `is_gift` — отвергнуто: цена 0 бывает
и у обычных позиций (акции), флаг даёт явную семантику для админки/витрины/возвратов. *(в)* Жёстко
падать при нехватке подарка — отвергнуто: ухудшает конверсию (оплаченный заказ не должен срываться
из-за бесплатного подарка).

**Последствия.** Миграция `0025` (`order_items.is_gift` + частичный индекс). Новый чистый
`giftQuoteLine` и `resolveGiftLine` (`lib/orders/{pricing,repository}.ts`); `quoteCart`/`createOrder`
выдают подарок; `OrderItem`/`OrderItemDto`/`QuoteLineDto` получили поле `isGift`. Расчёт итога
(anti-tamper) не изменён — подарок суммарно нулевой. Будущее (бэклог): множественные подарки,
явное уведомление витрины при недоступности подарка.

## ADR-017 — Платежи Т-Банк: слоистый порт паттерна СДЭК + mock-режим + Token-подписанный идемпотентный webhook (Этап 7)

**Статус:** принято (2026-06-17). Волны 1–2 реализованы (mock-first + боевой клиент + БД-аудит).
Полный дизайн и план волн — `docs/15-платежи-tbank.md`.

**Контекст.** Online-оплата картой/СБП через интернет-эквайринг Т-Банка (T-API). Поля оплаты на
`orders` (`payment_method`, `payment_status`, `paid_at`, `payment_ref`) и статус-машина платежа уже
заложены на Этапе 3 (`lib/orders/{types,status}.ts`). Нужен провайдер онлайн-оплаты БЕЗ рефактора ядра
заказов: при выключенном модуле/пустых ключах поведение прежнее (оплата при получении / счёт).

**Решение.**
1. **Слоистый модуль `lib/payments/tbank`** (`Manager→Client→Service`), как СДЭК (ADR-011), но **БЕЗ
   OAuth-токен-кеша** — у Т-Банка нет долгоживущего токена: каждый запрос подписывается полем `Token`
   (SHA-256 от конкатенации значений корневых полей + `Password`, отсортированных по ключу). Подпись —
   чистая функция `signToken`/`verifyNotificationToken` (`token.ts`), без сети/Redis.
2. **mock-first** при пустых `TBANK_TERMINAL_KEY`/`TBANK_PASSWORD` (`isTbankMock`, порт `isCdekMock`):
   сеть не дёргается, `Init` возвращает фейковый `PaymentId` + внутренний demo-`PaymentURL`. `client`
   в mock-режиме не инстанцируется. Demo/CI работают без боевого терминала.
3. **Webhook** (`app/api/payments/tbank/webhook`) — server-to-server: проверка `Token` в теле (главная
   защита; опц. IP-whitelist — доп. слой) + **идемпотентность** (UNIQUE `(payment_id, status)` в
   `tbank_payment_log`, ON CONFLICT DO NOTHING); на корректно подписанном событии всегда ответ `OK`
   (HTTP 200, text/plain), чтобы Т-Банк не ретраил вечно.
4. **`payment_provider` на `orders`** (миграция `0027`, NULLABLE + CHECK `tbank|manual`) — различает
   провайдеров онлайн-оплаты ОРТОГОНАЛЬНО `payment_method` (`card`/`sbp`). Статусы Т-Банка покрываются
   существующим `PaymentStatus` (`pending|authorized|paid|failed|refunded`) — новых значений в БД нет;
   `authorized` = hold двухстадийной оплаты.
5. **Суммы/статусы — только с сервера** (anti-tamper, ADR-010): `Amount` считается из
   `orders.grand_total` в БД, не из запроса витрины; `payment_status` меняется лишь webhook-ом/`GetState`,
   не доверяя `SuccessURL`. Переход применяется через `canTransition('payment', …)`.

**Рассмотренные альтернативы.** *(а)* Значение `payment_method='tbank'` — отвергнуто: провайдер
ортогонален способу оплаты (карта/СБП), смешивание усложнило бы аналитику и поддержку нескольких
провайдеров. *(б)* Хранить только `PaymentId` в `payment_ref` без отдельной таблицы — отвергнуто для
аудита/рефандов/идемпотентности webhook (нужен лог событий с UNIQUE-ключом). *(в)* OAuth-токен-кеш как у
СДЭК — не нужен: у Т-Банка авторизация подписью на каждом запросе (упрощает клиент).

**Последствия.** Миграция `0027` (`orders.payment_provider` + таблица `tbank_payment_log`, аддитивно).
Новый флаг модуля `payments` (`lib/config/modules.ts` + `moduleOverridesSchema`), 16 `TBANK_*` в
`env.ts`/`.env.example` (3 режима: mock / тестовый контур / боевой). Ядро заказов не изменено. Будущее
(волны 3–4): СБП (`GetQr`), чек 54-ФЗ (`Receipt`), двухстадийность (`Confirm`/`Cancel` под правом
`payments.manage`), cron-синхронизация «зависших» платежей через `GetState`, боевой запуск.

## ADR-019 — Процесс-широкие in-memory синглтоны в Next App Router хранятся на `globalThis`, не в модульных `let`

**Статус:** принято (2026-06-24). Исправляет баг кэша эффективных настроек (ADR-013).

**Контекст.** Слой эффективных настроек (env ⊕ БД, ADR-013) в `lib/config/settings.ts`
мемоизирует чтение БД процесс-локально. Memo жил в модульных переменных (`let cached` /
`let inflight` / `let cacheEpoch`). Обнаружен баг (2026-06-24): **Next.js инстанцирует один и тот
же серверный модуль в РАЗНЫХ бандлах** — отдельно для Server Actions и отдельно для Route
Handlers/RSC. Из-за этого у каждого бандла была **своя копия** модульного memo:
`invalidateSettingsCache()`, вызываемый из settings-action, сбрасывал memo своего экземпляра
модуля, а публичный `GET /api/storefront/v1/settings` (route-handler) читал memo **другого**
экземпляра и отдавал устаревший снимок настроек — вплоть до рестарта процесса. **Симптом:** правка
контента главной в админке сохранялась в БД, но не появлялась на витрине.

**Решение.**
1. **Процесс-широкое состояние кэша вынесено на `globalThis`** под registry-Symbol
   `Symbol.for('admik.config.settings.effectiveCache')`. `globalThis` — единственный объект, общий
   для всех экземпляров модуля в одном процессе; `Symbol.for` даёт один и тот же ключ во всех
   бандлах. В результате все экземпляры модуля делят **одно** состояние кэша и **одну**
   инвалидацию → read-your-own-writes работает кросс-бандл (settings-action и storefront-route
   видят один memo).
2. **Сохранены оба защитных механизма:** epoch-guard против TOCTOU (in-flight чтение кеширует
   результат только если `cacheEpoch` не сменился за время `await read()`) и in-flight
   дедупликация (конкурентные вызовы дают одно чтение БД). Добавлен guard «снимать только свой
   in-flight промис» (`if (state.inflight === mine)`) — чтобы при инвалидации во время чтения
   промис нового read не был затёрт завершившимся старым.

**Общий принцип (guidance для всех будущих этапов).** Любой **процесс-широкий in-memory синглтон**
в Next.js App Router (кэши, счётчики, пулы), который должен быть **общим для Server Actions и Route
Handlers/RSC**, обязан храниться на `globalThis` (через `Symbol.for`), а **не** в модульных `let`.
Иначе дублирование модуля между бандлами даёт расхождение состояния (каждый бандл получает свою
копию), и инвалидация в одном бандле не видна в другом.

**Кандидаты на тот же приём (на текущий момент безвредны, не трогаются).**
- `lib/db/client.ts` — модульный `let client` (пул postgres.js). Дубль пула между бандлами
  **безвреден**: данные консистентны (источник истины — БД), дублирование лишь даёт второй пул
  соединений.
- `lib/cdek/token-cache.ts` — модульный `let defaultStore` (и in-flight в `MemoryTokenStore`).
  Дубль **само-лечится по TTL**: короткоживущий OAuth-токен в худшем случае добывается дважды (СДЭК
  выдаёт оба валидными), per-instance кеш сознательно допустим (см. обоснование в шапке файла,
  docs/08 §4).

**Оговорка.** Прямая правка `shop_settings` через **внешний SQL** по-прежнему требует рестарт app
(`invalidateSettingsCache()` из неё не вызывается). Это **не регрессия** — поведение прежнее: memo
инвалидируется только через settings-action.

**Последствия.** Изменён только `lib/config/settings.ts` (memo на `globalThis` через
registry-Symbol; epoch-guard и in-flight дедупликация сохранены; добавлен guard «снимать только
свой in-flight»). ADR-013 не меняется по существу — это исправление его реализации кэша; данный ADR
фиксирует и баг, и общий принцип для будущего кода.
