## 0. Краткое следствие (TL;DR)

- Admik — **headless commerce backend** для внешних SPA-витрин (Netlify), а не монолит с
  собственной витриной. Витрины ходят в платформу через **публичный Storefront API**
  (`/api/storefront/v1/*`): read-only каталог, создание заказа, расчёт/ПВЗ СДЭК. → **ADR-008**.
- Две **совершенно разные тематики** (медицинская форма vs автозапчасти) на **одной платформе** —
  это эмпирическое подтверждение принципа универсальности (`CLAUDE.md`, ADR-003): различия
  магазинов укладываются в **данные и настройки**, а не в код.
- Общий знаменатель обеих витрин: **₽** + **СДЭК** + **корзина/чекаут** + **промо/скидки** +
  **фасетные фильтры**. Это ровно дорожная карта Этапов 2.x → 3 → 4 → 5.
- Текущая модель каталога (`docs/05`, миграции `0005–0010`) покрывает большинство, но **не
  имеет**: старой/акционной цены и скидки, флагов товара (New/Хит/Рекомендуемый), бренда как
  отдельной фильтруемой сущности, механики совместимости (подбор по VIN). Эти 4 пробела
  закрываются **минимально и универсально** одной миграцией `0011` (+ точкой расширения под
  fitment, без реализации). → §3, **ADR-009**.

## 1. Витрина №1 — THE CASE (премиальная медицинская форма)

**URL:** https://darling-lebkuchen-9b78cc.netlify.app/

### 1.1. Что это и как устроено

- Тематика: **премиальная медицинская одежда** (форма для врачей/медперсонала).
- Структура каталога: верхний раздел **Женское / Мужское**, внутри — **Костюмы / Халаты /
  Аксессуары**. → дерево категорий с пересечением «пол × тип изделия».
- **Варианты:** размеры и крой (модель/посадка) — типичные вариативные характеристики одежды.
- **Бейдж «New»** на части товаров — флаг новинки на карточке.
- Цены в **₽**.
- **Корзина**, **избранное (wishlist)**, **личный кабинет** покупателя.
- **Доставка СДЭК:** Москва 1–2 дня, регионы 3–5 дней.
- **Оплата:** СДЭК PAY, банковские карты, **СБП**.
- **Отслеживание заказа в ЛК.**

### 1.2. Что отсюда следует для платформы

| Наблюдение на витрине | Следствие для Admik |
|---|---|
| Дерево «Женское/Мужское → Костюмы/Халаты/Аксессуары» | покрыто `categories` (adjacency-tree) + M2M `product_categories` (товар в нескольких разделах) — **есть** |
| Размеры/крой как варианты | покрыто `product_variants` + вариативные атрибуты (`attributes.is_variant`) — **есть** |
| Бейдж «New» | **gap:** нет флага новинки → §3.2 (`products.is_new` + вычисляемая «новизна» по `created_at`) |
| Цена в ₽ | валюта — из настроек магазина (`SHOP_CURRENCY='RUB'`), **не хардкод** → §3.5 |
| Корзина, избранное, ЛК покупателя | корзина/заказы — Этап 3; ЛК покупателя и избранное — **на стороне витрины** (SPA), платформа отдаёт каталог и принимает заказ через Storefront API (ADR-008); покупательские аккаунты — будущий под-модуль |
| Доставка СДЭК (сроки по регионам) | Этап 4 (расчёт/ПВЗ/трекинг) — **будущий этап**, но Storefront API сразу проектирует точки для расчёта/ПВЗ |
| Оплата СДЭК PAY / карты / СБП | платежи — отдельный модуль (Этап 3/5); Storefront API создаёт заказ, оплата инициируется витриной/платформой |
| Отслеживание в ЛК | трекинг СДЭК (Этап 4) + чтение статуса заказа через Storefront API |

### 2.1. Что это и как устроено

| Наблюдение на витрине | Следствие для Admik |
|---|---|

| Дерево по системам авто | покрыто `categories` — **есть** (другой набор данных, тот же код) |

| Подбор по VIN (совместимость) | **gap/будущее:** отдельный опциональный модуль **vehicle fitment** → §3.4 (только точка расширения, **не реализуем сейчас**) |

| Промокоды, «3 по 2» | промо-механики — Этап 5.2 (усиление: промокоды, BOGO/«N по M») + применение в заказе (Этап 3) |

## 3. Gap-анализ модели каталога и решения (миграция `0011`)

Текущая схема (`docs/05`, миграции `0005–0010`): `products` имеет `base_price`, `status`,
`attributes_cache`; есть `product_variants` (с `price_override`/`price_delta`), EAV-атрибуты,
`inventory`, M2M-категории. **Нет:** старой/акционной цены, флагов товара (new/hit/featured),
бренда как поля/сущности, механики совместимости (VIN).

Принципы решений (ADR-003, ADR-007): **минимально, универсально, опционально** — новые поля
**nullable / с дефолтом**, чтобы магазин без них (THE CASE без брендов, любой магазин без акций)
работал без изменений; никаких хардкодов тематики; идемпотентные `ALTER ... ADD COLUMN IF NOT
EXISTS`. Обоснование выбора — **ADR-009**.

### 3.1. Старая/акционная цена + скидка %

**Проблема.** Gang Auto показывает старую (зачёркнутую) цену и текущую + процент скидки. Это
**поле каталога** (отображается на витрине всегда, а не только в рамках промо-кампании).

**Решение — единое поле `compare_at_price`** (модель Shopify/BigCommerce), а **не** пара
`old_price`/`sale_price`:
- **Источник истины актуальной цены остаётся `base_price`** (товар) / `price_override`+`price_delta`
  (вариант) — то, что покупатель платит. Это уже потребляется корзиной/заказом (Этап 3), менять
  семантику нельзя.
- **`compare_at_price`** — «цена для сравнения» (зачёркнутая, «было»). Если `compare_at_price >
  base_price` (эффективной цены) → товар «со скидкой»; иначе поле игнорируется.
- **Скидка вычисляется, не хранится:** `discount_pct = round((compare_at_price − effective_price)
  / compare_at_price * 100)`. Не дублируем производное поле (нет рассинхрона). «Со скидкой» как
  подборка/бейдж — **вычисляемый предикат** `compare_at_price IS NOT NULL AND compare_at_price >
  effective_price`.
- Поле добавляется **и в `products`, и в `product_variants`** (у варианта может быть своя «было»).
  Эффективная «цена для сравнения» варианта = `COALESCE(variant.compare_at_price,
  product.compare_at_price)`.

**Почему `compare_at_price`, а не `old_price`+`sale_price`:** одно опциональное поле вместо двух;
не вводит вторую «настоящую цену» (избегаем рассинхрона `sale_price` ↔ `base_price` и вопроса
«какая из них в корзине»); ровно покрывает «зачёркнуто/сейчас + %»; промо-кампании Этапа 5 —
поверх (динамические скидки), а каталожная «было/стало» — это `compare_at_price`. Глубже —
**ADR-009**.

`CHECK`: `compare_at_price IS NULL OR compare_at_price >= 0`. Соотношение
`compare_at_price > price` **не** форсируем `CHECK`-ом (цена может меняться); «скидочность»
определяется в запросе/коде, бейдж не показывается при `compare_at_price <= price`.

### 3.2. Флаги товара New / Хит / Рекомендуемый

**Проблема.** Обе витрины: бейдж «New» (THE CASE), подборки «Хиты/Новинки/Со скидкой» (Gang Auto).

**Решение — гибрид: явные boolean-флаги + вычисляемые бейджи.**
- **`products.is_featured boolean DEFAULT false`** — «Хит/Рекомендуемый/выведен в подборку».
  Это **редакторское/маркетинговое** решение (мерчандайзер вручную помечает) — его нельзя
  вычислить, поэтому хранимый флаг. Универсально: «featured» = «магазин хочет выделить», без
  привязки к тематике.
- **`products.is_new boolean DEFAULT NULL`** (троичная логика) — **переопределение** «новизны»:
  - `NULL` → новизна **вычисляется**: товар «новый», если `created_at >= now() − N дней`, где
    `N` — настройка магазина (`SHOP_NEW_PRODUCT_DAYS`, дефолт 30). Это покрывает «New» без ручного
    труда и без устаревания (старый товар сам перестаёт быть новинкой).
  - `true`/`false` → **явное** переопределение редактором (форсировать/снять «New» независимо от
    даты).
- **«Со скидкой»** — **не флаг**, а вычисляемый предикат из §3.1 (`compare_at_price > price`).
  Хранить отдельный `is_sale` запрещено — это рассинхрон с ценой.

**Почему гибрид, а не только boolean и не только вычисление.** Чисто вычисляемые бейджи не
покрывают «Хит» (нет формулы «хитовости» без аналитики продаж — отдельный будущий модуль). Чисто
ручные флаги заставляют мерчандайзера вручную снимать «New» у каждого товара по истечении срока —
не масштабируется на 2000+ SKU (Gang Auto). Поэтому: **`is_featured` — ручной**, **`is_new` —
вычисляемый с возможностью ручного override**, **«скидка» — производная цен**. Универсально: набор
бейджей фиксирован и тематико-независим. Обоснование — **ADR-009**.

> Расширяемость: если магазину нужны произвольные бейджи («Топ продаж», «Эко»), их можно завести
> как **EAV-атрибут типа boolean** с `is_filterable=true` (ADR-007) — модель это уже позволяет без
> новой миграции. В `0011` фиксируем только два общеупотребимых флага, остальное — данные.

### 3.3. Бренд / производитель

**Проблема.** Для Gang Auto бренд (Brembo, Bosch, KYB, Mann-Filter, Gates) — **первоклассный
фасет**: фильтр, логотип, страница бренда. Для THE CASE бренда нет вовсе.

**Решение — лёгкая опциональная таблица `brands` + `products.brand_id` (nullable)**, а **не**
EAV-атрибут.

| Критерий | Таблица `brands` (выбрано) | Бренд как EAV-атрибут |
|---|---|---|
| Логотип/описание/SEO бренда | есть поля (`logo_key`, `description`, `seo_*`) | негде хранить структурно |
| Фасетный фильтр по бренду | FK + индекс `products(brand_id)` — прямой и быстрый | через `product_attributes` JOIN — есть, но без метаданных бренда |
| Страница бренда `/brand/{slug}` | `brands.slug` готов | надо синтезировать из значения атрибута |
| Опциональность (THE CASE без брендов) | `brand_id` nullable, таблица пуста — **ноль накладных** | атрибут просто не заводится |
| Целостность (один «Bosch», не «bosch/BOSCH») | FK на `brands.id` | словарь `attribute_values` тоже даёт, но без лого |

**Выбор — таблица `brands`.** Бренд — это **сущность с собственными атрибутами** (лого, описание,
страница), а не просто значение характеристики; для фильтра/витрины это первоклассный фасет. При
этом она **опциональна и пуста** для магазинов без брендов (THE CASE), `products.brand_id` —
`NULL`. Это не нарушает универсальность: таблица существует всегда (одна схема для всех магазинов,
ADR-003), но **наполняется данными** только там, где бренды есть. Обоснование — **ADR-009**.

> Не путать с EAV: общие технические характеристики запчасти (диаметр, тип резьбы и т.п.) —
> по-прежнему EAV-атрибуты (ADR-007). Бренд выделен из EAV **намеренно** из-за лого/страницы/фасета.

### 3.4. Подбор по VIN / совместимость (vehicle fitment) — ТОЛЬКО точка расширения

**Проблема.** Подбор по VIN — ключевая фича Gang Auto: показать запчасти, **совместимые** с
конкретным авто. Это **тематико-специфичная** механика (нужна авто-магазину, бессмысленна для
формы THE CASE).

**Решение на текущем этапе — НЕ реализовывать, а спроектировать конфигурируемую точку
расширения** (модуль `fitment`), чтобы добавление совместимости позже было **данными + опциональным
модулем**, а не рефактором каталога и не хардкодом «авто» в ядре:
- **Отдельный опциональный модуль `fitment`** (флаг в `ADMIK_MODULES`, по умолчанию **выключен**).
  Выключен → каталог работает как сейчас; включён → появляются таблицы совместимости и фасет
  «подбор по авто».
- **Эскиз будущей модели (НЕ в `0011`):** обобщённая, не захардкоженная под «авто»:
  - `fitment_dimensions` — оси совместимости (для авто: марка/модель/год/двигатель; код-ключ),
    как **данные** — другой домен задаст свои оси;
  - `fitment_values` — словарь значений по оси;
  - `product_fitments` — связь товар ↔ набор значений (совместимость);
  - VIN → набор значений осей решается **внешним справочником/сервисом** (VIN-декодер), результат
    маппится на `fitment_*`. VIN-декодер — интеграция с mock-режимом (как СДЭК).
- В `0011` — **ничего по fitment** (только пометка-резерв в комментарии миграции). Реализация —
  отдельный этап после стабилизации каталога. Это сохраняет «обновление не ломает данные»
  (`docs/02`) и не тащит авто-специфику в общий каталог. Обоснование — **ADR-009**.

### 3.5. Валюта и единицы — из настроек магазина

**Проблема.** Обе витрины в **₽**, но платформа универсальна — валюту нельзя хардкодить.

**Решение (подтверждение `docs/05` §6, без новой схемы каталога):**
- **Валюта** — настройка магазина `SHOP_CURRENCY` (дефолт `'RUB'`), читается из `lib/config/env.ts`
  / таблицы настроек (Этап 5). Форматирование цен (символ ₽, разряды) — по локали магазина в
  слое представления / Storefront API, **не** в данных каталога. Цены в БД — `NUMERIC(14,2)` без
  символа валюты.
- **Единицы измерения характеристик** — поле `attributes.unit` (данные, ADR-007), уже есть.
- В `0011` поля валюты **не добавляем** — это настройка магазина (одна валюта на магазин,
  ADR-003: один магазин = одна БД), а не атрибут товара. Если позже понадобится мультивалютность —
  отдельный модуль; сейчас избыточно.

## 4. Спецификация миграции `0011_catalog_pricing_flags_brands.sql`

**Принципы (как `0005–0010`, `docs/05` §2):** идемпотентность (`ADD COLUMN IF NOT EXISTS`,
`CREATE TABLE/INDEX IF NOT EXISTS`, `ON CONFLICT DO NOTHING`); `NUMERIC(14,2)` для денег; `citext`
для slug; `TIMESTAMPTZ`; `app`-гранты на новую таблицу; запись в `schema_migrations`. Все новые
поля **nullable/с дефолтом** → не ломают существующие данные и магазины без этих фич.

### 4.1. DDL-эскиз

```sql
-- =============================================================================
-- 0011_catalog_pricing_flags_brands.sql
-- Этап 2.x — расширение каталога по референсным витринам (docs/06 §3, ADR-009):
--   * compare_at_price (цена «было») в products и product_variants — скидка вычисляется;
--   * флаги товара is_featured (ручной) и is_new (NULL=вычисляемо по дате, override);
--   * таблица brands (опциональна, пуста без брендов) + products.brand_id (nullable).
-- НЕ входит: vehicle fitment (VIN) — отдельный опциональный модуль `fitment` (docs/06 §3.4),
--   точка расширения, в этой миграции НЕ создаётся.
-- Идемпотентно: ADD COLUMN/CREATE ... IF NOT EXISTS, ON CONFLICT DO NOTHING.
-- =============================================================================

-- --- 3.1. Акционная цена «было» (скидка вычисляется в коде/запросе) ----------
ALTER TABLE products
  ADD COLUMN IF NOT EXISTS compare_at_price numeric(14,2)
    CHECK (compare_at_price IS NULL OR compare_at_price >= 0);

ALTER TABLE product_variants
  ADD COLUMN IF NOT EXISTS compare_at_price numeric(14,2)
    CHECK (compare_at_price IS NULL OR compare_at_price >= 0);

-- --- 3.2. Флаги товара -------------------------------------------------------
-- is_featured: ручной маркетинговый флаг «Хит/Рекомендуемый» (вычислить нельзя).
ALTER TABLE products
  ADD COLUMN IF NOT EXISTS is_featured boolean NOT NULL DEFAULT false;
-- is_new: NULL → новизна вычисляется по created_at и настройке SHOP_NEW_PRODUCT_DAYS;
--         true/false → явное переопределение редактором.
ALTER TABLE products
  ADD COLUMN IF NOT EXISTS is_new boolean;          -- nullable намеренно (троичная логика)

-- Частичные индексы под подборки витрины (фасеты «Хиты»/«Новинки»):
CREATE INDEX IF NOT EXISTS products_featured_idx ON products (is_featured) WHERE is_featured;
CREATE INDEX IF NOT EXISTS products_is_new_idx   ON products (is_new)       WHERE is_new;
-- «Со скидкой» — вычисляемый предикат (compare_at_price > base_price), отдельный флаг НЕ хранится;
-- при необходимости ускорить: CREATE INDEX ... WHERE compare_at_price IS NOT NULL.
CREATE INDEX IF NOT EXISTS products_has_compare_idx
  ON products (id) WHERE compare_at_price IS NOT NULL;

-- --- 3.3. Бренды (опционально; пусто для магазинов без брендов) --------------
CREATE TABLE IF NOT EXISTS brands (
  id              uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  slug            citext      NOT NULL,                  -- ЧПУ бренда, уникален
  name            text        NOT NULL,
  description     text        NOT NULL DEFAULT '',
  logo_key        text,                                  -- ключ объекта в S3/MinIO (как product_media.storage_key)
  is_active       boolean     NOT NULL DEFAULT true,
  sort            integer     NOT NULL DEFAULT 0,
  seo_title       text,
  seo_description text,
  created_at      timestamptz NOT NULL DEFAULT now(),
  updated_at      timestamptz NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX IF NOT EXISTS brands_slug_uniq  ON brands (slug);
CREATE INDEX        IF NOT EXISTS brands_active_idx ON brands (is_active);

-- products.brand_id — nullable: товар может быть без бренда (THE CASE).
-- ON DELETE SET NULL: удаление бренда не удаляет товары, лишь снимает привязку.
ALTER TABLE products
  ADD COLUMN IF NOT EXISTS brand_id uuid REFERENCES brands(id) ON DELETE SET NULL;
CREATE INDEX IF NOT EXISTS products_brand_idx ON products (brand_id);  -- фасетный фильтр по бренду

-- --- Гранты и регистрация миграции ------------------------------------------
GRANT SELECT, INSERT, UPDATE, DELETE ON brands TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0011', 'catalog_pricing_flags_brands') ON CONFLICT DO NOTHING;
```

### 4.2. Сводка изменений схемы

| Изменение | Таблица | Тип | Семантика |
|---|---|---|---|
| `compare_at_price` | `products` | `numeric(14,2)` NULL | цена «было» (скидка вычисляется) |
| `compare_at_price` | `product_variants` | `numeric(14,2)` NULL | «было» на уровне варианта (override товара) |
| `is_featured` | `products` | `boolean DEFAULT false` | ручной «Хит/Рекомендуемый» |
| `is_new` | `products` | `boolean` NULL | NULL=вычисляемо по дате; true/false=override |
| `brand_id` | `products` | `uuid` NULL → `brands` | бренд товара (опционален); ON DELETE SET NULL |
| таблица `brands` | — | новая | id/slug/name/logo_key/seo/… — фасет + страница бренда |

### 4.3. Что дополнить в `lib/catalog/*` (фактическая структура слоя)

> Фактический слой каталога — **единые файлы** `lib/catalog/{types,schemas,repository,actions,cache}.ts`
> (не пер-сущностное разбиение из эскиза `docs/05` §4). Правки минимальны и аддитивны:

- **`lib/catalog/types.ts`**
  - `Product`: добавить `compareAtPrice: number | null`, `isFeatured: boolean`,
    `isNew: boolean | null`, `brandId: string | null` (+ опц. развёрнутый `brand?: BrandRef`).
  - `ProductVariant`: добавить `compareAtPrice: number | null`.
  - `ProductListRow` / `ProductDetail`: добавить `compareAtPrice`, `isFeatured`, `effectiveIsNew`
    (вычисленный), `discountPct` (вычисленный), `brand?: { id; name; slug; logoKey }`.
  - Новый тип `Brand` (id, slug, name, description, **logoKey**, isActive, sort, seo*) и `BrandRef`
    (id, slug, name, **logoKey**). Доменная модель хранит ТОЛЬКО `logoKey` (ключ объекта в
    хранилище, как `product_media.storage_key`); публичный URL логотипа НЕ хранится в домене, а
    собирается на границе представления резолвером `storage.url(logoKey)` — зеркально og:image
    (`buildOgImageUrl` в `lib/seo/meta`). В админ-страницах `brands/*` URL резолвится на сервере
    перед передачей в `BrandList`/`BrandForm`. В Storefront DTO (`toBrandDto`/`toFullBrandDto`/
    `toProductListItemDto`/`toProductDetailDto`) `logoUrl` вычисляется инъецированным
    `PublicUrlResolver` (`storage.url`); без резолвера — `null` (обратная совместимость).
    ПРИЧИНА (регресс волны 4): ранее мапперы читали несуществующую колонку `logo_url`, из-за чего
    `logoUrl` был всегда `null` и в админке, и в API, хотя `logo_key` сохранялся.
- **`lib/catalog/schemas.ts`**
  - `ProductCreateSchema`/`ProductUpdateSchema`: опц. `compareAtPrice` (money|null), `isFeatured`
    (boolean, дефолт false), `isNew` (boolean|null), `brandId` (uuid|null).
  - `VariantCreateSchema`/`VariantUpdateSchema`: опц. `compareAtPrice` (money|null).
  - Новые: `BrandCreateSchema`, `BrandUpdateSchema`, `BrandIdSchema` (slug/name/description/logo/seo).
- **`lib/catalog/repository.ts`**
  - `SELECT` товаров/вариантов: добавить новые колонки; LEFT JOIN `brands` для развёрнутого бренда.
  - Вычисляемые поля в проекции: `discount_pct`, `effective_is_new`
    (`COALESCE(is_new, created_at >= now() - (SHOP_NEW_PRODUCT_DAYS||' days')::interval)`),
    предикат `on_sale` (`compare_at_price IS NOT NULL AND compare_at_price > effective_price`).
  - Фильтры списка: `brandId`, `isFeatured`, `isNew`, `onSale` (для админки и Storefront API).
  - Пагинация `listProducts`: `page`/`pageSize`; опц. `offset` (приоритет над `page`, clamp `>= 0`,
    без округления до границы страницы) — нужен публичному API под свободный `offset` (не кратный
    `pageSize`), чтобы товары не пропадали/не дублировались между «страницами». Не задан → фолбэк
    на `(page-1)*pageSize` (контракт админки не меняется).
  - Новые запросы брендов: `listBrands`, `getBrandBySlug`, `getBrandById`.
- **`lib/catalog/actions.ts`**
  - `createProduct`/`updateProduct`: писать `compare_at_price`, `is_featured`, `is_new`, `brand_id`.
  - `createVariant`/`updateVariant`: писать `compare_at_price`.
  - Новые `defineAction`: `createBrand`/`updateBrand`/`deleteBrand` (право `catalog.write`, аудит
    `catalog.brand.*`); загрузка лого бренда — через тот же `lib/storage`/паттерн медиа.
- **`lib/catalog/cache.ts`** — без структурных изменений (флаги/цена/бренд — в колонках, не в EAV);
  при желании добавить `brand`/`compareAtPrice` в `attributes_cache`-проекцию для быстрого чтения
  карточки (необязательно).
- **Настройка:** добавить `SHOP_CURRENCY` (дефолт `'RUB'`) и `SHOP_NEW_PRODUCT_DAYS` (дефолт `30`)
  в `lib/config/env.ts` / `.env.example` (используются при вычислении «новизны» и форматировании цен).
- **UI (`app/admin/catalog/*`)**: вкладка «Основное» товара — поля «Цена до скидки»
  (`compare_at_price`), чекбоксы «Хит» (`is_featured`) и «Новинка» (`is_new`, три состояния:
  авто/да/нет), выбор бренда; новый раздел `app/admin/catalog/brands/*` (CRUD брендов, лого).

### 4.4. Тесты (ADR-004 — раньше реализации)

- Миграция `0011` идемпотентна (двойной накат не меняет данные; колонки/таблица/индексы/грант на
  месте); `CHECK` на `compare_at_price >= 0` срабатывает.
- `discount_pct` и предикат `on_sale` корректны (включая `compare_at_price <= price` → не на
  распродаже); `effective_is_new` учитывает `is_new`-override и порог `SHOP_NEW_PRODUCT_DAYS`.
- Бренд: CRUD через `defineAction` (право/аудит), `ON DELETE SET NULL` обнуляет `brand_id` без
  удаления товара; фильтр по `brand_id` работает; магазин без брендов (пустая `brands`,
  `brand_id IS NULL`) функционирует штатно.

## 6. Резюме (для журнала)

1. **Admik = headless commerce backend** для внешних SPA-витрин (Netlify) + публичный **Storefront
   API** (`/api/storefront/v1`): read-каталог, заказ, СДЭК-расчёт/ПВЗ; auth по API-ключу/домену,
   CORS, версионирование, rate-limit. → **ADR-008**. Каталожные read-эндпоинты — уже в 2.x; заказы —
   Этап 3, СДЭК — Этап 4.
   **SECURITY (волна 4, баг A):** ключ ведра rate-limit (`lib/storefront/response.ts → rateKey`)
   берётся по API-ключу **только когда ключ ВАЛИДЕН** (`auth.via==='key'`, совпал с
   `STOREFRONT_API_KEYS`); иначе (origin/mock/невалидный ключ) — по **валидированному IP**
   (`normalizeClientIp`, как auth/session/audit). Это закрывает обход лимита ротацией мусорного
   `X-Api-Key` (раньше любой непустой ключ создавал своё ведро `count=0` на каждый запрос).
2. **Две разные тематики на одной платформе** (медформа THE CASE vs автозапчасти Gang Auto)
   подтверждают универсальность (ADR-003): различия — в **данных/настройках**, не в коде. Общий
   знаменатель: **₽ + СДЭК + корзина/чекаут + промо + фасеты**.
3. **Gap-анализ каталога → миграция `0011`** (минимально, универсально, опционально):
   - **`compare_at_price`** (в `products` и `product_variants`) — каталожная цена «было»; скидка %
     и «со скидкой» **вычисляются**, не хранятся. Выбор единого поля вместо `old_price`/`sale_price`
     обоснован (ADR-009).
   - **Флаги: `is_featured`** (ручной «Хит/Рекомендуемый») + **`is_new`** (NULL=вычисляемо по
     `created_at` и `SHOP_NEW_PRODUCT_DAYS`, true/false=override); «со скидкой» — производная цен.
   - **Бренды:** лёгкая опциональная таблица **`brands`** + `products.brand_id` (nullable, ON DELETE
     SET NULL) — бренд как фасет/лого/страница, пуст для магазинов без брендов. Выбран против EAV
     (ADR-009).
   - **VIN/совместимость:** только **точка расширения** — будущий опциональный модуль `fitment`
     (обобщённый, не «авто»-хардкод), в `0011` не реализуется.
   - **Валюта/единицы:** из настроек магазина (`SHOP_CURRENCY='RUB'` по умолчанию), не хардкод; в
     `0011` поля валюты не добавляются.
4. **Правки `lib/catalog/*`** (фактические единые файлы `types/schemas/repository/actions/cache`) —
   аддитивны (§4.3); новый UI брендов и полей цены/флагов.
5. **Роадмап (§5):** новый под-этап 2.x (миграция `0011` + Storefront API каталога), усиление
   Этапа 3 (промокоды, порог бесплатной доставки, возвраты), Этапа 4 (расчёт/ПВЗ/трекинг СДЭК —
   обязательны для обеих витрин), конкретизация Этапа 5 (промокоды, «N по M»), будущий модуль
   `fitment`.
