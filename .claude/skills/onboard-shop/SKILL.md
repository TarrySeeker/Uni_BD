# Онбординг магазина: прикрутить Admik к готовому макету витрины

Оркестратор полного цикла из `docs/20-онбординг-витрины-playbook.md`. Ты —
оркестратор мультиагентной системы (CLAUDE.md): декомпозируешь и поручаешь работу
профильным агентам, сам держишь план и проверки. Перенос на магазин = **конфигурация
+ данные**, НЕ правка кода платформы (ADR-003).

Перед запуском прочитай: `START-HERE.md`, `CLAUDE.md`, `docs/20`, `docs/21`.

## Инварианты (нарушать нельзя)
- **TDD:** сначала тест (падает) → код до зелёного.
- **Две проверки** перед коммитом: код-гейт `/gate` И живой прогон в браузере.
- **Мультитенантность:** ноль хардкода ниши в коде; всё — в `.env` и `shop_settings`.
- **Миграции иммутабельны/аддитивны** (`check-migrations.sh`); существующие не трогать.
- **Секреты только из env**; боевые ключи СДЭК/Т-Банк — на сервере, не в git.
- **Документируй каждую правку** (`documentation-manager` / `history-manager`).

## Шаги (по фазам docs/20)

1. **Фаза 0 — пред-полёт.** Найди макет витрины (ожидается `./storefront`; иначе
   уточни путь и поправь build-context в `docker-compose.yml`). `pnpm install`,
   `cp .env.example .env`. Проведи короткое интервью: ниша, модули
   (`catalog,orders,cdek,cms,payments`), домены, реквизиты, валюта, mock/боевые интеграции.

2. **Фаза 1 — анализ макета → профиль магазина.** Запусти параллельных read-only
   `Explore`-агентов по `./storefront`: каталог (категории/варианты/атрибуты/
   вес-габариты), навигация и страницы (→ CMS), брендинг/контакты/реквизиты,
   главная (hero/about/преимущества/доставка), способ корзины/оформления/оплаты,
   стек витрины. Сохрани профиль; зафиксируй резюме в `docs/00`.

3. **Фаза 2 — БД.** Поручи `devops-engineer`/`backend-engineer`: `make up` → `make init`
   (`init-shop.sh`: расширения+роли → миграции → seed прав/ролей/владельца → опц.
   demo-каталог) → `make smoke`. Проверь `/api/health=ok` и вход в `/admin`.

4. **Фаза 3 — ретематизация.** `.env` под магазин (SHOP_NAME, домены, ORDER_PREFIX,
   ADMIK_MODULES, STOREFRONT_*). Каталог под нишу: через админку (боевое наполнение)
   и/или адаптация `db/seed/demo-catalog.sql` (нейтральный идемпотентный пример) —
   с обновлением seed-теста (`backend-engineer`, TDD). Брендинг/контент/навигация/
   главная/CMS — через `/admin/settings` и `/admin/cms` (доезжают через `GET /settings`,
   `GET /pages`). Без хардкода в коде.

5. **Фаза 4 — сращивание.** Поручи `frontend-engineer`: перевести макет в потребителя
   Storefront API по маппингу `docs/21 §7` (products/categories/brands, cart/quote,
   orders+accessToken, orders/:number, cdek cities/pvz/calculate, payments/tbank/init,
   pages, settings, promotions, leads, newsletter). Два адреса API (SSR `http://app:3000`
   + браузерный `NEXT_PUBLIC_ADMIK_API_URL`), авторизация ключ/Origin. Anti-tamper:
   не слать цены/итоги/вес с витрины. Не хватает эндпоинта → проектирует
   `solution-architect`, реализует backend по TDD аддитивно (НЕ городить логику в витрине).

6. **Фаза 5 — две проверки.** `/gate` (typecheck/lint/vitest/check-migrations/security)
   → `qa-engineer`/`security-engineer`. Живой прогон браузером (`/deploy-stand` или
   `/verify` + playwright, путь покупателя + админка, mock). `/integration-check`
   (СДЭК+Т-Банк mock e2e). `make smoke`.

7. **Фаза 6 — документирование + коммит.** Обнови `docs/00`, `docs/10`, README инстанса;
   зафиксируй профиль магазина и изменённые конфиг/сид-файлы. Коммить ТОЛЬКО после
   обеих проверок. Боевой деплой — по `docs/09` (`SEED_DEMO_CATALOG=false`, реальные ключи).

## Definition of Done (см. START-HERE §2)
БД+миграции+seed готовы · каталог/атрибуты/брендинг ретематизированы конфигурацией
и данными (ноль правок бизнес-логики) · витрина подключена к Storefront API · админка
работает и заведённое доезжает до витрины · `/gate` зелёный + живой прогон пройден ·
всё задокументировано.

## Итог скилла
Путь макета, профиль магазина, изменённые конфиг/сид-файлы, статус сращивания по
разделам, результат `/gate` и живого прогона, ссылки на обновлённые доки.
