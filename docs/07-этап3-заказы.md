## 0. Краткое следствие (TL;DR)

- **Единая модель заказа** (ADR-002, урок 2x2/carre — никаких «трёх таблиц заявок»): `orders` +
  `order_items` + `order_status_history`. Позиции заказа — **денормализованный снимок** товара на
  момент покупки (name/sku/price/attributes копируются), чтобы изменение каталога не переписывало
  историю. → **ADR-010**.
- **Корзина живёт на витрине (клиентская), бэкенд её не хранит** для гостя. Платформа даёт два
  серверных эндпоинта: `POST /cart/quote` (расчёт итога: цены/остатки/промокод/доставка — заглушка
  под СДЭК) и `POST /orders` (создание заказа). **Сервер — единственный источник истины по ценам и
  итогу** (anti-tamper): витрине доверяем только `variant_id`+`qty`+выбор доставки/промокода, всё
  считаем заново из каталога. → **ADR-010**.
- **Деньги** — `NUMERIC(14,2)` в БД, строки в домене (как в каталоге, `lib/catalog/pricing.ts`),
  без потери точности. **PK** — `uuid`, **время** — `timestamptz`.
- **Резерв остатков** через уже заложенное в Этапе 2 поле `inventory.reserved`
  (`CHECK reserved <= quantity`): создание заказа → атомарный резерв; отгрузка → списание
  (`quantity -= n`, `reserved -= n`); отмена → возврат резерва. Атомарность — `UPDATE ... WHERE`
  с условием и проверкой числа затронутых строк, всё в одной транзакции.
- **Промокоды** (`promo_codes` + `promo_redemptions`): процент / фикс / бесплатная доставка;
  лимиты (всего/на пользователя), срок, мин.сумма; накладываются **поверх итога** (после каталожной
  скидки `compare_at_price`, ADR-009). Идемпотентность применения — через `promo_redemptions`.
- **Out of scope Этапа 3:** реальная онлайн-оплата (провайдеры — будущий под-этап; сейчас только
  фиксация способа + статуса оплаты, mock) и сама СДЭК-интеграция (Этап 4 — здесь только **поля**
  под доставку: тип/адрес/ПВЗ/стоимость/трек/статус). Акции «N по M»/«3 по 2» — **модель
  закладывается**, реализация может быть отложена в Этап 5.2.
- **Новые миграции:** `0012` (orders+items+history) · `0013` (customers) · `0014` (promo_codes) ·
  `0015` (promo_redemptions) · `0016` (нумератор заказов). Все — идемпотентные, уважают паттерн
  `0005–0011`. **Новых npm-зависимостей не требуется** (§9).
- **Права** `orders.read` / `orders.write` — **уже заведены** в `lib/auth/permissions.ts` и
  сидированы (admin/manager). Новых прав ядра не добавляем; промокоды — под `orders.write`.

## 1. Цель, охват и Definition of Done

### 1.1. Цель этапа

Дать платформе модуль **`orders`**: единую модель заказа, серверный расчёт корзины и оформление
(чекаут) через **публичный Storefront API** (витрины THE CASE / Gang Auto — внешние SPA, ADR-008),
управление заказами в админке (статус-машина, история, возвраты), промокоды и порог бесплатной
доставки. Модуль уважает флаг `orders` в `ADMIK_MODULES` (`lib/config/modules.ts`).

### 1.2. In scope

1. Схема БД заказов: `orders`, `order_items`, `order_status_history`, `customers`, `promo_codes`,
   `promo_redemptions`, нумератор заказов (§2).
2. Статус-машины: **статус заказа**, **статус оплаты**, **статус доставки** (последний
   синхронизируется со СДЭК в Этапе 4) (§2.7).
3. Промокоды и скидки: модель применения, приоритет с каталожной скидкой, порог бесплатной
   доставки, задел под «N по M» (§3).
4. Storefront API (расширение `/api/storefront/v1`): `POST /cart/quote`, `POST /orders`,
   `GET /orders/:number` (§4.2). **Цены/итог считает сервер** (anti-tamper, ADR-010).
5. Server Actions админки (через `defineAction`, право `orders.read`/`orders.write`, аудит):
   список/поиск/фильтр, карточка, смена статуса, отмена/возврат, ручное создание, экспорт;
   CRUD промокодов (§4.1).
6. UI админки `/admin/orders/*`: список с фильтрами, карточка с переходами статусов и историей,
   раздел промокодов (§5).
7. Резерв/списание остатков на `inventory.reserved`/`inventory.quantity` (атомарно, §6).
8. **Точки стыковки** со СДЭК (Этап 4) и онлайн-оплатой (будущее) — только поля/структура (§7).

### 1.3. Out of scope (явно)

| Не входит в Этап 3 | Куда отнесено | Что закладываем сейчас |
|---|---|---|
| Реальная онлайн-оплата (СДЭК PAY, карты, СБП, эквайринг) | Будущий под-этап (после Этапа 3) | Поля `payment_method`, `payment_status`, `paid_at`, `payment_ref`; mock-«оплата» меняет статус вручную/заглушкой |
| СДЭК-интеграция (расчёт/ПВЗ/создание отправления/трек/webhook) | **Этап 4** (`cdek`) | Поля доставки (тип/адрес/город/ПВЗ-код/стоимость/трек/статус доставки) + эндпоинт-заглушка расчёта в `quote` |
| Покупательские аккаунты с паролем, избранное (wishlist) | Будущий под-модуль; ЛК — на стороне витрины | `customers` как **задел** (гостевой чекаут по контактам + опц. связь заказ↔customer) |
| Динамические промо-кампании, BOGO-движок «3 по 2» (исполнение) | **Этап 5.2** (CMS+промо) | **Модель** типа акции `bogo` в `promo_codes.kind` + поля условий; исполнение может быть отложено |
| Возврат денег через платёжный шлюз | После онлайн-оплаты | Статус заказа `refunded` + `payment_status='refunded'` фиксируются вручную (mock) |

### 1.4. Definition of Done (этап)

- Миграции `0012–0016` накатываются одной командой на пустую БД и **идемпотентны** (повторный накат
  не меняет данные); гранты `admik_app` выданы; записи в `schema_migrations`.
- Создание заказа через `POST /api/storefront/v1/orders` **атомарно**: ре-валидация цен/остатков на
  сервере, снимок позиций, резерв остатка, применение промокода, генерация человекочитаемого
  **уникального** номера; повторная отправка с тем же idempotency-ключом **не дублирует** заказ.
- `POST /cart/quote` возвращает серверный итог (товары/скидка/доставка/промокод/итог), не доверяя
  ценам витрины; невалидный промокод/недостаток остатка отражён в ответе **без** создания заказа.
- Админка: менеджер (право `orders.write`) меняет статус по допустимым переходам, каждый переход
  пишет `order_status_history` и `audit_log`; недопустимый переход отклоняется на сервере.
- Резерв/списание: при создании — `reserved += qty` (без ухода `reserved > quantity`); при отгрузке —
  `quantity -= qty, reserved -= qty`; при отмене — `reserved -= qty`. Гонки исключены (§6).
- Модуль уважает флаг `orders`: выключен → UI-раздел скрыт, Storefront-эндпоинты заказов отдают 404
  `module_disabled`, Server Actions недоступны из меню.
- Никаких хардкодов магазина: валюта (`SHOP_CURRENCY`), порог бесплатной доставки
  (`SHOP_FREE_DELIVERY_THRESHOLD`), префикс номера заказа (`SHOP_ORDER_PREFIX`) — из настроек/env.
- **«Сначала тесты, потом код»** (ADR-004): для каждой задачи сперва падающие тесты, затем код до
  зелёного. typecheck/lint зелёные, `next build` успешен.

## 2. Схема БД (миграции `0012–0016`)

**Конвенции (как `0005–0011`, `docs/05` §2):** идемпотентность (`CREATE TABLE/INDEX IF NOT EXISTS`,
`ADD COLUMN IF NOT EXISTS`, `ON CONFLICT DO NOTHING`); **`uuid`** PK (`gen_random_uuid()`);
**`timestamptz`** для времени; деньги — **`numeric(14,2)`** с `CHECK >= 0`; `citext` для
кодов/slug; гранты `admik_app`; запись в `schema_migrations`. FK/`ON DELETE` — сводно в §2.8.

### 2.1. `orders` — заголовок заказа (миграция `0012`)

```sql
-- =============================================================================
-- 0012_orders.sql  (Этап 3 — модуль orders)
-- Единая модель заказа (ADR-002, без «трёх таблиц заявок»): orders + order_items
--   + order_status_history. Позиции — снимок каталога (ADR-010). Деньги numeric(14,2).
-- Поля доставки/оплаты заложены под Этап 4 (СДЭК) и будущую онлайн-оплату — без интеграций.
-- =============================================================================

CREATE TABLE IF NOT EXISTS orders (
  id                uuid          PRIMARY KEY DEFAULT gen_random_uuid(),

  -- Человекочитаемый уникальный номер (генерируется из нумератора, §2.6): напр. 'GA-2026-000123'.
  number            text          NOT NULL,

  -- Статус-машина заказа (§2.7). CHECK перечисляет допустимые значения.
  status            text          NOT NULL DEFAULT 'new'
                    CHECK (status IN ('new','awaiting_payment','paid','packed',
                                      'shipped','delivered','completed','cancelled','refunded')),

  -- ---- Суммы (все numeric(14,2), считаются СЕРВЕРОМ при создании, ADR-010) ----
  items_total       numeric(14,2) NOT NULL CHECK (items_total       >= 0),  -- сумма позиций (с каталожной скидкой compare_at_price)
  discount_total    numeric(14,2) NOT NULL DEFAULT 0 CHECK (discount_total >= 0),  -- скидка промокода (фикс/процент), не доставка
  delivery_total    numeric(14,2) NOT NULL DEFAULT 0 CHECK (delivery_total >= 0),  -- стоимость доставки (0 при бесплатной/самовывозе)
  grand_total       numeric(14,2) NOT NULL CHECK (grand_total       >= 0),  -- итог к оплате = items_total - discount_total + delivery_total
  currency          text          NOT NULL DEFAULT 'RUB',                    -- из SHOP_CURRENCY (ADR-009 §5), снимок на момент заказа

  -- ---- Оплата (онлайн-провайдеры — будущее; сейчас фиксация способа+статуса, mock) ----
  payment_method    text          NOT NULL DEFAULT 'unset'
                    CHECK (payment_method IN ('unset','cod','card','sbp','cdek_pay','invoice')),
  payment_status    text          NOT NULL DEFAULT 'pending'
                    CHECK (payment_status IN ('pending','authorized','paid','failed','refunded')),
  paid_at           timestamptz,                              -- момент успешной оплаты (mock/реальной)
  payment_ref       text,                                     -- внешний id транзакции провайдера (будущее)

  -- ---- Доставка (поля под СДЭК Этапа 4; сама интеграция — Этап 4, §7) ----
  delivery_type     text          NOT NULL DEFAULT 'courier'
                    CHECK (delivery_type IN ('courier','pvz','pickup')),  -- курьер / ПВЗ СДЭК / самовывоз
  delivery_status   text          NOT NULL DEFAULT 'pending'
                    CHECK (delivery_status IN ('pending','registered','in_transit','delivered','returned','cancelled')),
  delivery_city     text,                                     -- город доставки (для расчёта/тарифа)
  delivery_address  text,                                     -- адрес (для курьера)
  delivery_pvz_code text,                                     -- код ПВЗ СДЭК (для delivery_type='pvz')
  delivery_cost     numeric(14,2) CHECK (delivery_cost IS NULL OR delivery_cost >= 0),  -- расчётная стоимость (СДЭК/заглушка)
  cdek_uuid         text,                                     -- id заказа в СДЭК (Этап 4)
  cdek_track        text,                                     -- трек-номер (Этап 4)

  -- ---- Промокод (денормализованный код + ссылка) ----
  promo_code_id     uuid          REFERENCES promo_codes(id) ON DELETE SET NULL,
  promo_code        text,                                     -- снимок кода (история не зависит от удаления промокода)

  -- ---- Покупатель (гостевой чекаут — контакты в заказе; опц. связь с customers) ----
  customer_id       uuid          REFERENCES customers(id) ON DELETE SET NULL,  -- задел под ЛК (§2.5)
  customer_name     text          NOT NULL,
  customer_email    citext        NOT NULL,
  customer_phone    text          NOT NULL,

  comment           text          NOT NULL DEFAULT '',        -- комментарий покупателя

  -- ---- Идемпотентность создания (anti-double-submit, §4.2) ----
  idempotency_key   text,                                     -- ключ от витрины; UNIQUE ниже (partial)

  -- ---- Метки времени и происхождение ----
  source            text          NOT NULL DEFAULT 'storefront'
                    CHECK (source IN ('storefront','admin')),  -- откуда создан (витрина/ручной в админке)
  ip                text,                                      -- ip оформления (как audit_log)
  created_at        timestamptz   NOT NULL DEFAULT now(),
  updated_at        timestamptz   NOT NULL DEFAULT now()
);

-- Уникальность человекочитаемого номера:
CREATE UNIQUE INDEX IF NOT EXISTS orders_number_uniq ON orders (number);
-- Идемпотентность создания заказа витриной (partial — NULL не конфликтует):
CREATE UNIQUE INDEX IF NOT EXISTS orders_idempotency_uniq
  ON orders (idempotency_key) WHERE idempotency_key IS NOT NULL;
-- Фильтры/сортировки админки:
CREATE INDEX IF NOT EXISTS orders_status_idx     ON orders (status);
CREATE INDEX IF NOT EXISTS orders_created_idx    ON orders (created_at DESC);
CREATE INDEX IF NOT EXISTS orders_email_idx      ON orders (customer_email);
CREATE INDEX IF NOT EXISTS orders_promo_idx      ON orders (promo_code_id);
CREATE INDEX IF NOT EXISTS orders_payment_idx    ON orders (payment_status);

GRANT SELECT, INSERT, UPDATE, DELETE ON orders TO admik_app;
```

> **Почему денормализованный `promo_code`/`customer_name` рядом со ссылками:** ссылка
> (`promo_code_id`/`customer_id`) даёт навигацию и `ON DELETE SET NULL` (удаление промокода/покупателя
> не рушит заказ), а денормализованный снимок (`promo_code`/`customer_*`) гарантирует, что история
> заказа читаема, даже если связанная сущность удалена/изменена (ADR-010, anti-history-rewrite).

### 2.2. `order_items` — позиции заказа (снимок каталога, миграция `0012`)

```sql
CREATE TABLE IF NOT EXISTS order_items (
  id            uuid          PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id      uuid          NOT NULL REFERENCES orders(id) ON DELETE CASCADE,  -- позиция без заказа не нужна

  -- Ссылки на каталог (для аналитики/повторного заказа); НЕ источник цены/имени в истории:
  product_id    uuid          REFERENCES products(id)         ON DELETE SET NULL,  -- товар могут удалить — история остаётся
  variant_id    uuid          REFERENCES product_variants(id) ON DELETE SET NULL,

  -- ---- СНИМОК на момент заказа (ADR-010): изменение каталога не меняет историю ----
  name_snapshot       text          NOT NULL,                  -- название товара/варианта на момент покупки
  sku_snapshot        text          NOT NULL,                  -- артикул
  attributes_snapshot jsonb         NOT NULL DEFAULT '{}'::jsonb,  -- выбранные атрибуты (размер/цвет) на момент покупки
  unit_price          numeric(14,2) NOT NULL CHECK (unit_price          >= 0),  -- эффективная цена за единицу (что платят)
  compare_at_snapshot numeric(14,2) CHECK (compare_at_snapshot IS NULL OR compare_at_snapshot >= 0),  -- «было» на момент покупки (для чека/возврата)

  quantity      integer       NOT NULL CHECK (quantity > 0),
  line_total    numeric(14,2) NOT NULL CHECK (line_total >= 0),  -- = unit_price * quantity (считает сервер)

  created_at    timestamptz   NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS order_items_order_idx   ON order_items (order_id);
CREATE INDEX IF NOT EXISTS order_items_product_idx ON order_items (product_id);
CREATE INDEX IF NOT EXISTS order_items_variant_idx ON order_items (variant_id);

GRANT SELECT, INSERT, UPDATE, DELETE ON order_items TO admik_app;
```

> **Снимок — ключевое решение (ADR-010).** Цена/название/атрибуты копируются в `order_items` при
> создании. Дальнейшее изменение каталога (цена выросла, товар переименован, вариант удалён) **не
> переписывает** прошлый заказ. Ссылки `product_id`/`variant_id` остаются для навигации/аналитики, но
> при удалении товара → `SET NULL` (а не CASCADE) — история позиции сохраняется по снимку.

### 2.3. `order_status_history` — история смен статуса (миграция `0012`)

```sql
CREATE TABLE IF NOT EXISTS order_status_history (
  id          uuid        PRIMARY KEY DEFAULT gen_random_uuid(),
  order_id    uuid        NOT NULL REFERENCES orders(id) ON DELETE CASCADE,
  kind        text        NOT NULL DEFAULT 'order'
              CHECK (kind IN ('order','payment','delivery')),  -- какая из трёх статус-машин сменилась
  from_status text,                                  -- NULL для первой записи (создание)
  to_status   text        NOT NULL,
  actor_user_id uuid      REFERENCES users(id) ON DELETE SET NULL,  -- кто сменил (NULL = система/витрина)
  comment     text        NOT NULL DEFAULT '',
  created_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS order_status_history_order_idx ON order_status_history (order_id, created_at);

GRANT SELECT, INSERT, UPDATE, DELETE ON order_status_history TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0012','orders') ON CONFLICT DO NOTHING;
```

> История — **append-разумная** (записываем каждый переход). Дублирует с `audit_log` (общесистемный
> аудит через `defineAction`) намеренно: `order_status_history` — **доменная** лента в карточке
> заказа и основа трекинга для ЛК витрины; `audit_log` — сквозной журнал безопасности. Оба пишутся в
> одном action.

### 2.4. `customers` — покупатели (опционально, миграция `0013`)

**Обоснование (гость vs аккаунт).** Витрины — **внешние SPA** (ADR-008); покупательский ЛК и
авторизация покупателя реализуются **на стороне витрины** (`docs/06` §1.2). Поэтому на Этапе 3:

- **Базовый сценарий — гостевой чекаут:** контакты покупателя хранятся **прямо в заказе**
  (`orders.customer_name/email/phone`). Заказ самодостаточен, аккаунт не обязателен.
- **Таблица `customers` — задел (опциональная связь):** агрегирует заказы по email/phone для будущего
  «Личный кабинет/отслеживание» (THE CASE) и аналитики «постоянный покупатель». **Без пароля** на
  этом этапе (аутентификация покупателя — будущий под-модуль/витрина). `orders.customer_id` —

### 2.8. Статус-машины

Три **независимые, но связанные** статус-машины (поля `orders.status` / `payment_status` /
`delivery_status`). Все переходы валидируются на сервере (whitelist), пишут `order_status_history`
и `audit_log`.

**(A) Статус заказа (`orders.status`).**

```
                         ┌───────────────► cancelled  (из любого до shipped)
                         │
new ──► awaiting_payment ──► paid ──► packed ──► shipped ──► delivered ──► completed
  │            │                                                              │
  └────────────┴──► cancelled                                                 └──► refunded
                                          (refunded — после возврата, из delivered/completed)
```

| Из \ В | awaiting_payment | paid | packed | shipped | delivered | completed | cancelled | refunded |
|---|---|---|---|---|---|---|---|---|
| **new** | ✓ | ✓ | — | — | — | — | ✓ | — |
| **awaiting_payment** | — | ✓ | — | — | — | — | ✓ | — |
| **paid** | — | — | ✓ | — | — | — | ✓ | ✓ |
| **packed** | — | — | — | ✓ | — | — | ✓ | ✓ |
| **shipped** | — | — | — | — | ✓ | — | — | ✓ |
| **delivered** | — | — | — | — | — | ✓ | — | ✓ |
| **completed** | — | — | — | — | — | — | — | ✓ |

- `new` — заказ создан (наложенный платёж/оплата при получении) либо ждёт выбора оплаты.
- `awaiting_payment` — выбрана онлайн-оплата, ждём подтверждения (mock/будущий провайдер).
- `paid` — оплачен (mock или реально). Синхронизируется с `payment_status='paid'`.
- `packed` → `shipped` → `delivered` → `completed` — логистический цикл (`shipped`/`delivered`
  синхронизируются со СДЭК в Этапе 4 через `delivery_status`).
- `cancelled` — отмена до отгрузки (возврат резерва остатков, §6).
- `refunded` — возврат (Gang Auto «возврат 14 дней», `docs/06`); ставит `payment_status='refunded'`,
  возврат денег — будущий под-этап. Эффект над остатками зависит от исходного статуса (§6): refund
  ДО отгрузки (`paid/packed`) возвращает резерв; refund ПОСЛЕ отгрузки (`shipped/delivered/completed`)
  не трогает `inventory` (резерв уже списан commit-ом — release запрещён, anti-oversell).

**(B) Статус оплаты (`orders.payment_status`):** `pending → authorized → paid`; ветви `→ failed`,
`paid → refunded`. На Этапе 3 переходы — ручные/mock (нет провайдера). `paid` проставляет `paid_at`.

**(C) Статус доставки (`orders.delivery_status`):** `pending → registered → in_transit →
delivered`; ветви `→ returned`, `→ cancelled`. **Источник истины в Этапе 4 — СДЭК webhook**
(`docs/03` §4.5): маппинг статусов СДЭК → `delivery_status`. На Этапе 3 — только поля и ручная
смена в админке.

> Переходы заданы **данными** (таблица допустимых переходов в `lib/orders/status.ts`), а не разбросаны
> по коду — единый источник, тестируемо, как RBAC по кодам (ADR-005). UI рисует кнопки только для
> разрешённых из текущего статуса переходов.

> **Защита от TOCTOU-гонки смены статуса (все три машины).** `canTransition(from→to)` проверяется по
> прочитанному `from`, но между чтением и записью статус может сменить конкурентный запрос. Поэтому
> сам `UPDATE` — **guarded**: `UPDATE orders SET <col> = :to ... WHERE id = :id AND <col> = :from
> RETURNING id`. Побочные эффекты (резерв остатков, откат промокода) и запись `order_status_history`
> выполняются **в той же транзакции и только при `affected rows = 1`**; иначе бросается
> `OrderError('conflict')` и транзакция откатывается (история/эффекты не пишутся). Два параллельных
> перехода из одного `from` сериализуются на блокировке строки `orders`, второй видит уже изменённый
> статус → 0 строк → конфликт. Юнит-логика покрыта `tests/orders/actions.test.ts` (guarded UPDATE по
> affected rows), полная конкурентность — интеграционным тестом на живой БД (`tests/orders/repository.test.ts`,
> `skipIf` без `DATABASE_URL`).

## 3. Промокоды и скидки

### 3.1. Модель применения

Промокод применяется **к итогу заказа поверх каталожных цен** (после того как товарные цены уже
включают каталожную скидку `compare_at_price`, ADR-009). Порядок расчёта итога (детерминированный):

1. `items_total` = Σ(`unit_price` × `qty`), где `unit_price` — **эффективная** цена из каталога
   (`base_price`/`price_override`+`price_delta`), уже с учётом каталожной «было/стало». Каталожная
   скидка `compare_at_price` **не** попадает в `discount_total` (она уже «в цене») — `discount_total`
   только про промокод.
2. `discount_total` = скидка промокода (см. §3.2), не больше `items_total`.
3. `delivery_total` = стоимость доставки (заглушка/СДЭК), с учётом **порога бесплатной доставки** и
   промокода `free_delivery` (§3.3).
4. `grand_total` = `items_total − discount_total + delivery_total` (≥ 0, иначе ошибка расчёта).

### 3.2. Типы промокода

| `kind` | Эффект | `discount_total` | Примечание |
|---|---|---|---|
| `percent` | −value % от `items_total` | `round(items_total × value/100)`, обрезано `max_discount` | Gang Auto |
| `fixed` | −value ₽ | `min(value, items_total)` | фикс-скидка |
| `free_delivery` | бесплатная доставка | 0 (влияет на `delivery_total → 0`) | переопределяет порог |
| `bogo` | «N по M» / «3 по 2» | задел: считается из позиций (дешёвая бесплатно) | **модель**; исполнение — Этап 5.2 |

> **«N по M» (`bogo`)** проектируется как тип акции: поля `bogo_buy_qty`/`bogo_pay_qty` (напр. 3/2 =
> «3 по 2»). Расчёт скидки: на каждые `buy_qty` одинаковых/подходящих позиций бесплатна
> `(buy_qty − pay_qty)` самых дешёвых. На Этапе 3 — **схема и валидатор-заглушка** (промокод с
> `kind='bogo'` принимается, но движок расчёта может быть реализован в Этапе 5.2). Так модель заложена
> без блокировки этапа.

### 3.3. Порог бесплатной доставки

Из настроек магазина (**не хардкод**): `SHOP_FREE_DELIVERY_THRESHOLD` (env/`lib/config/env.ts`,
по умолчанию `0` = выключено; для Gang Auto = `3000`). Правило в `quote`/создании заказа:

```
delivery_total = (items_total - discount_total) >= SHOP_FREE_DELIVERY_THRESHOLD  → 0
                 промокод kind='free_delivery' активен                           → 0
                 иначе                                                           → delivery_cost (СДЭК/заглушка)
```

> Порог сравнивается с суммой **после** скидки промокода (`items_total − discount_total`) — типичная
> e-commerce-семантика; вынесено в чистую функцию (тестируемо). Для THE CASE порог не задаётся
> (стоимость доставки всегда из расчёта СДЭК).

### 3.4. Валидация промокода (при `quote` и при создании заказа)

Единая чистая функция `validatePromo(code, ctx)` → `{ ok, discount, reason? }`, вызывается **дважды**:
в `POST /cart/quote` (показать скидку) и **повторно** внутри транзакции `POST /orders` (anti-tamper —
витрине нельзя доверять предрасчитанную скидку). Проверки:

1. промокод существует, `is_active`, `now()` в `[starts_at, ends_at]`;
2. `items_total >= min_order_total`;
3. **лимит всего:** `used_count < usage_limit` (NULL = безлимит);
4. **лимит на покупателя:** число строк в `promo_redemptions` по (`promo_code_id`, `customer_email`)
   `< per_customer_limit`;
5. расчёт скидки по `kind` (§3.2), обрезка `max_discount`.

**Идемпотентность/гонки:** инкремент `used_count` и вставка `promo_redemptions` — **в той же
транзакции**, что создание заказа; `UNIQUE(promo_code_id, order_id)` исключает двойной учёт при
повторной отправке (idempotency-ключ заказа). Лимит «всего» проверяется и инкрементится атомарно:
`UPDATE promo_codes SET used_count = used_count + 1 WHERE id = $1 AND (usage_limit IS NULL OR
used_count < usage_limit) RETURNING used_count;` — если 0 строк, лимит исчерпан гонкой → откат
транзакции (заказ не создаётся / создаётся без промокода — по политике, по умолчанию ошибка `quote`
устаревает, см. §4.2).

**Откат применения при отмене/возврате.** При переходе заказа в `cancelled`/`refunded`, если у
заказа есть `promo_code_id`, в **той же транзакции** перехода статуса (§2.8) выполняется откат учёта:
`DELETE FROM promo_redemptions WHERE order_id = :id AND promo_code_id = :pid RETURNING id`, затем —
**только если удалена хотя бы одна строка** — `UPDATE promo_codes SET used_count = GREATEST(used_count
- :deletedCount, 0) WHERE id = :pid`. Это **идемпотентно**: повторная отмена/возврат уже без
редемпшна удалит 0 строк и счётчик не тронет; `GREATEST(...,0)` страхует от ухода в минус. Так
освобождается слот лимита для покупателя/всего. Покрыто `tests/orders/actions.test.ts` (откат при
cancel/refund, отсутствие отката без промокода и при `shipped`, идемпотентность повторного отката).

## 4. Server Actions (админка) + Storefront API (витрина)

### 4.1. Server Actions админки (через `defineAction`, право + аудит)

Все — через `lib/server/action.ts` (`guard → Zod → БД → invalidate → audit`), право
`orders.read`/`orders.write` (уже в `lib/auth/permissions.ts`), аудит-события `orders.*`. Новый слой
`lib/orders/{types,schemas,repository,actions,status,pricing,promo}.ts` (единые файлы, как
`lib/catalog/*` — без пер-сущностного дробления).

```ts
// lib/orders/actions.ts (сигнатуры — эскиз)

// ---- Чтение/поиск (orders.read) ----
listOrders(input: {
  q?: string;                 // поиск по номеру/email/телефону
  status?: OrderStatus;
  paymentStatus?: PaymentStatus;
  deliveryType?: DeliveryType;

// ---- Мутации (orders.write; каждая пишет order_status_history + audit) ----
changeOrderStatus(input: { id: string; to: OrderStatus; comment?: string }): ActionResult<OrderDetail>;
   // валидирует переход по таблице §2.8(A); эффект над остатками — по ПАРЕ from→to
   // (§6, stockEffectFor): cancel/refund ДО отгрузки → release; →shipped → commit;
   // refund отгруженного (shipped/delivered/completed) → НИЧЕГО (anti-oversell);
   // атомарно в транзакции.
setPaymentStatus(input: { id: string; to: PaymentStatus; comment?: string }): ActionResult<OrderDetail>;
setDeliveryStatus(input: { id: string; to: DeliveryStatus; comment?: string }): ActionResult<OrderDetail>; // Этап 4 синхронизирует автоматически
cancelOrder(input: { id: string; reason?: string }): ActionResult<OrderDetail>;     // = changeOrderStatus→cancelled + возврат резерва
refundOrder(input: { id: string; reason?: string }): ActionResult<OrderDetail>;     // → refunded + payment_status=refunded (деньги — будущее)
createOrderManually(input: ManualOrderInput): ActionResult<OrderDetail>;            // source='admin'; та же ре-валидация цен/остатков/резерв, что у витрины
exportOrders(input: { /* те же фильтры, что listOrders */ }): ActionResult<{ csv: string }>; // выгрузка CSV

// ---- Промокоды (CRUD; orders.write) ----
listPromoCodes(input: { q?: string; isActive?: boolean; page?: number }): ActionResult<{ rows: PromoCode[]; total: number }>;
createPromoCode(input: PromoCreateInput): ActionResult<PromoCode>;
updatePromoCode(input: PromoUpdateInput): ActionResult<PromoCode>;
deletePromoCode(input: { id: string }): ActionResult<{ id: string }>;   // history заказов не рушится (snapshot promo_code + ON DELETE SET NULL)
```

**Контракт частичного обновления промокода (`PromoUpdateInput`, data-integrity, баги #17/#18).**
`PromoUpdateSchema` строится из отдельного `promoUpdateShape` — поля `.optional()` **БЕЗ
`.default(...)`** (а не `.partial()` поверх create-схемы). Иначе Zod при отсутствии ключа
подставляет DEFAULT, и handler через `COALESCE(${default}, col)` затирает реальные значения
(`value→'0'`, `isActive→true`, `comment→''`, `priority→100`, `stackable→false`), а
`applyScope→'cart'` дополнительно включает управление таргетами и **стирает `promo_targets`**.
Правило: передавай в `updatePromoCode` только реально изменяемые поля.
- Опущенный ключ → колонка **не трогается** (handler видит `undefined` → пишет `null` в COALESCE).
- Явный `null` у nullish-полей (`maxDiscount`, `usageLimit`, `startsAt`, …) → колонка **очищается**.
- `promo_targets` пересоздаются (DELETE+INSERT) **только** если передан `applyScope` (в т.ч.
  `'cart'` — тогда чистим) **или** массив `targets`. Без упоминания scope/targets таргеты
  сохраняются. Семантика create (`PromoCreateSchema` с дефолтами) не изменилась.

### 4.2. Storefront API (расширение `/api/storefront/v1`, ADR-008/010)

Новые route handlers под `app/api/storefront/v1/`, через тот же конвейер `lib/storefront/response.ts`
(authorize → module-flag → rate-limit → CORS). **Важно:** заказные эндпоинты проверяют флаг
**`orders`** (а текущий `runStorefront` проверяет `catalog`) — конвейер расширяется параметром
требуемого модуля. Создание заказа (мутация) — **строже** по auth (требует API-ключ, не только
Origin; ADR-008).

```
POST /api/storefront/v1/cart/quote
  body: { items: [{ variantId|productId, qty }], promoCode?, delivery?: { type, city, pvzCode? } }
  → СЕРВЕР: цены из каталога (lib/catalog/repository) → проверка остатков (inventory) →
    промокод (validatePromo) → доставка (заглушка/поля под СДЭК Этапа 4) →
    { itemsTotal, discountTotal, deliveryTotal, grandTotal, currency,
      lines:[{ name, sku, unitPrice, qty, lineTotal, inStock }],
      promo:{ applied, code, discount, reason? },
      delivery:{ type, cost, free, freeThresholdMet } }
  Ничего не создаёт. Недостаток остатка/невалидный промокод → отражён в ответе (200 с флагами или 400).
  Проверка остатка — по СУММАРНОМУ спросу на юнит (productId+variantId), а не полинейно
  (BUG C, волна 4): две линии одного юнита по qty каждая агрегируются и сверяются с
  доступным остатком нарастающим итогом — как кумулятивный резерв createOrder. Иначе quote
  говорил «можно» (каждая линия ≤ остатка), а createOrder падал out_of_stock (вторая линия
  резервируется поверх первой). Оверселла не было, но поведение было рассогласовано.

POST /api/storefront/v1/orders                       (auth строже: API-ключ; flag orders)
  headers: Idempotency-Key (рекоменд.)               → orders.idempotency_key (UNIQUE partial)
  body: { items:[{variantId|productId, qty}], customer:{name,email,phone},
          delivery:{type,city,cityCode?,address?,pvzCode?}, paymentMethod, promoCode?, comment? }
          // city — имя города (несёт назначение для КУРЬЕРСКОЙ доставки → расчёт СДЭК);
          // cityCode — опц. числовой код города СДЭК (точнее имени, если витрина его знает).
  → ТРАНЗАКЦИЯ (ADR-010, anti-tamper):
     1) ре-валидация цен/остатков ЗАНОВО из каталога (НЕ доверяем телу запроса);
     2) проверка/расчёт промокода и доставки заново (validatePromo, порог);
     3) резерв остатка: inventory.reserved += qty атомарно (§6); недостаток → 409 + откат;
     4) выдача номера (§2.7), вставка orders + order_items (СНИМОК) + order_status_history(new);
     5) учёт промокода: used_count++, promo_redemptions (idempotency, §3.4);
     6) upsert customers по email (если включено);
  → { number, status, grandTotal, currency, accessToken? }   // accessToken — для чтения статуса в ЛК
  Повтор с тем же Idempotency-Key → возвращает существующий заказ (не дублирует).
  ГОНКА двух параллельных запросов с одним Idempotency-Key: второй нарывается на
  UNIQUE orders_idempotency_uniq (PostgreSQL 23505) внутри транзакции — слой данных
  ловит это, перечитывает строку по ключу и возвращает её как идемпотентный повтор
  (reused), а не падает с 500 (reliability-инвариант, см. createOrder catch).

GET /api/storefront/v1/orders/{number}               (flag orders)
  auth заказа: ?token=<accessToken> ИЛИ заголовок + совпадение email (защита от перебора номеров)
  → { number, status, paymentStatus, deliveryStatus, grandTotal, currency,
      items:[{name, sku, qty, unitPrice, lineTotal}], delivery:{type,city,track?},
      history:[{ status, at }] }      // трекинг для ЛК витрины (THE CASE «отслеживание»)
```

> **Anti-tamper (ADR-010):** витрина присылает только `variantId`+`qty`+выборы; **все деньги
> считает сервер** из каталога. `quote` и создание заказа считают независимо; цена из тела запроса
> игнорируется. Это защищает от подмены цены/итога на клиенте. `GET /orders/{number}` защищён
> токеном заказа или email-совпадением (нельзя перебрать чужие номера).

> **Границы входа (Zod, `lib/orders/schemas.ts`).** `qty` каждой позиции — целое **1..10000**
> (`quantitySchema`); массив `items` (quote и создание) — **1..200** позиций (`MAX_CART_ITEMS`). Это
> защищает корректность расчёта: `lineTotalMinor = toMinor(unitPrice) × qty` считается в `number`, и
> при чрезмерном `qty` произведение могло бы превысить `Number.MAX_SAFE_INTEGER` с тихой потерей
> точности суммы. Дополнительный рубеж — в `pricing.lineTotalMinor`: при переполнении
> `MAX_SAFE_INTEGER` бросается явная ошибка вместо записи «битой» суммы. Покрыто
> `tests/orders/schemas.test.ts` и `tests/orders/pricing.test.ts`.

> **Доменные ошибки доходят до UI (`lib/orders/errors.ts`).** `OrderError` наследует
> `PublicActionError` (`lib/server/action.ts`), поэтому её человекочитаемый `message` («Заказ не
> найден», «Недопустимый переход статуса…», «Промокод с таким кодом уже существует», конфликт
> параллельного перехода) маппится пайплайном `defineAction` в `{ ok:false, error:'validation',
> message }`, а не в безликий `error:'internal'`. Машиночитаемый `code` (`not_found` /
> `invalid_transition` / `duplicate_code` / `conflict` / `out_of_stock` / …) остаётся в логах и
> тестах, наружу отдельно не утекает.

## 6. Резерв / списание остатков (атомарность, гонки)

Опирается на существующую таблицу `inventory` (Этап 2.5, `0010`): `quantity` (физический остаток),
`reserved` (резерв), `CHECK (reserved <= quantity)`, `CHECK (quantity >= 0)`, `CHECK (reserved >= 0)`.
**Доступно к продаже** = `quantity − reserved` (вычисляется).

| Событие | Операция над `inventory` | Гонко-безопасность |
|---|---|---|
| **Создание заказа** (резерв) | `UPDATE inventory SET reserved = reserved + ${qty} WHERE <unit> AND quantity - reserved >= ${qty} RETURNING id` | условие `quantity - reserved >= qty` + проверка affected rows = 1; иначе откат всей транзакции заказа (409 «нет остатка») |
| **Отгрузка** (`status→shipped`, из `packed`) | `UPDATE inventory SET quantity = quantity - ${qty}, reserved = reserved - ${qty} WHERE <unit> AND reserved >= ${qty} AND quantity >= ${qty}` | CHECK-и БД + условие; списываем ровно зарезервированное (commit: резерв заказа снят) |
| **Отмена/возврат ДО отгрузки** (`new/awaiting_payment/paid/packed → cancelled/refunded`) | `UPDATE inventory SET reserved = reserved - ${qty} WHERE <unit> AND reserved >= ${qty}` | возврат резерва: резерв заказа ещё держится (commit не выполнялся) |
| **Возврат ПОСЛЕ отгрузки** (`shipped/delivered/completed → refunded`) | **НИЧЕГО не делаем над `inventory`** (ни release, ни авто-restock) | резерв заказа уже списан commit-ом при отгрузке — release запрещён (см. ниже); физический restock — отдельная операция приёмки возврата |

- **Эффект над резервом зависит от ПАРЫ `from → to`, а не только от `to`** (anti-oversell, волна 6 —
  см. `stockEffectFor(from, to)` в `lib/orders/actions.ts`). До волны 6 эффект определялся только по
  `to`: для `cancelled/refunded` всегда делался `release`. Это был **CRITICAL-баг (oversell)**: граф §2.8(A)
  разрешает `shipped/delivered/completed → refunded`, но при входе в `shipped` уже выполнен commit
  (`reserved -= qty`) — резерв этого заказа снят. `releaseReservation` гардит по **глобальному** агрегату
  строки `inventory` (`WHERE reserved >= qty`) без привязки к заказу (per-order резерва в схеме `0010`
  нет), поэтому `release` при refund отгруженного не находил «свой» резерв (он списан commit-ом), но при
  наличии резерва **других** открытых заказов на том же SKU гард проходил и `release` крал **чужой**
  резерв → повреждение резервов + oversell (`CHECK inventory_reserved_le_qty` это не ловит). Фикс:
  `release` делается **только** когда `from ∈ {new, awaiting_payment, paid, packed}` (резерв ещё держится);
  для `from ∈ {shipped, delivered, completed}` эффект — `none`. Авто-restock (`quantity += qty`) при
  возврате отгруженного **намеренно НЕ делаем** (консервативно: не раздуваем остаток товаром, который мог
  не вернуться физически; приёмка возврата на склад — отдельная операция вне этого перехода).
- Все списания/резервы заказа — **в одной транзакции** с изменением статуса/созданием заказа. Если
  хоть одна позиция не зарезервировалась — `ROLLBACK`, заказ не создаётся.
- `<unit>` = `(product_id, COALESCE(variant_id, ...), warehouse_code='main')` — единичный остаток
  (как `inventory_unit_uniq` в `0010`). Мультисклад (`warehouse_code`) — задел, на Этапе 3 всегда
  `'main'`.
- Конкурентные заказы на последний остаток: атомарный `UPDATE ... WHERE quantity - reserved >= qty`
  гарантирует, что только один заказ зарезервирует (второй получит affected rows = 0 → откат). Без
  явных блокировок строк (оптимистично на условии), что исключает дедлоки.

## 7. Точки стыковки со СДЭК (Этап 4) и онлайн-оплатой (будущее)

**Заложено в схеме `0012` (структура, без интеграции):**

- **Доставка (Этап 4, `cdek`):** `delivery_type` (courier/pvz/pickup), `delivery_city`,
  `delivery_address`, `delivery_pvz_code`, `delivery_cost`, `cdek_uuid`, `cdek_track`,
  `delivery_status` (отдельная статус-машина §2.8(C)). В `POST /cart/quote` — **заглушка расчёта**
  (поле `delivery.cost`, заполняется фиксированной/нулевой стоимостью + порог бесплатной доставки);
  реальный расчёт/ПВЗ — эндпоинты `…/delivery/cdek/*` Этапа 4 (ADR-008). Webhook СДЭК Этапа 4
  обновляет `delivery_status`/`cdek_track`/`orders.status` (registered→in_transit→delivered).
- **Оплата (будущий под-этап):** `payment_method` (cod/card/sbp/cdek_pay/invoice), `payment_status`
  (pending→…→paid/refunded), `paid_at`, `payment_ref`. На Этапе 3 — фиксация выбора + ручная/mock
  смена статуса. Провайдеры (СДЭК PAY/эквайринг/СБП) подключаются как отдельный модуль позже без
  изменения схемы заказа (поля уже есть).

> Структура полей подобрана так, чтобы Этап 4/оплата были **аддитивными** (заполнение полей +
> воркеры/webhook), а не рефактором модели заказа — «обновление не ломает данные» (`docs/02`).

## 8. Декомпозиция Этапа 3 на задачи (параллелизуемые пакеты)

Принципы (роадмап §«Принципы»): **«сначала тесты, потом код»** (ADR-004 — первый пункт каждой
задачи), идемпотентные миграции, флаг модуля `orders`, переиспользование ядра/каталога без
дублирования. Ниже — пакеты с **непересекающимися файлами** (можно вести параллельно разными
агентами Backend/Frontend), с критериями приёмки и зависимостями.

### Пакет A — Миграции и доменный слой схемы заказа (фундамент)
- **Файлы:** `db/migrations/0012_orders.sql`, `0013_customers.sql`, `0014_promo_codes.sql`,
  `0015_promo_redemptions.sql`, `0016_order_number_counter.sql`; `lib/orders/types.ts`,
  `lib/orders/status.ts` (таблицы переходов §2.8), `lib/orders/schemas.ts` (Zod).
- **Тесты (раньше кода):** идемпотентность всех миграций (двойной накат); CHECK-и (суммы ≥ 0,
  reserved≤quantity косвенно, статусы из whitelist); валидатор переходов статусов (допустимые/
  недопустимые); генерация номера заказа (формат, уникальность, конкурентный инкремент).
- **Критерии приёмки:** накат одной командой; повтор безопасен; переходы статусов — чистая

- **Критерии приёмки:** anti-tamper (цена из каталога, не из входа); транзакция атомарна; промокод

- **Зависимости:** A. Переиспользует `lib/catalog/repository.ts` (цены), `lib/catalog/pricing.ts`,

### Пакет D — Storefront API заказов (публичный контур)
- **Файлы:** `app/api/storefront/v1/cart/quote/route.ts`, `app/api/storefront/v1/orders/route.ts`,

- **Тесты (раньше кода):** `quote` считает сервер (подмена цены в теле игнорируется); создание
  атомарно + idempotency (повтор не дублирует); недостаток остатка → 409; auth строже для создания
  (нужен ключ); `GET /orders/{number}` защищён токеном/email; флаг `orders` выключен → 404.
- **Критерии приёмки:** ADR-010 (anti-tamper) соблюдён; CORS/rate-limit как у каталога; DTO без

### Пакет E — UI админки `/admin/orders/*` + промокоды
- **Файлы:** `app/admin/(panel)/orders/*` (список/карточка/new), `app/admin/(panel)/promo/*`;
  пункты меню в `lib/admin/nav.ts`.
- **Тесты (Playwright a11y + Vitest):** список/фильтры; карточка с кнопками только допустимых
  переходов; скрытие раздела при выключенном модуле/без права; форма промокода.
- **Критерии приёмки:** менеджер управляет заказами по RBAC; переходы пишут историю; раздел уважает

### Пакет F — Конфигурация и документация
- **Файлы:** `lib/config/env.ts` (+`SHOP_FREE_DELIVERY_THRESHOLD` default 0, `SHOP_ORDER_PREFIX`

  никаких хардкодов магазина.
- **Зависимости:** параллельно всем; финализуется после A–E (синхронизация статусов роадмапа).

## 10. Резюме решений (для журнала)

1. **Единая модель заказа** (ADR-002): `orders` + `order_items` + `order_status_history` + задел
   `customers`; промокоды `promo_codes` + `promo_redemptions`; нумератор `order_number_counters`.
   Миграции **`0012–0016`**, идемпотентные, гранты `admik_app`. Деньги `numeric(14,2)`, PK `uuid`,
   время `timestamptz`.
2. **Снимок позиций** (ADR-010): `order_items` денормализует name/sku/price/attributes — изменение
   каталога не переписывает историю; ссылки `product_id`/`variant_id` — `ON DELETE SET NULL`.
3. **Корзина — клиентская (на витрине), бэкенд считает** (ADR-010): `POST /cart/quote` (расчёт без
   создания) и `POST /orders` (создание). **Сервер — единственный источник цен/итога (anti-tamper)**;
   витрине доверяем только `variantId`+`qty`+выборы.
4. **Три статус-машины** (заказ/оплата/доставка) с whitelist-переходами (данными, тестируемо);
   `delivery_status` синхронизируется со СДЭК в Этапе 4. Переходы пишут `order_status_history` +
   `audit_log`.
5. **Промокоды** percent/fixed/free_delivery (+`bogo` как модель «N по M», исполнение — Этап 5.2);
   поверх итога, после каталожной скидки `compare_at_price`; порог бесплатной доставки —
   `SHOP_FREE_DELIVERY_THRESHOLD` (не хардкод); идемпотентность через `promo_redemptions` UNIQUE.
6. **Резерв/списание** на `inventory.reserved`/`quantity` (Этап 2): резерв при создании, списание при
   отгрузке, возврат при отмене; атомарно `UPDATE ... WHERE quantity-reserved>=qty` (гонко-безопасно,
   без блокировок).
7. **Out of scope:** онлайн-оплата (только поля+mock; провайдеры — будущее) и СДЭК-интеграция
   (Этап 4 — только поля доставки/заглушка расчёта). Структура полей делает Этап 4/оплату аддитивными.
8. **Права** `orders.read`/`orders.write` — уже есть; новых прав ядра нет; промокоды под
   `orders.write`. Модуль уважает флаг `orders`.
9. **Новых npm-зависимостей нет.** Новый ADR-010 (anti-tamper + снимки) — в конце `docs/01`.
