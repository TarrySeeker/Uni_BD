# Онбординг магазина: прикрутить Admik к готовому макету витрины

Оркестратор полного цикла из `docs/20-онбординг-витрины-playbook.md`. Ты —
оркестратор мультиагентной системы (CLAUDE.md): декомпозируешь и поручаешь работу
профильным агентам, сам держишь план, чекпоинты и проверки. Перенос на магазин =
**конфигурация + данные**, НЕ правка кода платформы (ADR-003).

Перед запуском прочитай: `START-HERE.md`, `CLAUDE.md`, `docs/20`, `docs/21`, `docs/22`.

## Инварианты (нарушать нельзя)
- **TDD:** сначала тест (падает) → код до зелёного.
- **Две проверки** перед коммитом: код-гейт `/gate` И живой прогон в браузере.
- **Мультитенантность:** ноль хардкода ниши в коде; всё — в `.env` и `shop_settings`.
- **Миграции иммутабельны/аддитивны** (`check-migrations.sh`); существующие не трогать.
- **Секреты только из env**; боевые ключи СДЭК/Т-Банк — на сервере, не в git.
- **Документируй каждую правку** (`documentation-manager` / `history-manager`).

## Принцип чекпоинтов и резюмируемости
Владелец — не программист, и сессия может прерваться. Поэтому **каждый Этап
завершается чекпоинтом**:
1. **Запись в `docs/00`** (журнал: что сделано на этапе) **и в `docs/10`** (текущее
   состояние + «как продолжить с этого места»).
2. **Список «что нужно от владельца, чтобы продолжить»** (недостающие данные/ключи/
   решения). Отсутствие данных — не повод останавливаться: помечай пункт, подставляй
   заглушку/mock и иди дальше по тому, что можно сделать без владельца.
Возобновление прерванной сессии — всегда с чтения `docs/10` + `docs/00`.

## Этапы (в порядке владельца; детали — в docs/20)

### Этап 0 — Преполёт
`bash .claude/setup.sh` (доверие к папке + включение MCP), `pnpm install`, найди
готовый макет витрины в `./storefront` (иначе уточни путь и поправь build-context в
`docker-compose.yml`), `cp .env.example .env` (пока дефолты/mock — заполнишь в Этапе 4).
Прочитай `docs/02`, `docs/09`, `docs/16`, `docs/21`. Контракты библиотек (Next 16 /
React 19 / Zod 4 / postgres.js) сверяй через MCP **context7**, не по памяти.
**Чекпоинт:** окружение готово (макет на месте, `pnpm install` ок, `.env` создан).

### Этап 1 — ИЗУЧИТЬ ФРОНТЕНД ПЕРВЫМ → «профиль магазина»
До интервью и до БД. Запусти **параллельных read-only** `Explore`-агентов по
`./storefront` и собери профиль: каталог (категории/варианты/атрибуты/вес-габариты),
навигация и страницы (→ CMS), брендинг/контакты/реквизиты, главная (hero/about/
преимущества/доставка), способ корзины/оформления/оплаты, стек витрины (Router,
Dockerfile, порт). **Чекпоинт:** черновой профиль магазина зафиксирован в `docs/00`;
список «что не удалось вычитать из макета» → вопросы для Этапа 2.

### Этап 2 — ДЕТАЛЬНОЕ ИНТЕРВЬЮ С ВЛАДЕЛЬЦЕМ (простыми словами)
Пройди с владельцем опросник **`docs/22-интервью-владельца.md`** — 14 блоков простым
нетех-языком: ниша/ассортимент, категории, варианты (размеры/цвета/объёмы), бренды/
характеристики, валюта и цены, доставка (СДЭК?), оплата (Т-Банк? чек 54-ФЗ?), реквизиты/
контакты/юр.данные, домены, нужные модули, брендинг/лого/цвета, тексты главной, CMS-
страницы/политики, режим сейчас (mock/боевой). **Опирайся на профиль из Этапа 1** —
что видно в макете, подтверждай, а не спрашивай с нуля; чего нет — спрашивай подробно.
**Чекпоинт:** профиль магазина дополнен интервью и записан в `docs/00`; незакрытые
пункты (реквизиты/ключи/тексты) → список «что нужно от владельца».

### Этап 3 — СТАДИЙНО ПОДНЯТЬ БД на своём VPS
Поручи `devops-engineer`/`backend-engineer`, по одному шагу с проверкой:
`make up` (app + postgres + redis + minio + caddy) → `make init` (`init-shop.sh`:
расширения+роли → миграции → seed прав/ролей/владельца → опц. demo-каталог) →
`make smoke` → проверь `GET /api/health` = `{"status":"ok"}` и вход в `/admin` под
`OWNER_EMAIL`. **Чекпоинт:** БД поднята, миграции накатаны, health зелёный, владелец
заходит в админку; в `docs/00`/`docs/10` — как переподнять; список «что нужно»
(например, боевые пароли БД/MinIO для сервера — если это уже боевой VPS).

### Этап 4 — Ретематизация (конфигурация + данные, БЕЗ правок бизнес-логики)
`.env` под магазин из профиля: `SHOP_NAME`, домены, `SHOP_ORDER_PREFIX`, `SHOP_CURRENCY`,
`ADMIK_MODULES`, `STOREFRONT_*`, `NEXT_PUBLIC_ADMIK_API_URL`, доступы БД/MinIO/Redis;
`CDEK_*`/`TBANK_*` пустыми = mock. Каталог под нишу: через админку (боевое наполнение)
и/или адаптация нейтрального `db/seed/demo-catalog.sql` (с обновлением seed-теста по TDD).
Брендинг/контент/навигация/главная/CMS — через `/admin/settings` и `/admin/cms`
(доезжают через `GET /settings`, `GET /pages`). Ноль хардкода ниши в коде.
Валюту задай **явно и полностью**: помимо `SHOP_CURRENCY` выставь символ, локаль и число
дробных знаков (symbol/locale/fractionDigits) — иначе неявный дефолт рантайма даёт «неродной»
формат сумм. Страница настроек может иметь **несколько независимых кнопок «Сохранить»** (по
секциям) — нажми каждую и верифицируй сохранение по API (`GET /settings`/`GET /pages` вернули
новые значения), а не по одной кнопке.
**Чекпоинт:** `.env` заполнен, каталог/атрибуты/брендинг/страницы соответствуют профилю,
seed-тесты зелёные; в `docs/00` — список изменённых конфиг/сид-файлов.

### Этап 5 — Сращивание витрины со Storefront API
Поручи `frontend-engineer`: перевести макет в чистого потребителя API по маппингу
`docs/21 §7` (products/categories/brands, cart/quote, orders+accessToken, orders/:number,
cdek cities/pvz/calculate, payments/tbank/init, pages, settings, promotions, leads,
newsletter). Два адреса API (SSR `http://app:3000` + браузерный `NEXT_PUBLIC_ADMIK_API_URL`),
авторизация ключ/Origin. Anti-tamper: не слать цены/итоги/вес с витрины. Не хватает
эндпоинта → проектирует `solution-architect`, реализует backend по TDD аддитивно
(НЕ городить логику в витрине). **Чекпоинт:** витрина рендерит каталог/карточку из API,
корзина считается через `quote`, заказ оформляется, доставка/оплата работают (mock);
статус сращивания по разделам — в `docs/00`.

### Этап 6 — Две проверки + документирование
**Проверка 1 (код):** `/gate` (typecheck/lint/vitest/check-migrations/security) →
`qa-engineer`/`security-engineer`. **Проверка 2 (браузер):** живой прогон (`/deploy-stand`
или `/verify` + playwright, путь покупателя + админка, mock) + `/integration-check`
(СДЭК+Т-Банк mock e2e) + `make smoke`. Затем **документирование:** обнови `docs/00`,
`docs/10`, README инстанса; зафиксируй профиль магазина и изменённые конфиг/сид-файлы.
Коммить ТОЛЬКО после обеих проверок. Боевой деплой — по `docs/09`
(`SEED_DEMO_CATALOG=false`, реальные ключи). **Финальный чекпоинт:** Definition of Done
достигнут; в `docs/10` — как поднять именно этот магазин.

## Definition of Done (см. START-HERE §2)
БД+миграции+seed готовы · каталог/атрибуты/брендинг ретематизированы конфигурацией
и данными (ноль правок бизнес-логики) · витрина подключена к Storefront API · админка
работает и заведённое доезжает до витрины · все обязательные CMS-страницы (из ссылок
футера/чекаута и интервью) не только созданы, но и **опубликованы** (`status=published` —
draft витрина не отдаёт), юр-ссылки чекбокса согласия ведут на существующую страницу ·
каждое поле настроек/контента из профиля реально заполнено, а не оставлено на дефолтную
заглушку платформы (сверено по `GET /settings`/`GET /pages`) · `/gate` зелёный + живой
прогон пройден · всё задокументировано.

## Итог скилла

их списками «что нужно от владельца», ссылки на обновлённые доки.
