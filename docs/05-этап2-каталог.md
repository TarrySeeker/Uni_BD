## 0. Что уже готово в ядре (переиспользуем, не переписываем)

Этап 1 завершён (`docs/04`, `docs/03`). Этап 2 строится **поверх** этих контрактов:

| Готовый контракт | Файл | Как используем в каталоге |
|---|---|---|
| Клиент БД `sql` (tagged templates, роль `app`) | `lib/db/client.ts` | все запросы каталога — только через `sql\`...\`` (анти-SQLi) |
| `defineAction({ permission, input, handler })` | `lib/server/action.ts` | все мутации каталога — единый пайплайн guard→Zod→БД→invalidate→audit |
| Права `catalog.read` / `catalog.write` | `lib/auth/permissions.ts` | **уже заведены** — новые права для каталога не нужны |
| `can` / `requirePermission` / `AuthUser` | `lib/auth/rbac.ts` | серверные гварды; owner всесилен |
| `writeAudit(entry, ctx)` + санитизация | `lib/audit/log.ts` | аудит изменений каталога через `handler` → `audit` |
| Пункт меню «Каталог» (`module:'catalog'`) | `lib/admin/nav.ts` | **уже есть** — раздел сам скрывается при выключенном модуле/без права |
| Флаг модуля `catalog` | `lib/config/modules.ts` | UI/Server Actions каталога уважают `isModuleEnabled('catalog')` |
| S3-переменные окружения | `lib/config/env.ts` | `S3_ENDPOINT/REGION/ACCESS_KEY/SECRET_KEY/BUCKET/PUBLIC_URL` **уже описаны** |
| Формат миграций `NNNN_*.sql` + `schema_migrations` | `db/migrations/*` | следующий номер — **0005**; стиль 1:1 как `0001…0004` |
| Формат seed `ON CONFLICT DO NOTHING` | `db/seed/*` | демо-каталог сидируется идемпотентно |

> Следствие: новых прав RBAC, новых пунктов меню и правок ядра Этап 2 **не требует** —
> только новые миграции `0005+`, новый слой `lib/catalog/*` + `lib/storage/*`, страницы
> `app/admin/catalog/*` и seed демо-данных.

## 1. Цель и охват этапа

### 1.1. Цель

Дать платформе модуль управления товарным каталогом: **категории (дерево), товары,
варианты, гибкие характеристики, медиа (S3/MinIO) и остатки**. Каталог — модуль, включаемый
флагом `catalog` (ADR-003, `lib/config/modules.ts`): при выключенном модуле раздел не
отдаётся (UI скрыт уже в `nav.ts`; Server Actions и роуты дополнительно проверяют флаг).
Каталог — фундамент Этапа 3 (заказы потребляют товары и остатки).

### 1.2. В охвате (in scope)

- **Таблицы** (миграции `0005+`): `categories`, `products`, `product_categories`,
  `product_variants`, `attributes`, `attribute_values`, `product_attributes`,
  `product_media`, `inventory`.
- **Хранилище медиа**: модуль `lib/storage/*` — S3-совместимое (MinIO локально / Timeweb/AWS
  в проде) + **mock-режим** (локальная ФС/заглушка) для магазина без боевых ключей.
- **Слой данных** `lib/catalog/*`: репозитории-запросы через `sql`; Zod-схемы; мутации через
  `defineAction` с правами `catalog.read`/`catalog.write`, инвалидацией и аудитом.
- **UI** `app/admin/catalog/*`: список товаров (TanStack Table — фильтры/поиск/пагинация),
  карточка товара (вкладки: основное / варианты / характеристики / медиа / SEO), дерево
  категорий, загрузчик медиа, управление справочником атрибутов.
- **Seed** демо-каталога (категории/атрибуты/товары/варианты/медиа-плейсхолдеры) —
  идемпотентный, для smoke и демонстрации.

### 1.3. Вне охвата (out of scope, в следующие этапы)

- **Корзина/заказы** — Этап 3 (`orders`). Каталог лишь отдаёт товары/остатки.
- **Резервы остатков под заказы, списание при оформлении** — потребитель в Этапе 3; здесь
  закладываем модель остатков (`inventory`) с заделом под резервы, но саму логику резерва не
  реализуем.
- **Импорт CSV (реализация)** — здесь только **проектная заготовка** (точка расширения,
  §6.3); полноценный импортёр — отдельная задача после стабилизации модели.
- **SEO-генерация (sitemap/robots, ЧПУ-рендер витрины)** — Этап 5.3. На Этапе 2 у сущностей
  есть SEO-поля (slug/seo_title/seo_description) и их редактирование в админке, но генерация
  карт сайта и публичная витрина — позже.
- **Несколько складов, поставки, оприходование** — `inventory` спроектирована расширяемой
  (`warehouse_code`), но мультисклад/документооборот — не сейчас.
- **Промо/скидки на товары** — Этап 5.2.

### 1.4. Критерии завершения этапа (Definition of Done)

- [ ] Миграции `0005…00NN_*.sql` накатываются `scripts/init-shop.sh` на пустую БД **одной
      командой**, повторный накат **безопасен** (идемпотентность подтверждена тестом).
- [ ] Все таблицы каталога имеют `app`-гранты (`SELECT/INSERT/UPDATE/DELETE`), UUID-PK
      (`gen_random_uuid`), `TIMESTAMPTZ`, деньги `NUMERIC(14,2)`, `citext` для slug/sku.
- [ ] CRUD категорий (дерево), товаров, вариантов, характеристик, медиа и остатков работает
      **только через `defineAction`**; каждая мутация требует `catalog.write`, чтение —
      `catalog.read`; значимые действия пишутся в `audit_log` (кто/что/до→после).
- [ ] Любое действие без права **блокируется на сервере** (не только скрыто в UI).
- [ ] Загрузка медиа проверяет **magic-bytes** (тип по содержимому, не по расширению),
      ограничивает тип и размер; работает в **mock-режиме** без боевых S3-ключей (smoke
      зелёный без реального S3).
- [ ] При выключенном модуле (`ADMIK_MODULES` без `catalog`) раздел не отдаётся: меню скрыто,
      роуты `/admin/catalog/*` → 404/redirect, Server Actions каталога отклоняются.
- [ ] Список товаров: серверная пагинация/фильтры/поиск (по названию/SKU), сортировки.
- [ ] Нет хардкодов магазина: характеристики/категории/единицы/валюта — данные/настройки, не
      код; копи-paste-развёртывание не нарушено (`docs/02`, ADR-003).
- [ ] Покрытие тестами по ADR-004: тесты написаны **раньше** реализации и зелёные (Vitest —
      логика/валидация/запросы; Playwright — e2e/a11y страниц каталога).

## 2. Схема БД (DDL-эскизы PostgreSQL)

**Общие правила (ADR-002, повтор контракта Этапа 1):**
- Идемпотентность: `CREATE TABLE/INDEX IF NOT EXISTS`, `ADD COLUMN IF NOT EXISTS`,
  `... ON CONFLICT DO NOTHING`. Никаких «голых» `CREATE` без `IF NOT EXISTS`.
- PK бизнес-сущностей — `uuid DEFAULT gen_random_uuid()` (расширение `pgcrypto` уже включено
  миграцией `0001`). Справочные коды (атрибуты) — стабильный `citext`-код-ключ, как у прав.
- Время — `TIMESTAMPTZ` (UTC). Деньги — `NUMERIC(14,2)`. Регистронезависимые строки
  (slug, sku, коды атрибутов) — `citext` (расширение уже включено в `0001`).
- Каждая миграция завершается `INSERT INTO schema_migrations(version,name) ... ON CONFLICT DO NOTHING`
  и выдаёт `app`-гранты на свои таблицы (как `0002…0004`).
- Для FTS — расширение `pg_trgm` (включается в `0005`).

**Состав миграций Этапа 2** (нумерация продолжает `0001…0004`):

| Файл | Содержимое |
|---|---|
| `0005_catalog_extensions_categories.sql` | `CREATE EXTENSION pg_trgm`; `categories` (дерево) + индексы + гранты |
| `0006_catalog_products.sql` | `products`, `product_categories` (M2M) + индексы/FTS + гранты |
| `0007_catalog_variants.sql` | `product_variants` + индексы + гранты |
| `0008_catalog_attributes.sql` | `attributes`, `attribute_values`, `product_attributes` + индексы + гранты |
| `0009_catalog_media.sql` | `product_media` + индексы + гранты |
| `0010_catalog_inventory.sql` | `inventory` + индексы + гранты |

> Разбиение по одной сущности на миграцию упрощает параллельную работу агентов и обзор diff.
> Допустимо объединить близкие (`0007`+`0008`), если так удобнее исполнителю — порядок и
> идемпотентность важнее числа файлов.

### 2.1. `categories` — дерево категорий (миграция `0005`)

```sql
-- 0005_catalog_extensions_categories.sql
CREATE EXTENSION IF NOT EXISTS pg_trgm;   -- триграммный поиск (FTS по названию)

CREATE TABLE IF NOT EXISTS categories (
  id              uuid         PRIMARY KEY DEFAULT gen_random_uuid(),
  parent_id       uuid         REFERENCES categories(id) ON DELETE RESTRICT,  -- self-FK, дерево
  slug            citext       NOT NULL,                       -- ЧПУ, уникален (регистронезависимо)
  name            text         NOT NULL,
  description     text         NOT NULL DEFAULT '',
  sort            integer      NOT NULL DEFAULT 0,             -- порядок среди соседей
  is_active       boolean      NOT NULL DEFAULT true,
  -- SEO-поля (редактируются на Этапе 2; генерация карт сайта — Этап 5.3):
  seo_title       text,
  seo_description text,
  created_at      timestamptz  NOT NULL DEFAULT now(),
  updated_at      timestamptz  NOT NULL DEFAULT now(),
  -- защита от тривиального цикла «сам себе родитель»:
  CONSTRAINT categories_no_self_parent CHECK (parent_id IS DISTINCT FROM id)
);

CREATE UNIQUE INDEX IF NOT EXISTS categories_slug_uniq   ON categories (slug);
CREATE INDEX        IF NOT EXISTS categories_parent_idx  ON categories (parent_id);
CREATE INDEX        IF NOT EXISTS categories_active_idx  ON categories (is_active);
CREATE INDEX        IF NOT EXISTS categories_sort_idx    ON categories (parent_id, sort);

GRANT SELECT, INSERT, UPDATE, DELETE ON categories TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0005', 'catalog_extensions_categories') ON CONFLICT DO NOTHING;
```

- **Дерево — adjacency list** (`parent_id` self-FK): простая, понятная, достаточная для
  каталога ИМ (глубина деревьев мала). Поддеревья читаются рекурсивным CTE
  (`WITH RECURSIVE`). Альтернативы (closure table / `ltree`) избыточны для текущего масштаба
  и усложняют запись; при росте — можно добавить материализованный путь без потери данных.
- `ON DELETE RESTRICT` на `parent_id`: **нельзя удалить категорию с подкатегориями** — защита
  от случайного обрушения ветки. Перед удалением UI требует перенести/удалить детей (явное
  действие). Циклы глубже одного уровня предотвращаются на уровне Server Action (проверка, что
  новый `parent_id` не входит в поддерево перемещаемого узла) — см. §4.3.
- `slug` уникален глобально (простые ЧПУ-маршруты витрины); `citext` снимает регистрозависимость.

### 2.2. `products` + `product_categories` (миграция `0006`)

Связь товар↔категория — **многие-ко-многим** (`product_categories`), обоснование — **ADR-007**.

```sql
-- 0006_catalog_products.sql
CREATE TABLE IF NOT EXISTS products (
  id               uuid         PRIMARY KEY DEFAULT gen_random_uuid(),
  sku              citext       NOT NULL,                  -- артикул товара, уникален
  slug             citext       NOT NULL,                  -- ЧПУ, уникален
  name             text         NOT NULL,
  description      text         NOT NULL DEFAULT '',       -- rich-text (Tiptap → HTML/JSON)
  status           text         NOT NULL DEFAULT 'draft'
                   CHECK (status IN ('draft','active','archived')),
  base_price       numeric(14,2) NOT NULL DEFAULT 0 CHECK (base_price >= 0),  -- базовая цена
  -- Презентационный кеш характеристик (ADR-007: JSONB-проекция поверх EAV-источника истины):
  attributes_cache jsonb        NOT NULL DEFAULT '{}'::jsonb,
  -- SEO:
  seo_title        text,
  seo_description  text,
  created_at       timestamptz  NOT NULL DEFAULT now(),
  updated_at       timestamptz  NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX IF NOT EXISTS products_sku_uniq    ON products (sku);
CREATE UNIQUE INDEX IF NOT EXISTS products_slug_uniq   ON products (slug);
CREATE INDEX        IF NOT EXISTS products_status_idx  ON products (status);
CREATE INDEX        IF NOT EXISTS products_created_idx ON products (created_at DESC);
-- Полнотекстовый поиск по названию/артикулу через триграммы (pg_trgm):
CREATE INDEX        IF NOT EXISTS products_name_trgm_idx ON products USING gin (name gin_trgm_ops);
CREATE INDEX        IF NOT EXISTS products_sku_trgm_idx  ON products USING gin (sku  gin_trgm_ops);

-- M2M товар ↔ категория (ADR-007). is_primary — основная категория (хлебные крошки/канонический URL).
CREATE TABLE IF NOT EXISTS product_categories (
  product_id   uuid    NOT NULL REFERENCES products(id)   ON DELETE CASCADE,
  category_id  uuid    NOT NULL REFERENCES categories(id) ON DELETE CASCADE,
  is_primary   boolean NOT NULL DEFAULT false,
  PRIMARY KEY (product_id, category_id)
);
CREATE INDEX IF NOT EXISTS product_categories_cat_idx ON product_categories (category_id);
-- Не более одной основной категории на товар:
CREATE UNIQUE INDEX IF NOT EXISTS product_categories_primary_uniq
  ON product_categories (product_id) WHERE is_primary;

GRANT SELECT, INSERT, UPDATE, DELETE ON products           TO admik_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON product_categories TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0006', 'catalog_products') ON CONFLICT DO NOTHING;
```

- **Статус `draft/active/archived`** через `CHECK` (не отдельная таблица — это фиксированный
  жизненный цикл, не данные магазина). `archived` — мягкое удаление: товар не показывается в
  витрине, но остаётся в истории заказов (Этап 3). Жёсткое `DELETE` — только для `draft`
  без заказов.
- **FTS:** `pg_trgm` + GIN по `name`/`sku` даёт быстрый «contains/похоже»-поиск для списка
  товаров в админке (`ILIKE '%...%'` / `% similarity`). Это проще полноценного `tsvector`
  и лучше для поиска по артикулу/части слова; если позже понадобится морфология — добавим
  `tsvector`-колонку отдельной миграцией без потери данных.
- `attributes_cache` (JSONB) — **денормализованная проекция** характеристик для быстрого
  чтения карточки/списка (ADR-007). Источник истины — `product_attributes` (§2.4); кеш
  пересобирается при изменении характеристик (в том же Server Action).
- M2M-каскады `ON DELETE CASCADE`: удаление товара/категории чистит связки; сами сущности
  при этом защищены своими правилами (категория с детьми — RESTRICT, §2.1).

### 2.3. `product_variants` — вариации товара (миграция `0007`)

```sql
-- 0007_catalog_variants.sql
CREATE TABLE IF NOT EXISTS product_variants (
  id            uuid          PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id    uuid          NOT NULL REFERENCES products(id) ON DELETE CASCADE,
  sku           citext        NOT NULL,                 -- собственный артикул варианта, уникален
  name          text          NOT NULL DEFAULT '',      -- напр. «Красный / M» (можно автособирать)
  -- Цена варианта: либо абсолютная price_override, либо надбавка price_delta к base_price.
  price_override numeric(14,2) CHECK (price_override >= 0),  -- NULL → берём base_price (+delta)
  price_delta    numeric(14,2) NOT NULL DEFAULT 0,           -- доп. цена относительно base_price
  is_active      boolean       NOT NULL DEFAULT true,
  sort           integer       NOT NULL DEFAULT 0,
  -- атрибуты варианта (цвет/размер именно этой вариации) — нормализованно через
  -- product_attributes (variant_id), см. §2.4; здесь — презентационный кеш:
  attributes_cache jsonb       NOT NULL DEFAULT '{}'::jsonb,
  created_at     timestamptz   NOT NULL DEFAULT now(),
  updated_at     timestamptz   NOT NULL DEFAULT now()
);

CREATE UNIQUE INDEX IF NOT EXISTS product_variants_sku_uniq    ON product_variants (sku);
CREATE INDEX        IF NOT EXISTS product_variants_product_idx ON product_variants (product_id, sort);

GRANT SELECT, INSERT, UPDATE, DELETE ON product_variants TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0007', 'catalog_variants') ON CONFLICT DO NOTHING;
```

- **Эффективная цена варианта** = `COALESCE(price_override, base_price + price_delta)`.
  Две стратегии ценообразования покрывают оба распространённых случая: фиксированная цена
  варианта и «базовая + надбавка» (например +200 ₽ за размер XXL). Вычисляется в запросе/коде,
  не хранится (источник истины — `products.base_price` + поля варианта).
- **Остаток варианта — НЕ в этой таблице:** вынесен в `inventory` (§2.6), чтобы учёт остатков
  был единообразным и расширяемым (резервы/склады). Обоснование — §2.6.
- Атрибуты, отличающие вариант (цвет=Красный, размер=M), хранятся в `product_attributes` с
  заполненным `variant_id` (§2.4); `attributes_cache` — быстрый снимок для UI/списка.

### 2.4. Характеристики: `attributes` / `attribute_values` / `product_attributes` (миграция `0008`)

Модель — **справочная EAV** (атрибуты как данные), обоснование и сравнение с JSONB — **ADR-007**.

```sql
-- 0008_catalog_attributes.sql

-- Определение характеристики (метаданные → админка строит форму автоматически).
CREATE TABLE IF NOT EXISTS attributes (
  id          uuid    PRIMARY KEY DEFAULT gen_random_uuid(),
  code        citext  NOT NULL,                  -- стабильный код: 'color','size','material' (ключ импорта)
  name        text    NOT NULL,                  -- человекочитаемое имя для UI
  type        text    NOT NULL DEFAULT 'select'  -- тип значения характеристики
              CHECK (type IN ('select','text','number','boolean')),
  unit        text,                              -- единица измерения ('см','кг','мл'); из настроек, не хардкод
  is_variant  boolean NOT NULL DEFAULT false,    -- участвует ли в образовании вариантов (цвет/размер)
  is_filterable boolean NOT NULL DEFAULT true,   -- показывать в фасетных фильтрах витрины
  is_required boolean NOT NULL DEFAULT false,
  sort        integer NOT NULL DEFAULT 0,
  created_at  timestamptz NOT NULL DEFAULT now(),
  updated_at  timestamptz NOT NULL DEFAULT now()
);
CREATE UNIQUE INDEX IF NOT EXISTS attributes_code_uniq ON attributes (code);

-- Словарь допустимых значений для select-атрибутов (единые «Красный/Синий» между товарами).
CREATE TABLE IF NOT EXISTS attribute_values (
  id           uuid   PRIMARY KEY DEFAULT gen_random_uuid(),
  attribute_id uuid   NOT NULL REFERENCES attributes(id) ON DELETE CASCADE,
  value        text   NOT NULL,                 -- 'Красный', 'M', '42'
  slug         citext,                          -- опц. ЧПУ-код значения для фильтров
  sort         integer NOT NULL DEFAULT 0,
  UNIQUE (attribute_id, value)
);
CREATE INDEX IF NOT EXISTS attribute_values_attr_idx ON attribute_values (attribute_id, sort);

-- Привязка характеристики к товару ИЛИ к варианту.
CREATE TABLE IF NOT EXISTS product_attributes (
  id            uuid   PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id    uuid   NOT NULL REFERENCES products(id)         ON DELETE CASCADE,
  variant_id    uuid   REFERENCES product_variants(id)          ON DELETE CASCADE,  -- NULL → атрибут товара
  attribute_id  uuid   NOT NULL REFERENCES attributes(id)       ON DELETE CASCADE,
  -- Значение: ссылка на словарь (для select) ИЛИ инлайн-значение (text/number/boolean).
  value_id      uuid   REFERENCES attribute_values(id)          ON DELETE RESTRICT,
  value_text    text,                                            -- для type text/number/boolean
  CONSTRAINT product_attributes_value_present
    CHECK (value_id IS NOT NULL OR value_text IS NOT NULL)
);
CREATE INDEX IF NOT EXISTS product_attributes_product_idx ON product_attributes (product_id);
CREATE INDEX IF NOT EXISTS product_attributes_variant_idx ON product_attributes (variant_id);
CREATE INDEX IF NOT EXISTS product_attributes_attr_idx    ON product_attributes (attribute_id);
-- Фасетная фильтрация «товары с значением X»:
CREATE INDEX IF NOT EXISTS product_attributes_value_idx   ON product_attributes (attribute_id, value_id);
-- Один атрибут на уровне товара не дублируется (для не-вариантных):
CREATE UNIQUE INDEX IF NOT EXISTS product_attributes_uniq
  ON product_attributes (product_id, COALESCE(variant_id, '00000000-0000-0000-0000-000000000000'::uuid), attribute_id, COALESCE(value_id, '00000000-0000-0000-0000-000000000000'::uuid));

GRANT SELECT, INSERT, UPDATE, DELETE ON attributes         TO admik_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON attribute_values   TO admik_app;
GRANT SELECT, INSERT, UPDATE, DELETE ON product_attributes TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0008', 'catalog_attributes') ON CONFLICT DO NOTHING;
```

- **`attributes` хранит метаданные** (тип, единица, обязательность, «вариантность»,
  «фильтруемость») → админка строит форму ввода характеристик **сама**, без кода под
  конкретный магазин (универсальность, ADR-003/ADR-007).
- **`value_id` vs `value_text`:** для `type='select'` значение берётся из словаря
  `attribute_values` (целостность, единые значения, фасеты) — `ON DELETE RESTRICT`, чтобы не
  осиротить привязки; для `text/number/boolean` — инлайн в `value_text`. `CHECK` гарантирует,
  что хотя бы одно значение задано. (`number` хранится строкой и валидируется Zod на входе;
  при необходимости числовой фильтрации добавим типизированную колонку миграцией.)
- **`variant_id`:** NULL — характеристика уровня товара; заполнен — характеристика конкретного
  варианта (то, чем варианты отличаются: цвет/размер). Атрибуты с `is_variant=true`
  используются UI для генерации сетки вариантов.

### 2.5. `product_media` — медиа товаров/вариантов (миграция `0009`)

```sql
-- 0009_catalog_media.sql
CREATE TABLE IF NOT EXISTS product_media (
  id          uuid    PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id  uuid    NOT NULL REFERENCES products(id)        ON DELETE CASCADE,
  variant_id  uuid    REFERENCES product_variants(id)         ON DELETE SET NULL,  -- опц. привязка к варианту
  storage_key text    NOT NULL,                 -- ключ объекта в S3/MinIO (или относительный путь в mock)
  url         text,                             -- публичный URL (S3_PUBLIC_URL + key); вычислим, кешируем
  type        text    NOT NULL DEFAULT 'image'
              CHECK (type IN ('image','video','document')),
  mime        text    NOT NULL,                 -- реальный MIME по magic-bytes (см. §3)
  alt         text    NOT NULL DEFAULT '',      -- alt-текст (a11y/SEO)
  width       integer,                          -- для изображений (после обработки sharp)
  height      integer,
  size_bytes  bigint,
  sort        integer NOT NULL DEFAULT 0,        -- порядок в галерее
  is_primary  boolean NOT NULL DEFAULT false,    -- главное изображение товара
  created_at  timestamptz NOT NULL DEFAULT now()
);

CREATE INDEX IF NOT EXISTS product_media_product_idx ON product_media (product_id, sort);
CREATE INDEX IF NOT EXISTS product_media_variant_idx ON product_media (variant_id);
-- Не более одного главного изображения на товар:
CREATE UNIQUE INDEX IF NOT EXISTS product_media_primary_uniq
  ON product_media (product_id) WHERE is_primary;

GRANT SELECT, INSERT, UPDATE, DELETE ON product_media TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0009', 'catalog_media') ON CONFLICT DO NOTHING;
```

- **БД хранит метаданные и `storage_key`, а не сами байты** — файлы лежат в S3/MinIO (или в
  mock-хранилище). `url` кешируется для удобства рендера; при смене `S3_PUBLIC_URL` можно
  пересобрать из `storage_key`.
- `variant_id` — `ON DELETE SET NULL`: удаление варианта не удаляет общий медиафайл товара,
  лишь снимает привязку. Удаление товара (`CASCADE`) удаляет строки; **физическое удаление
  объектов в S3** делает Server Action/фоновая чистка (БД-каскад байты в S3 не трогает —
  см. §4.6).
- `mime` пишется по **реальному типу из magic-bytes** (§3), не из имени файла.

### 2.6. `inventory` — остатки (миграция `0010`)

**Решение: остатки — в отдельной таблице `inventory`, а не полем `stock` в варианте.**

```sql
-- 0010_catalog_inventory.sql
CREATE TABLE IF NOT EXISTS inventory (
  id             uuid    PRIMARY KEY DEFAULT gen_random_uuid(),
  product_id     uuid    NOT NULL REFERENCES products(id)        ON DELETE CASCADE,
  variant_id     uuid    REFERENCES product_variants(id)         ON DELETE CASCADE,  -- NULL → товар без вариантов
  warehouse_code citext  NOT NULL DEFAULT 'main',   -- задел под мультисклад (сейчас всегда 'main')
  quantity       integer NOT NULL DEFAULT 0 CHECK (quantity >= 0),   -- физический остаток, не уходит в минус
  reserved       integer NOT NULL DEFAULT 0 CHECK (reserved >= 0),   -- задел: резерв под заказы (Этап 3)
  updated_at     timestamptz NOT NULL DEFAULT now(),
  CONSTRAINT inventory_reserved_le_qty CHECK (reserved <= quantity)  -- резерв не больше наличия
);

-- Одна строка остатка на (товар/вариант, склад):
CREATE UNIQUE INDEX IF NOT EXISTS inventory_unit_uniq
  ON inventory (product_id, COALESCE(variant_id, '00000000-0000-0000-0000-000000000000'::uuid), warehouse_code);
CREATE INDEX IF NOT EXISTS inventory_variant_idx ON inventory (variant_id);

GRANT SELECT, INSERT, UPDATE, DELETE ON inventory TO admik_app;

INSERT INTO schema_migrations (version, name)
VALUES ('0010', 'catalog_inventory') ON CONFLICT DO NOTHING;
```

**Почему отдельная таблица, а не `stock` в `product_variants`:**
- **Задел под склады и резервы без рефактора схемы с данными** (`docs/02`: «обновление не
  ломает данные»). `warehouse_code` уже в ключе → мультисклад добавляется строками, не
  миграцией структуры. `reserved` — точка интеграции Этапа 3 (резерв при оформлении заказа).
- **Доступное к продаже** = `quantity - reserved` (вычисляется, не хранится).
- **Конкурентные списания** делаются атомарно: `UPDATE inventory SET reserved = reserved + $n
  WHERE ... AND quantity - reserved >= $n` (условие в `WHERE` + `CHECK reserved <= quantity`
  не дают уйти в минус при гонке). Полноценная логика резерва — Этап 3; здесь — модель и
  ручное управление остатком из админки (§4.7).
- **Товары без вариантов** держат остаток с `variant_id = NULL`; с вариантами — по строке на
  вариант. Не переусложняем: одна таблица, один склад `'main'` по умолчанию.

## 3. Хранилище медиа (S3/MinIO + mock-режим)

### 3.1. Выбор: presigned upload vs серверная загрузка

**Решение: серверная загрузка (через Server Action / route handler), не presigned PUT с
клиента — на Этапе 2.** Причины:
- **Проверка magic-bytes на сервере** (паттерн 2x2, ADR-002) обязательна до помещения файла в
  хранилище; при presigned-PUT файл попадает в bucket напрямую, мимо серверной валидации.
- Единая точка для генерации превью (`sharp`), записи метаданных в `product_media`, аудита и
  проверки права `catalog.write`.
- Проще обеспечить mock-режим (один интерфейс хранилища, см. §3.3).
- Изображения товаров обычно небольшие (единицы МБ) — серверный проксинг приемлем.

> Presigned-загрузка остаётся **точкой расширения** для очень больших файлов (видео): её можно
> включить позже, добавив серверную пост-валидацию по событию загрузки. На Этапе 2 не делаем.

### 3.2. Безопасность загрузок (паттерн 2x2)

- **Magic-bytes:** реальный тип определяется по сигнатуре содержимого (например через
  `file-type`), а не по расширению/`Content-Type` из браузера. Несовпадение → отказ.
- **Белый список типов:** по умолчанию `image/jpeg`, `image/png`, `image/webp`, `image/avif`
  (для `type='image'`); прочие — только при явном расширении конфигом. Опасные типы (SVG с
  активным содержимым, исполняемые) запрещены.
- **Лимит размера:** из настроек/env (например `MEDIA_MAX_SIZE_BYTES`, дефолт 10 МБ); проверка
  до и во время чтения потока.
- **Ключ объекта:** генерируется сервером (`products/{productId}/{uuid}.{ext}`), имя файла от
  пользователя **не используется** в ключе (анти-path-traversal).
- **Обработка изображений:** `sharp` — нормализация, ресайз/превью, чтение реальных
  `width/height`, опциональная конвертация в `webp`. Метаданные пишутся в `product_media`.

### 3.3. Абстракция хранилища и mock-режим (эскиз `lib/storage/*`)

Единый интерфейс хранилища с двумя реализациями. **Mock-режим включается автоматически**,
когда S3-переменные не заданы (магазин без боевых ключей — требование `docs/02`, роадмап п.5),
чтобы `init-shop` + smoke проходили без реального S3.

```
lib/storage/
  types.ts        # интерфейс ObjectStorage + типы
  s3.ts           # S3-реализация (@aws-sdk/client-s3) — MinIO/Timeweb/AWS
  local.ts        # mock/локальная ФС-реализация (без боевых ключей)
  index.ts        # getStorage(): выбор реализации по env (S3_* заданы → s3, иначе → local)
  validate.ts     # magic-bytes + лимиты типа/размера (file-type)
  image.ts        # обработка изображений (sharp): превью, размеры, webp
```

```ts
// lib/storage/types.ts (эскиз)
export interface PutObjectInput {
  key: string;            // 'products/{id}/{uuid}.webp'
  body: Buffer | ReadableStream;
  contentType: string;    // реальный MIME (после magic-bytes)
  cacheControl?: string;
}
export interface ObjectStorage {
  put(input: PutObjectInput): Promise<{ key: string; url: string; size: number }>;
  delete(key: string): Promise<void>;
  publicUrl(key: string): string;     // S3_PUBLIC_URL + key (или /media/{key} в mock)
  mode: 's3' | 'local';
}

// lib/storage/index.ts (эскиз)
export function getStorage(): ObjectStorage;   // s3, если задан S3_ENDPOINT+S3_BUCKET+ключи; иначе local

// lib/storage/validate.ts (эскиз)
export interface MediaValidationResult { ok: boolean; mime?: string; reason?: string; }
export async function validateUpload(
  bytes: Buffer,
  opts: { allowedMime: string[]; maxSizeBytes: number },
): Promise<MediaValidationResult>;            // magic-bytes + размер
```

- **`local`-реализация (mock):** кладёт файлы в `./.storage/` (gitignore) или во временную
  папку, `publicUrl` отдаёт через локальный route `/media/[...key]` (или статику). Тот же
  контракт `ObjectStorage`, что и `s3` → код каталога не знает, какой режим активен.
- **`s3`-реализация:** `@aws-sdk/client-s3` с `forcePathStyle: true` (совместимость с MinIO),
  `endpoint`/`region`/креды — из env. `publicUrl` = `S3_PUBLIC_URL` + key.
- **Новые env (добавить в `lib/config/env.ts` и `.env.example`):**
  `MEDIA_MAX_SIZE_BYTES` (опц., дефолт 10485760), `MEDIA_ALLOWED_MIME` (опц., csv). S3-набор
  переменных **уже описан** в env (см. §0) — менять не нужно.

### 3.4. Поток загрузки (через `defineAction` / route)

1. Гвард: `catalog.write` (через `defineAction` или проверка в route handler).
2. Чтение байтов → `validateUpload` (magic-bytes + лимиты). Невалид → структурированный отказ.
3. (image) `sharp`: нормализация/превью, реальные `width/height`, опц. `webp`.
4. `getStorage().put({ key, body, contentType: realMime })`.
5. `INSERT INTO product_media (...)` через `sql` (метаданные + `storage_key` + `mime` + размеры).
6. `revalidatePath('/admin/catalog/[id]')`; аудит `catalog.media.upload`.
7. При ошибке после шага 4 (упал INSERT) — компенсация: `getStorage().delete(key)` (не
   оставлять «осиротевший» объект).

### 4.3. Категории — CRUD (`categories.actions.ts`)

```ts
export const createCategory = defineAction({ permission: 'catalog.write', input: CategoryCreateSchema, handler: /* INSERT; audit 'catalog.category.create' */ });
export const updateCategory = defineAction({ permission: 'catalog.write', input: CategoryUpdateSchema, handler: /* UPDATE; audit before→after */ });
export const moveCategory   = defineAction({ permission: 'catalog.write', input: CategoryMoveSchema,   handler: /* смена parent_id/sort; чтение рёбер + ПРОВЕРКА на цикл + UPDATE — в ОДНОЙ транзакции под advisory-xact-блокировкой дерева (анти-TOCTOU, баг #6); audit */ });
export const deleteCategory = defineAction({ permission: 'catalog.write', input: z.object({ id: z.string().uuid() }), handler: /* запрет при наличии детей (RESTRICT) → понятная ошибка; audit */ });
```

- **Защита дерева от циклов** — в `moveCategory`: проверяем, что новый
  `parent_id` не входит в поддерево перемещаемого узла (`canMoveCategory` по плоскому списку
  рёбер); иначе `validation`-ошибка. БД-уровень ловит только тривиальный self-parent (`CHECK
  categories_no_self_parent`), но НЕ многоузловой цикл A↔B.
- **Анти-TOCTOU (баг #6)** — чтение рёбер, проверка цикла и `UPDATE` выполняются в ОДНОЙ
  транзакции (`sql.begin`), в начале которой берётся сериализующая блокировка на всё дерево:
  `SELECT pg_advisory_xact_lock(hashtext('categories_tree'))`. Без неё два конкурентных
  перемещения (`{A→B}` и `{B→A}`) читали бы одинаковое ДО-состояние, оба проходили бы проверку
  и оба коммитили → устойчивый цикл, делающий узлы неудаляемыми (FK `RESTRICT` по детям).
  Один advisory-ключ на дерево сериализует любые параллельные `moveCategory`. Рёбра читаются
  именно ВНУТРИ транзакции (под блокировкой), а не через `listCategoryEdges` (отдельное
  соединение вне tx).

```ts
export const updateAttribute      = defineAction({ permission: 'catalog.write', input: AttributeUpdateSchema, handler: /* ... */ });
export const addAttributeValue    = defineAction({ permission: 'catalog.write', input: AttributeValueSchema,  handler: /* в словарь attribute_values */ });
// Привязка характеристик к товару/варианту:
export const setProductAttributes = defineAction({ permission: 'catalog.write', input: ProductAttributesSchema, handler: /* upsert product_attributes; ПЕРЕСБОР products.attributes_cache (cache.ts); audit */ });
```

### 4.6. Медиа (`media.actions.ts`) — см. §3.4

```ts
export const uploadMedia   = defineAction({ permission: 'catalog.write', input: MediaUploadSchema, handler: /* validateUpload→sharp→storage.put→INSERT product_media; audit 'catalog.media.upload' */ });
export const deleteMedia   = defineAction({ permission: 'catalog.write', input: z.object({ id: z.string().uuid() }), handler: /* DELETE row + storage.delete(storage_key); audit */ });
export const reorderMedia  = defineAction({ permission: 'catalog.write', input: MediaReorderSchema, handler: /* UPDATE sort/ is_primary; audit */ });
```

- **Удаление S3-объекта** — явно в `deleteMedia` (и компенсацией при сбое загрузки, §3.4).
  БД-каскад (`ON DELETE CASCADE` при удалении товара) чистит строки, но **не** объекты в S3:
  для этого — фоновая чистка-«сборщик мусора» (точка расширения) либо удаление медиа до
  удаления товара в Server Action `deleteProduct`.

### 4.7. Остатки (`inventory.actions.ts`)

```ts
export const setStock    = defineAction({ permission: 'catalog.write', input: StockSetSchema,    handler: /* UPSERT inventory.quantity; CHECK не даёт <0; audit before→after */ });
export const adjustStock = defineAction({ permission: 'catalog.write', input: StockAdjustSchema, handler: /* атомарный delta: UPDATE ... WHERE quantity+delta>=0; audit */ });
// reserve/release остатков под заказ — Этап 3 (orders), здесь только модель и ручное управление.
```

- Списания/корректировки — атомарными `UPDATE ... WHERE` с условием неотрицательности (§2.6),
  чтобы конкурентные операции не уводили остаток в минус.

## 5. UI админки (`/admin/catalog/*`)

Маршруты под правами; пункт меню «Каталог» **уже есть** в `nav.ts` (`module:'catalog'`,
`permission:'catalog.read'`) и сам скрывается при выключенном модуле/без права. Решение о
доступе — **только серверное** (loaders вызывают `requireUser`/проверку права; мутации — через
`defineAction`). Списки — TanStack Table (ADR-001) с **серверной** пагинацией/фильтрами/поиском.

### 5.1. Структура маршрутов

```
app/admin/catalog/
  page.tsx                  # список товаров (TanStack Table): поиск, фильтры (статус/категория), пагинация — catalog.read
  new/page.tsx              # создание товара — catalog.write
  [id]/page.tsx             # карточка товара (вкладки, ниже) — catalog.read; правки — catalog.write
  categories/page.tsx       # дерево категорий (drag-sort, CRUD) — catalog.read/write
  attributes/page.tsx       # справочник характеристик и значений — catalog.read/write
```

### 5.2. Список товаров (`page.tsx`)

- TanStack Table: колонки — фото (primary media), название, SKU, статус (бейдж), базовая цена,
  остаток (сумма по inventory), категории, дата.
- **Серверная** пагинация/сортировка/фильтрация (через `listProducts`, §4.1): поиск по
  `name`/`sku` (pg_trgm), фильтр по статусу и категории. Параметры — в URL (shareable).
- Массовые действия (архивировать/активировать) — через Server Actions, по праву `catalog.write`.

### 5.3. Карточка товара (`[id]/page.tsx`) — вкладки

| Вкладка | Содержимое |
|---|---|
| **Основное** | sku, slug, name, описание (Tiptap rich-text → `description`), статус, базовая цена, выбор категорий (M2M) + основная |
| **Варианты** | таблица вариантов (sku, атрибуты, цена override/delta, активность, остаток); генерация сетки из вариантных атрибутов (`is_variant`) |
| **Характеристики** | значения атрибутов уровня товара; форма **строится из метаданных** `attributes` (тип/единица/обязательность) — без хардкода |
| **Медиа** | загрузчик (§5.5), галерея с drag-sort, выбор главного, alt-тексты |
| **SEO** | seo_title, seo_description, предпросмотр slug/URL (генерация sitemap — Этап 5) |

### 5.4. Дерево категорий (`categories/page.tsx`)

- Древовидный список (раскрытие/сворачивание), drag-sort внутри уровня (`sort`), смена
  родителя (`moveCategory` с проверкой цикла). CRUD + SEO-поля категории.
- Удаление запрещено при наличии детей (понятная ошибка из RESTRICT, §4.3).

### 5.5. Загрузчик медиа

- Drag-and-drop зона, прогресс, превью; клиент шлёт файл в Server Action/route `uploadMedia`.
- Все проверки (magic-bytes, тип, размер) — **на сервере** (§3); клиент лишь подсказывает.
- В mock-режиме (нет S3-ключей) работает через локальное хранилище — для demo/smoke.

## 6. Универсальность и мультимагазинность (чек-лист Этапа 2)

Прямое следствие ADR-003 и `docs/02`. Для каждого пункта — как соблюдаем:

- [ ] **Характеристики — данные, не код.** `attributes`/`attribute_values` заводятся через UI
      и seed; код не знает названий конкретных характеристик (ADR-007). Новый магазин — свой
      набор атрибутов без правки кода.
- [ ] **Категории — данные.** Дерево/SEO — строки в `categories`, не хардкод.
- [ ] **Единицы измерения** — поле `attributes.unit` (данные), не зашиты в код.
- [ ] **Валюта** — из настроек/env (`SHOP_CURRENCY`, добавить в env/настройки; форматирование
      цен по локали магазина). В коде каталога нет хардкода «₽».
- [ ] **Никаких хардкодов магазина** в товарах/категориях/медиа; брендинг/реквизиты — ядро/Этап 5.
- [ ] **Флаг модуля `catalog`** уважается в UI (nav уже) + Server Actions + роутах.
- [ ] **Mock-режим хранилища** — магазин без боевых S3-ключей поднимается и проходит smoke.
- [ ] **Импорт-заготовка (CSV)** как точка расширения (§6.3) — проектно, не реализуем сейчас.
- [ ] **Идемпотентные миграции** `0005+` и идемпотентный seed демо-каталога.

### 6.3. Импорт каталога (CSV) — точка расширения (проектно)

Не реализуется на Этапе 2, но закладывается архитектурно, чтобы не переделывать модель:
- **Стабильные ключи импорта:** `products.sku`, `attributes.code`, `categories.slug` —
  человекочитаемые стабильные идентификаторы для upsert из CSV (`ON CONFLICT (sku) DO UPDATE`).
- **Формат:** CSV со столбцами `sku,name,slug,status,base_price,categories(slug;slug),attr:color,attr:size,...`;
  колонки `attr:<code>` маппятся на атрибуты по коду (создаются значения в словаре при
  отсутствии).
- **Реализация позже:** парсер + предпросмотр + пакетный upsert через те же репозитории/Server
  Actions (право `catalog.write`, аудит партии). Заводится отдельной задачей после стабилизации
  модели §2.

## 7. Декомпозиция Этапа 2 на задачи (параллельные пакеты, «сначала тесты»)

Принципы (ADR-004 и `docs/03`, «Принципы, обязательные для каждой задачи») действуют для всех
задач: сперва падающие тесты, затем реализация до зелёного; идемпотентные миграции; уважение
флага `catalog`; секреты в env; mock-режим хранилища; никаких хардкодов магазина;
переиспользование ядра (`defineAction`, `sql`, `writeAudit`, права `catalog.*`).

**Карта зависимостей и параллельности (5 пакетов):**

```
            ┌────────────────────────────────────────────────┐
   П1 (БД/миграции 0005–0010) ──┬──> П3 (data+actions) ──┬──> П4 (UI)
            │                   │                         │
   П2 (storage: s3/local/      ─┘ (медиа-actions зависят  │
       validate/image)             от П2)                 │
            │                                             │
   П5 (seed демо-каталога) ──────── зависит от П1 (схема) ┘ (наполняет UI данными)
```

- **П1 и П2 стартуют параллельно** (непересекающиеся файлы: `db/migrations/*` vs `lib/storage/*`).
- **П3** стартует после П1 (нужна схема); медиа-часть П3 ждёт П2.
- **П4** (UI) стартует, как только появятся сигнатуры Server Actions из П3 (можно вести против
  контрактов/моков параллельно, интеграция — по готовности П3).
- **П5** (seed) — после П1 (схема); полезен П4 для наполнения интерфейса.

### Пакет П1 — БД и миграции (`db/migrations/0005–0010`) — агент Backend/DB

- **Тесты (первыми):** двойной накат `0005–0010` на временную БД не падает и не меняет данные
  (идемпотентность); существуют `app`-гранты на все таблицы каталога; FK/ON DELETE ведут себя
  по §2.7 (категория с детьми не удаляется; CASCADE чистит связки); `CHECK`-ограничения
  (статус, неотрицательные цена/остаток, `reserved<=quantity`) срабатывают.
- **Состав:** миграции §2.1–2.6 (categories, products+product_categories, variants,
  attributes+values+product_attributes, media, inventory) + `pg_trgm`.
- **Приёмка:** `init-shop.sh` накатывает всё одной командой; повтор безопасен; индексы/FTS на
  месте; гранты выданы.
- **Зависимости:** Этап 1 (миграции `0001–0004`, расширения `pgcrypto`/`citext`).

### Пакет П2 — хранилище медиа (`lib/storage/*`) — агент Backend/Storage

- **Тесты (первыми):** `validateUpload` принимает валидные jpeg/png/webp по magic-bytes и
  отклоняет подделку расширения/превышение размера; `getStorage()` выбирает `local` без
  S3-env и `s3` при заданных ключах; `local.put/delete/publicUrl` работают на временной папке;
  (с мок-S3) `s3.put` формирует корректный ключ/URL.
- **Состав:** `types.ts`, `s3.ts` (`@aws-sdk/client-s3`), `local.ts` (mock), `validate.ts`
  (`file-type`), `image.ts` (`sharp`), `index.ts`; новые env (`MEDIA_MAX_SIZE_BYTES`,
  `MEDIA_ALLOWED_MIME`) в `env.ts`/`.env.example`.
- **Приёмка:** загрузка безопасна (тип по содержимому, лимиты); mock-режим работает без боевых
  ключей; единый контракт `ObjectStorage`.
- **Зависимости:** env (Этап 0). Не зависит от П1 — может идти параллельно.

### Пакет П3 — слой данных и Server Actions (`lib/catalog/*`) — агент Backend

- **Тесты (первыми):** Zod-схемы (валидные/невалидные входы); репозитории строят корректные
  параметризованные запросы (анти-SQLi); каждая мутация требует `catalog.write` и пишет аудит
  (через `defineAction` + моки `deps`); `moveCategory` отвергает циклы; `adjustStock` не уводит
  в минус; пересбор `attributes_cache` корректен; Server Actions каталога отклоняются при
  выключенном модуле `catalog`.
- **Состав:** §4 — repos + actions для категорий/товаров/вариантов/атрибутов/медиа/остатков +
  `schemas.ts` + `cache.ts`.
- **Приёмка:** весь CRUD через `defineAction` (`catalog.read`/`catalog.write`), аудит и
  инвалидация на месте; флаг модуля уважается.
- **Зависимости:** П1 (схема) для всех; медиа-actions — ещё и П2.

### Пакет П4 — UI админки (`app/admin/catalog/*`) — агент Frontend

- **Тесты (первыми, Playwright + Vitest):** a11y страниц каталога; список — серверная
  пагинация/поиск/фильтры; карточка-вкладки рендерятся; форма характеристик строится из
  метаданных атрибутов; загрузчик медиа шлёт на сервер; **раздел недоступен при выключенном
  модуле/без права** (redirect/404, пункт меню скрыт — логика `nav.ts` уже покрыта).
- **Состав:** §5 — список, карточка (5 вкладок), дерево категорий, справочник атрибутов,
  загрузчик медиа; TanStack Table; Tiptap для описания.
- **Приёмка:** маршруты под правами (серверное решение); список/карточка/дерево/медиа работают;
  a11y зелёные.
- **Зависимости:** контракты Server Actions из П3 (можно вести параллельно против сигнатур,
  интеграция — по готовности П3).

### Пакет П5 — seed демо-каталога (`db/seed/catalog*.sql` или `.mjs`) — агент Backend/Data

- **Тесты (первыми):** идемпотентность (повтор не плодит дублей — `ON CONFLICT`); создаются
  категории/атрибуты/значения/товары/варианты/остатки; согласованность с `lib/catalog`
  (демо-данные проходят те же Zod-схемы).
- **Состав:** небольшой демонстрационный каталог (несколько категорий-дерево, атрибуты
  color/size, 5–10 товаров с вариантами и остатками, плейсхолдеры медиа). Подключение к
  `init-shop.sh` опционально (демо-режим), не обязательно для пустого боевого магазина.
- **Приёмка:** seed идемпотентен; наполняет UI данными для smoke/демо; не нарушает
  copy-paste (демо отделено от обязательного владельца/ролей).
- **Зависимости:** П1 (схема).

> **Порядок внутри Этапа 2 (роадмап `2.1→2.5`)** сохраняется как логическая последовательность
> приёмки (категории → товары → варианты+характеристики → медиа → остатки), но **исполнение
> распараллеливается** по пакетам П1–П5 выше: схема и storage готовятся сразу, actions и UI
> наслаиваются по мере готовности схемы.

## 8. Новые зависимости

| Пакет | Назначение | Почему он |
|---|---|---|
| `@aws-sdk/client-s3` | S3-клиент (MinIO/Timeweb/AWS) | стандарт для S3-совместимых хранилищ; `forcePathStyle` для MinIO; модульный (только S3, без всего SDK) |
| `sharp` | обработка изображений: превью, ресайз, webp, размеры | де-факто стандарт в Node; нативная скорость; нужен для превью и чтения `width/height` |
| `file-type` | определение MIME по magic-bytes | безопасность загрузок (ADR-002, паттерн 2x2): тип по содержимому, не по расширению |
| `@tanstack/react-table` | таблицы админки (список товаров) | уже в плане ADR-001; серверная пагинация/сортировка/фильтры |
| `@tiptap/react` (+ extensions) | rich-text описания товара | уже в стеке ADR-001 (Tiptap); редактор описания/контента |
| `zod` | валидация входа Server Actions | уже в стеке; `schemas.ts` каталога |

- **`@aws-sdk/client-s3` vs `minio`:** выбираем AWS SDK v3 — он покрывает и MinIO
  (`forcePathStyle`), и боевые AWS/Timeweb одним клиентом; широкая поддержка, модульная
  установка (ставится только S3-подпакет). Это убирает развилку «два разных клиента для dev/prod».
- **`sharp`** — нативный модуль; учесть в Docker-образе (Этап 0/6) совместимую сборку (alpine
  → `linux-musl` бинарь). На Этапе 2 это лишь зависимость слоя storage.
- **`file-type`** — лёгкий, читает сигнатуру из первых байтов; не доверяем `Content-Type`/имени.
- Установка через pnpm (ADR-001), фиксация в lock-файле. TanStack Table и Tiptap уже
  заявлены в ADR-001 — формально новыми не являются, перечислены как фактически вводимые в строй.

## 9. Резюме принятых решений (для журнала)

1. **Характеристики — справочная EAV** (`attributes`/`attribute_values`/`product_attributes`),
   атрибуты как **данные** (seed/UI), метаданные (тип/единица/обязательность) → админка строит
   форму сама; + JSONB-кеш `attributes_cache` для быстрого чтения карточки. Обоснование —
   **ADR-007** (контроль словаря значений, фасетная фильтрация, универсальность).
2. **Товар↔категория — many-to-many** (`product_categories`, `is_primary`) — товар в нескольких
   разделах/акциях без рефактора схемы с данными. **ADR-007.**
3. **Остатки — отдельная таблица `inventory`** (не поле `stock` в варианте): задел под склады
   (`warehouse_code`) и резервы (`reserved`, Этап 3) без миграции структуры; атомарные списания
   с `CHECK`-неотрицательностью.
4. **Дерево категорий — adjacency list** (`parent_id` self-FK, `ON DELETE RESTRICT`),
   обход рекурсивным CTE; защита от циклов — в Server Action `moveCategory`.
5. **Медиа — S3/MinIO, серверная загрузка** (не presigned) с **magic-bytes** (`file-type`),
   лимитами типа/размера, `sharp`-превью; абстракция `ObjectStorage` с **mock-режимом**
   (`local`) при отсутствии S3-ключей. БД хранит метаданные + `storage_key`, не байты.
6. **Миграции `0005–0010`**, идемпотентные, в стиле `0001–0004`; `pg_trgm` для поиска;
   `app`-гранты на все таблицы; `schema_migrations` обновляется. Совместимо с `init-shop.sh`.
7. **Переиспользование ядра без правок:** права `catalog.read`/`catalog.write` (уже есть),
   `defineAction`, `sql`, `writeAudit`, пункт меню «Каталог» (уже есть), флаг модуля `catalog`.
   Новых прав/пунктов меню/правок ядра не требуется.
8. **Новые зависимости:** `@aws-sdk/client-s3`, `sharp`, `file-type` (+ вводятся в строй
   `@tanstack/react-table`, `@tiptap/react`, уже заявленные ADR-001).

### Список задач Этапа 2 (пакеты для параллельных агентов)

1. **П1 — БД/миграции `0005–0010`** (categories, products+product_categories, variants,
   attributes/values/product_attributes, media, inventory; pg_trgm; гранты).
   Тесты: идемпотентность, FK/ON DELETE, CHECK, гранты. *(зависит от Этапа 1)*
2. **П2 — хранилище медиа `lib/storage/*`** (s3/local/validate/image/index; magic-bytes;
   mock-режим; новые env). Тесты: валидация по содержимому, лимиты, выбор реализации.
   *(параллельно П1)*
3. **П3 — data+actions `lib/catalog/*`** (repos + `defineAction`-мутации для всех сущностей;
   `attributes_cache`; флаг модуля; аудит). Тесты: Zod, анти-SQLi, права/аудит, циклы дерева,
   неотрицательные остатки. *(зависит от П1; медиа-часть — от П2)*
4. **П4 — UI `app/admin/catalog/*`** (список TanStack Table; карточка-вкладки; дерево
   категорий; справочник атрибутов; загрузчик медиа; Tiptap). Тесты: a11y, серверные
   пагинация/фильтры, скрытие при выключенном модуле/без права. *(против контрактов П3)*
5. **П5 — seed демо-каталога** (категории/атрибуты/товары/варианты/остатки, идемпотентно).
   Тесты: идемпотентность, согласованность со схемами `lib/catalog`. *(зависит от П1)*
